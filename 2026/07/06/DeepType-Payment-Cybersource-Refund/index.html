Cybersource-Refund

Created2026-07-06|Updated2026-07-10|Payment
|Post Views:

每小時 02 分，SQL Job 排程往 NMQV2 的 Task 表塞入一筆「退款分群」任務 → NMQ 撈出所有狀態為 RefundRequestProcessing 的退款單並依 TGCode 分群 → 針對 Cybersource（CreditCardOnce_Cybersource）逐筆建立 Finish Task → Finish Job 呼叫 PaymentMiddleware 執行 Refund（首次退款）或 RefundQuery（查詢退款結果），再依回應 ReturnCode 更新退款單狀態與訂單大表

入口類別：PaymentMiddleWareRefundRequestGroupingProcess : ThirdPartyPaymentRefundRequestGroupingProcess，以 ThirdPartyPaymentTypeDefEnum.PaymentMiddleWare 作為分類，代表本 Job 統一處理所有透過 PaymentMiddleware 串接的 28 種付款方式（含 Cybersource）。

步驟 說明
Step 1 GetRefundRequestData() 對每個 payType（含 CreditCardOnce_Cybersource）撈出 RefundRequestProcessing 狀態退款單，並額外呼叫 payChannelService.GetRedoRefundRequestList(payType) 撈重試清單（Cybersource 目前回傳 null，尚未實作 Redo）
Step 2 GroupBy(TradesOrderGroupId) 依購物車（TGCode）分群，方便後續 Razer 合併退款；Cybersource 雖不合併金額，仍共用同一個分群/Task 建立流程
Step 3 UpdateRefundRequestStatus → RefundRequestGrouping 鎖定退款單狀態，避免下次 :02 分排程重複撈取同一批
Step 4 建立 NMQ Task Cybersource 對應 JobName：**CreditCardOnceCybersourceRefundRequestFinish**
Step 5 RefundTaskBookingTime() Cybersource 覆寫回傳 null → 不做限速排程，Task 立即可執行

3.3 Phase 2：Finish Job（DoRefundRequestFinish） — Cybersource 走 RefundByRequestId

Cybersource 的 CanGroupingRefund()（繼承 AbstractPayChannelService 預設值）為 false，因此必定走逐筆退款路線，不會像 Razer 一樣合併同 TGCode 金額。

  • IsQueryRefund：CybersourcePayChannelService.IsQueryRefund() 判斷 RefundRequest_TransactionId 是否已有值 —— 第一次退款時為空 → 呼叫 Refund；Refund 回應 PENDING 後，TransactionId 已寫回退款單 → 下次排程改呼叫 RefundQuery。
  • IsRefund：Cybersource 未覆寫，預設為 true。
  • GetRefundExtendInfo / GetRefundQueryExtendInfo：由 CybersourcePayChannelService 提供，分別帶入 TradesOrderGroupCode 與 RefundRequestTransactionId。

4.2 Refund（新退款）

欄位 來源
transactionId（URL） 付款時 Cybersource 回傳的 transaction ID
clientReferenceInformation.code RefundRequestExtendInfo.TradesOrderGroupCode
orderInformation.amountDetails.totalAmount request.Amount
orderInformation.amountDetails.currency request.Currency

特殊設計 — HTTP 錯誤不拋出：

即使 HTTP 非 2xx（如 DECLINED 案例常見的 4xx/5xx），也會捕捉例外並改為解析 response body 的 errorInformation，確保不因 HTTP 狀態碼而遺失錯誤細節。

Status → ReturnCode：

Status ReturnCode 說明
PENDING RefundPending 受理中，需後續 RefundQuery
500 RefundFailed 系統錯誤，含 errorInformation
DECLINED RefundFailed 發卡行拒絕，含 errorInformation
其他未知值 ⚠️ throw NotImplementedException Cybersource 若新增狀態需修改程式碼

4.3 RefundQuery（查詢既有退款）

注意：URL 帶入的是退款動作回傳的 transaction ID，非原始付款 transaction ID。

Status/ReasonCode → ReturnCode：

applicationInformation.status reasonCode ReturnCode 說明
PENDING 任意 RefundPending Cybersource 僅受理，尚未送出
TRANSMITTED 任意 Success 已傳送至發卡行，視為退款完成
其他 100 RefundPending Fallback：reasonCode 成功但狀態非預期，仍視為處理中
其他 非 100 RefundRejected 退款最終被拒絕

為何 TRANSMITTED 才算成功而非 PENDING？
Cybersource 退款生命週期為：PENDING（受理）→ TRANSMITTED（送出至發卡行）→ 最終到帳。PENDING 僅代表系統收到請求，尚未真正動作。

5. 退款狀態機（整體視角）

stateDiagram-v2
    [*] --> RefundRequestProcessing: 建立退款單
    RefundRequestProcessing --> RefundRequestGrouping: SQL排程:02分\n撈取並鎖定
    RefundRequestGrouping --> CallingRefund: TransactionId為空\n→呼叫 Refund API
    RefundRequestGrouping --> CallingRefundQuery: TransactionId已存在\n→呼叫 RefundQuery API

    CallingRefund --> RefundRequestProcessing: status=PENDING\n(寫入TransactionId)\nReturnCode=RefundPending
    CallingRefund --> RefundRequestFail: status=500/DECLINED\nReturnCode=RefundFailed

    CallingRefundQuery --> RefundRequestProcessing: status=PENDING 或\nreasonCode=100(fallback)\nReturnCode=RefundPending
    CallingRefundQuery --> Finish: status=TRANSMITTED\nReturnCode=Success
    CallingRefundQuery --> RefundRequestFail: 其他status且reasonCode≠100\nReturnCode=RefundRejected

    RefundRequestFail --> RefundRequestProcessing: 符合重試條件\n(3天內+可重試錯誤碼)\n→ 改回Processing重新嘗試
    RefundRequestFail --> [*]: 逾3天/不可重試\n人工介入
    Finish --> [*]: IsClosed=true\n寫入ConfirmDate\nUpdateOrderSlaveFlow

⚠️ Cybersource 目前 GetRedoRefundRequestList() 回傳 null（TODO 待補），代表 Cybersource 沒有自動化失敗重試機制，與 Razer/KPay（有 PR020/PR015/29523 可重試錯誤碼）不同，RefundFailed 後需人工介入處理。

    participant SQL as SQL Agent 排程
    participant NMQDB as NMQV2DB.Task
    participant Grouping as GroupingProcess
(PaymentMiddleWareRefundRequestGroupingProcess)
    participant Service as PaymentMiddleWareRefundRequestService
    participant PayChSvc as CybersourcePayChannelService
    participant Finish as FinishJob
(DoRefundRequestFinish)
    participant PMW as PaymentMiddleware
    participant CS as Cybersource API

    SQL->>NMQDB: INSERT Task(Ready, JobId=PaymentMiddleWareRefundRequestGrouping)
    NMQDB-->>Grouping: NMQ Worker 撈取 Ready Task
    Grouping->>Service: CreateRefundRequestFinish()
    Service->>Service: GetRefundRequestData()（含 Cybersource）
    Service->>PayChSvc: GetRedoRefundRequestList("CreditCardOnce_Cybersource")
    PayChSvc-->>Service: null（無 redo 機制）
    Service->>Service: GroupBy(TradesOrderGroupId)
    Service->>Service: UpdateRefundRequestStatus → RefundRequestGrouping
    Service->>NMQDB: 建立 Task(CreditCardOnceCybersourceRefundRequestFinish)

    NMQDB-->>Finish: Task 到點觸發
    Finish->>PayChSvc: CanGroupingRefund()
    PayChSvc-->>Finish: false → 走 RefundByRequestId
    Finish->>PayChSvc: IsQueryRefund(refundRequest)

    alt TransactionId 為空（首次退款）
        PayChSvc-->>Finish: false
        Finish->>PMW: POST /Refund/Cybersource（TradesOrderGroupCode, Amount, Currency）
        PMW->>CS: POST pts/v2/payments/{transactionId}/refunds/
        CS-->>PMW: status = PENDING / DECLINED / 500
        PMW-->>Finish: ReturnCode = RefundPending / RefundFailed
    else TransactionId 已存在（查詢中）
        PayChSvc-->>Finish: true
        Finish->>PMW: GET /RefundQuery/Cybersource（RefundRequestTransactionId）
        PMW->>CS: GET tss/v2/transactions/{RefundRequestTransactionId}
        CS-->>PMW: applicationInformation.status / reasonCode
        PMW-->>Finish: ReturnCode = RefundPending / Success / RefundRejected
    end

    Finish->>Finish: ChangeRefundRequestStatus + RefundDelay()（空實作）
    Finish->>Service: UpdateRefundRequest + UpdateOrderSlaveFlow
    Note over Service: Success → Finish結案(IsClosed=true)
Pending → 維持Processing，等下次:02分再次撈取
Rejected/Failed → RefundRequestFail
階段 ReturnCode 對應退款單狀態 說明
Refund RefundPending RefundRequestProcessing（保留） 已受理，寫回 RefundRequest_TransactionId，下次改走 RefundQuery
Refund RefundFailed RefundRequestFail HTTP 500 或發卡行 DECLINED
RefundQuery RefundPending RefundRequestProcessing（保留） 仍在處理中，下次繼續 Query
RefundQuery Success Finish（結案） TRANSMITTED，寫入 ConfirmDate，IsClosed=true
RefundQuery RefundRejected RefundRequestFail 最終被拒絕
Grouping共通 00 Finish 成功結案
Grouping共通 PR016 RefundRequestProcessing 超過退款期限，TypeDef 改為 Remittance（改走匯款）
Grouping共通 PR021 RefundRequestProcessing Pending，等待中
  1. 未知 Status 會拋例外：Refund 回應若出現非 PENDING/500/DECLINED 的新狀態，NotImplementedException 會中斷處理，需持續留意 Cybersource 是否新增狀態值。
  2. 無 Redo 重試機制：GetRedoRefundRequestList() 回傳 null（TODO），RefundFailed 之後不會自動重試，須靠其他機制（人工/其他排程）補救。
  3. RefundQuery 長時間 PENDING：Cybersource 查詢回應可能長達數小時停留在 PENDING，若系統邏輯有 timeout 中斷，最後一次查詢結果可能被誤判為 RefundPending (4003)，需靠下次排程 Redo 才能拿到最終 TRANSMITTED。
  4. 運費退款不更新大表：SourceDef == SalesOrderFee 時，Finish 階段不會呼叫 UpdateOrderSlaveFlow，只更新退款單本身。
  5. 多付款方式（IsMultiPayment）：Cybersource 若涉及多付款方式退款，Finish 完成後不直接更新大表，改建立 SyncRefundRequestFinishJob，待所有退款單皆完成後統一更新。

使用者在後台按下退款時，系統不會馬上呼叫 Cybersource API，而是先寫一筆 RefundRequestProcessing 退款單，等下一次 :02 分排程才真正發送。

面向 好處 壞處
非同步解耦 後台退款操作可以立即回應使用者（不用等第三方 API），退款單建立與實際扣款動作分離，避免 UI 因外部 API 緩慢而卡住 使用者按下退款後最多要等將近 1 小時才會真正送出（若剛好錯過 :02 分），退款體感速度慢，客服/客戶容易誤以為系統沒反應
批次處理 / 削峰填谷 大量退款單可以一次性批次撈取、統一分群處理，避免尖峰時段瞬間大量獨立 API 呼叫打爆 PaymentMiddleware 或第三方金流 排程頻率固定（每小時一次），無法針對急件做即時處理；若當次排程處理量過大，反而可能拖慢單筆退款的實際處理時間
與主服務解耦（不同 Process/Server） SQL Job 只負責「通知」，實際運算在 NMQ Worker 進程執行，避免退款邏輯佔用 Web/API 主服務資源，也方便獨立擴展 Worker 多一層基礎設施（SQL Agent + NMQV2DB.Task + NMQ Worker），架構複雜度提高，除錯時要橫跨 DB 排程、Task 表、Worker Log 三個地方才能還原全貌
防呆機制（cfn_GetJobWhetherToPerform 判斷讀寫角色） 避免在 DR / 唯讀複本上誤觸發寫入動作，保護資料一致性 多一層條件判斷，維運人員需理解 readwrite 語意才能排查「為什麼排程沒有觸發」

核心設計哲學：這是典型的 「Polling Queue」模式，用 DB 當作簡易 Message Queue（Task 表 + Task_Status），優點是實作簡單、不需要額外導入 Kafka/RabbitMQ 等中介軟體，缺點是輪詢間隔即是延遲上限，且 DB 本身成為隱性瓶頸（大量 Task 堆積時，讀寫壓力都在同一個 DB）。
