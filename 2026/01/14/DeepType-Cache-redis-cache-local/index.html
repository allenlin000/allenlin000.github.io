Redis 伺服器沒有設定密碼保護

Redis 是一個高速的記憶體資料庫，。
這背後的本質問題是：

例如，一家公司在 AWS 架了一台 Redis 伺服器，沒有設定密碼，也沒有限制來源 IP。
駭客掃描全網開放的 6379 port，就發現了這台 Redis。
他用以下指令：

redis-cli -h 203.xxx.xxx.xxx

即可直接連進去，然後透過 Redis 的 CONFIG SET 功能，將自己的 SSH 公鑰寫入伺服器的 .ssh/authorized_keys 檔案。
之後駭客只要用：

ssh -i id_rsa username@203.xxx.xxx.xxx

就能遠端登入這台伺服器，完全控制系統。

在 Redis 主機上直接查看設定檔（最可靠）
登入 Redis 主機（透過 SSH 到應該是運行 Redis 的那台主機）：

找出 redis process 與使用的 config 檔

ps aux | grep redis-server

或檢查常見路徑

sudo grep -E “requirepass|aclfile|bind|protected-mode” /etc/redis/redis.conf /etc/redis/*.conf 2>/dev/null

若系統以 systemd 啟動，查看 unit 參數

systemctl status redis

或查看啟動命令以找到 conf 檔路徑

在 redis.conf 中尋找：requirepass、aclfile（與 bind、protected-mode 設定）。如果 aclfile 指向一個檔案，查看該檔內容以確認使用者與密碼設定。

檢查 AWS 網路層（Security Group / NACL / VPC）
到 AWS Console 確認該 instance 的 Security Group 是否允許從公司內網（或 0.0.0.0/0）存取 6379。若允許外部或很大範圍的來源，風險更高。
