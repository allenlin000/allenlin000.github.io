Docker for Dotnet and share

Created2026-05-04|Updated2026-05-15|Docker
|Post Views:

一開始怎麼知道要這樣寫?

首先，我們要從這個問題開始：我的程式要跑起來，需要哪些步驟？

而其實 Dockerfile 只是把這些步驟搬進 container 裡，並且我們可以一段一段作

  1. 本機 dotnet run 成功
  2. 本機 dotnet publish 成功
  3. 本機 dotnet ./publish/MyMvcApp.dll 成功
  4. 寫 SDK-only Dockerfile
  5. docker build 成功
  6. docker run 成功
  7. 改成 multi-stage Dockerfile
  8. 再 docker build / docker run

撰寫 dockerfile

SDK only

假設想先測非 multi-stage 版本也可以先測試

雖然它只有一個 stage，但它同時做了 build + runtime，只是 runtime 也繼續用 SDK image，假設現在在 MyMvcApp 專案資料夾，也就是有 .csproj 的那一層

成功後執行

然後打開瀏覽器 http://localhost:5000，如果畫面出現 MVC 首頁，就代表成功

因為 一個 stage 可以跑起來，但 multi-stage 是為了讓正式部署的 image 更乾淨、更小、更安全、更像正式環境

一個 stage 的問題是這種 SDK-only 版本正式部署時，container 裡會包含很多其實不需要的東西

  • .NET SDK
  • 原始碼
  • NuGet restore/build 工具
  • 編譯工具
  • 可能還有測試檔案、Dockerfile、README、.git 內容

可是網站真正執行時，其實只需要

  • .NET Runtime
  • publish 出來的 dll
  • appsettings
  • wwwroot
  • 相關相依檔案

Multi-stage 把「建置」和「執行」分開

build stage：負責編譯

runtime stage：負責執行

意思是第一段 SDK image 用來 restore / build / publis，而第二段 ASP.NET Runtime image 只拿 publish 結果來跑網站，最後真的留下來的 image 只有第二段，第一段 build stage 不會被帶進正式 image 裡
