理解

首先，原實驗的文章中有提到，會造成 Deadlock 的結果原因在於 ASP.MET 的內建機制會去偵測目前所處的 SynchronizationContext 並記錄下來，確保非同步作業完成後繼續用 SynchronizationContext 執行後續程式，因此就是，當 await 內部的工作完成後，正準備用 ASP.NET Context 處理後續作業將結果傳回呼叫端，但是，Index () 使用 .Result 卡住了 ASP.NET Context ，要求他盡忠職守的等待 GetRemoteData () 的回傳結果否則不准做其他事情，因此造成了循環等待，DeadLock!

但進入 ASPNET.CORE 的世界，SynchronizationContext 被移除掉了，原本 ASPNET 傳統框架在非同步的工作結束後，要回來完成剩下的工作時，這個剩下的工作會被排隊到 SynchronizationContext，等待其他任務完成才可以處理他，而且這個過程會有一些額外工作要處理，包含 “重新進入” 請求上下文、設置 HttpContext.Current 也就是當前 Web Request 相關的資訊、設置當前Identity、Culture 等等為了包持一致性，但 CORE 改成當非同步恢復執行程序时，將會從 Thread Pool 中直接獲取 Thread 並完成剩餘工作，增加了效能，所以我們在剛剛的實驗沒有被鎖死就是這個原因

參考 : https://blog.stephencleary.com/2017/03/aspnetcore-synchronization-context.html
