Content Delivery Network

Created2024-08-24|Updated2025-05-01|程思舞想
|Post Views:

在一個大熱天的午後，你走在勤美綠園道上，突然想吃冰冰涼涼又綿密的布丁。你走路 3 分鐘到附近的全家，打開冰櫃看到 「東京限定布丁」，而這就是 CDN 的概念，因為正常來說，這種東西要從日本原廠寄過來，不但要等幾天、還可能因為海關卡關而延遲。但現在不一樣了，因為：

🎉「全家便利商店」早就跟日本合作，把布丁大量送到全台分店的冷藏庫裡，你走路 3 分鐘就能買到！

回到開發者的視角，我們維護一個網站 example.com，若使用 CDN Cache，當使用者首次開啟網頁，CDN 還沒有 cache，就會去你的主機拿資料（稱為「miss」）。CDN 把資料快取起來，設定 1 小時不更新。接下來 1 小時內，所有人打開網站都直接從 CDN 拿資料（稱為「hit」）。這就是 CDN Cache 的運作過程。


高清版

用 S3 圖片實作 CDN 快取

實際上，只要搭配 AWS S3 + CloudFront，你就能輕鬆體驗 CDN 的快取效果。

教學 (S3 + CloudFront)

舉例來說，你可以將圖片 logo.png 上傳到 S3，經由 CloudFront 發佈出去，並設定 Cache-Control: max-age=3600，這樣就能讓使用者在 1 小時內都從 CDN 拿到這張圖片，而非每次都回源載入。
連結教學包含設定資料來源 ( Origin Settings )、設定 cache 行為 ( Default Cache Behavior Settings )，以新手操作來說已經非常完整!

CDN 的本質

CDN 的本質其實就一句話：

用空間換時間（Trade Space for Time）

雖然聽起來很高大上，但還是犧牲了許多東西的

CDN 可以解決甚麼問題?


🔹在主機當機時，提供部分的支援


快取的資源照樣能用（如果在 cache 裡）


如果你把靜態資源（如 HTML、JS、CSS、圖片、影片）預先快取在 CDN 上：

即使主機掛了，這些資源還是能被存取（直到 TTL 過期或強制回源）。

這在前端 SPA、展示型網站、Landing Page 上特別有效。

✅ 適用場景：官網、宣傳頁、文件站

具備「回源容錯機制」的 CDN


有些高級 CDN（例如 Cloudflare、Akamai）支援：

Origin Shielding：先透過中介節點減少主機壓力

Always Online 模式：即使主機失聯，也可提供舊版快取內容

但他仍解不了動態資料（API、登入、購物車），因為這些資料不能快取，得即時向主機請求 → 主機掛了，功能就會失效
像是結帳流程、會員登入、後台管理 → 一定會出錯，而沒有快取的頁面會直接失敗，如果某些頁面是每次都回源的（Cache-Control: no-cache），CDN 也幫不了你 → 會出現 5xx 錯誤

因此 CDN 比較像是支援者的角色，用更更完整的備援策略應該是與 CDN 搭配組合
當主機掛掉，我們可以加上 負載平衡 + 多機熱備援（Active-Active），而 CDN 只快取靜態資源並且搭配動態 API 做故障轉移（Failover API），當單點資料庫失效時，因為我們採用分散式資料庫、快照備份、延遲同步等機制讓我們有時間修復

總結來說，CDN 是一個「緩衝」的幫手，不是「全能」的備援系統，它能「在主機掛掉時，維持部分服務不斷」，但不能「接手整個網站邏輯」。
