Host

Created2024-08-10|Updated2024-08-12
|Post Views:

過去 ASP.NET 只能運作在 Windows 上，而後，ASP.NET Core 做為可以跨平台執行的框架而誕生，但一開始他是以ASP.NET vNext這個有點神祕的代號出現，隨後，它被正式命名為 ASP.NET 5，經過不斷的完善和成熟，它最終以 ASP.NET Core 的名字亮相，”Core” 這個名字傳達了幾個重要資訊:

1.不僅是舊版本的升級，而是從零開始重新設計的框架。
2.核心精簡：它保留了最核心、最重要的功能，去除冗餘。
3.跨平台能力：Core 代表它的普遍適用性，就像 CPU 可以運行各種程式一樣。

但更重要的點是，它不再依賴 Internet Information Services（IIS），可以在世界的各個角落自由奔放，並且，擁有所謂的 Self-Hosting 能力，ASP.NET Core 可以像一個獨立的系統一樣運作。

ASP.NET VS ASP.NER CORE

傳統的 ASP.NET 與 IIS 的關係緊密，深度整合，ASP.NET 中的 System.Web namespace 包含許多直接呼叫 IIS API 的方法，，也就是 ASP.NET 應用程式實際上是寄居在 IIS 進程中的。

而 ASP.NET Core 相比之下，是可以完全獨立的，並且自帶裝備，擁有自己的 Web Server —— Kestrel，就像年輕人有了自己的小套房(?)，它的功能相對簡單，足以應對基本需求，但還是需要保護，通常還是會隔一層反向代理伺服器來增強安全性，而雖然 ASP.NET Core 可以獨立生存，但在 Windows 環境中，它仍然可以與 IIS 共處，實現上微軟提供了 UseIISIntegration 方法，讓 ASP.NET Core 可以與 IIS 整合，反過來說，IIS 也新增了 AspNetCoreModule 模組，負責 ASP.NET Core 的 Up and Down，而且可以監聽 ASP.NET Core 的狀態。

接著我們把鏡頭轉到 ASP.NET CORE 的邏輯，他是怎麼建構自己的 Host 呢?

Host

以一個簡單的 ConsoleApp來說，程式的執行起始點在 Program class 的 Main 方法

至於甚麼是 Host?

是一個封裝應用程式資源的物件，它：

1.應用程式的入口點：Host 作為應用程式的主要入口點，啟動和停止應用
2.依賴注入容器：包含並管理依賴注入容器
3.Config 管理：Host 負責載入和管理應用程式的 Config，包括從各種來源（如 JSON 檔案、環境變數、CLI參數等）讀取配置。
4.logger 管理：Host 設置和配置 Logger Provider
5.環境抽象：Host 提供了 IHostEnvironment 介面，使得應用程式可以確定當下的執行環境（如 Dev、QA、Prod等）。
6.生命週期管理：Host 管理應用程式的啟動和關閉過程，包括 graceful shutdown。

Generic Host 和 Web Host

Web Host (IWebHost)

是 ASP.NET Core 2.x 中使用的原始主機模型。
專門為 Web 應用設計。
使用 WebHost.CreateDefaultBuilder() 來創建。
配置通過 Startup Class 進行。

Generic Host (IHost)

ASP.NET Core 3.0 中引入。
設計用在任何類型的 .NET Core 應用，不僅限於 Web 。
使用 Host.CreateDefaultBuilder() 來創建。
可以通過 ConfigureWebHostDefaults 來配置 Web 特定設置。

WebApplication (最新)

在 .NET 6 中引入
結合了 Generic Host 的優點和 Web 應用程序的特定需求。
使用 WebApplication.CreateBuilder() 來創建。
簡化了配置過程，不再需要單獨的 Startup Class。
