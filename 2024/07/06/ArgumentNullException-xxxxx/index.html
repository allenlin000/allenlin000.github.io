Exception Handling (一. ArgumentException)

Created2024-07-06|Updated2024-12-15|C# 舞想
|Post Views:

定義

“拿到不該拿的資料時，拋出錯誤”

這個不該拿,可能是因為型別、可能是因為無資料等等

開發中，參數驗證是避免執行出錯的第一道防線。而選擇正確的例外類型能讓問題更清晰，並幫助後續的調試。

是不是說了等於沒說呢? 讓我們看看幾個實際的例子吧

核心主題一：參數驗證與例外處理

應該使用 ArgumentException 的例外情境

適用於：參數雖然傳遞了，但值不正確

Invalid User Input

檢查必填資料是否不合預期

Invalid Array Length
Invalid Enum Value

我們已經定義好可以進來的選項了，但奇怪的東西還是跑進來了

應該使用 ArgumentNullException 的情境

參數傳進來是 null 時

原因是我們有更精確的錯誤可以使用

若本來就不應該有 null 一直往下走，最好越早把驗證做掉，而不是在方法最內層才在檢查!

使用 ArgumentOutOfRangeException

參數傳進來在預期範圍之外

不該使用 ArgumetException 的情境

屬於使用者商業邏輯的一環

經常發生在使用者查詢資料時

傳入參數是”可選的”

Optional 參數，卻拋出 “某某資料未填寫完成!” 的錯誤 跟流氓有什麼兩樣呢

核心主題二：如何更靈活處理參數

OverLoading

若我們延伸想要問是，假設我們想要容許不同類型的參數傳入相同的方法呢?

沒錯，就是 OverLoading!

核心主題三：避免 NullReference Exception

若我們不允許傳入資料為 null 應該在方法被呼叫之前就做好檢查
而例如 string 這種型別的參數，本來傳入就還是有可能會是 null 的狀況下，仍務必要在傳入後也做檢查

但若是允許 Null ，更要因為你為甚麼接受引入 null，而作相對應的處理(就不是拋 ArgumentException 了)

Nullable

想明確 Handle 數值為”沒有資料”，而非使用 0 這種意義模糊的數值
可選的篩選

參數直接帶入預設值

Logging
設定 Timeout 時間 / 開關
遊戲角色建立

Inititialize 集合

好處就是可以因為引入沒有實例化的集合引發 NullException，不用每個人使用這個 Class 都要建立一次
而 Initialize 也有分宣告式的與建構子內，差別主要在是否需要做較為複雜的邏輯例如加一下判斷式的訊息等等

今日精神能量分析

精神能量 : 🏊

今天聽聞阿冠一家人花蓮遊玩因為蘇花公路不通打算游泳回台中，稍微研究了一下交通部氣象局的地圖，並分析了一下各段的情況

1.花蓮 - 台東

大概有 100 km，雖然可享受太平洋壯麗的景觀，但這邊的黑潮強勁且向北流動，阿冠一家會受到很大的阻力，但聽說黑潮是暖流，應該會滿溫暖喔

2.台東 - 墾丁

大約有 150 公里，這段路途更長，沿著東南部海岸線游泳，依然受到黑潮影響，而且水流更加複雜，容易被漩渦捲進去，就像沖馬桶一樣就被沖掉了

3.墾丁 - 高雄

約有 100 km，進入巴士海峽，所以有機會搭巴士，但搭不到就要繼續游，好消息是，這一段因為西南季風關係，水流是順向的

4.高雄 - 台中

約有 150 km，這一段台灣海峽俗稱黑水溝，所以阿冠可能會變黑，這點要特別注意，但因為夏天的西南季風幫助，阿冠有可能一下就隨著浪潮被帶回家了
