關於 null (一.)

Created2024-07-06|Updated2024-07-14|C#
|Post Views:

「這個變數拿到一個 null」

究竟 null 是什麼？這個概念一直以來都不太直觀。畢竟今天去買東西，你要的東西沒賣，店員總不會給你一個 null 吧

這篇討論主要是幫助我自己對 null 有更深入、更直觀的理解，讓這個 抽象的概念在心中變得更加具體

關於 value type 的 “無值”

value type 例如 : int, DateTime, bool…使用時應該有發現他不能被賦值為 null，為甚麼?

我們想一想，當一個數值被賦值為 “不存在” 是有意義的嗎

我們使用 int，先不論電腦世界的 0, 1，在我們的應用邏輯上，通常是數量、順序的概念，也就是有多寡、前後

例如一筆訂單

OrderId: 即使是 0 也是有意義的（可能代表系統中的第一個訂單）。
OrderDate: 一個具體的日期和時間。
Quantity: 訂單中商品的數量。0 是有意義的（可能表示空訂單）。
TotalAmount: 訂單總金額。0 也是有效值（免費訂單）。
IsShipped: 這是一個 bool，要麼是 true（已發貨），要麼是 false（未發貨）。

但..我們知道邏輯上是這樣沒錯，那我們實務上也需要防範他是 null 的狀況吧?

這就已經交給 .NET 的框架處理了，對於 .NET 來說 ，value type 都會有一個預設值，例如 bool 預設是 false

等等…那…例如折扣呢? 如果我們這筆訂單沒有折扣，讓他賦值為 null 會比 0 更有意義吧?

我認為，這就脫離數值的職責了，我們不應該讓數值去判定”有沒有”這件事，而且這讓我們看到 Discount = 0 時難以判斷這是折扣系統本來就有設計好 Discount = 0，還是說，這是一個失誤，我們不喜歡這種曖昧不清的關係(雖然有聽過一種說法是曖昧是最美好的時期)，因此應該修改為

接著，說到背後機制上，value type 的記憶體分配在 stack 上，且他不會有額外的記憶體來儲存 “是否為 null” 的資訊，也就不用判斷是不是要引用 heap 上的位址，以省去這個記憶體空間

那 strcut 呢? struct 內部也可以有 reference type 成員嗎? 那也可以為 null?

在 .NET ，struct 是一個 value type：

因此

struct 的 instance 整體是按照 value 複製的
struct 不能為 null（除非使用Null）。
struct 直接存在 stack 上

原因在於通常這是給輕、小量的資料建構用的，還是希望快，至於屬性成員包含 reference type 的話，例如 string，strcut 就會存取他的引用位置而不是值，如果為 null，就標記為不引用任何位置的值

null 在記憶體是怎麼運作的?

null 並不真正 “存在” 於記憶體中。它是一個概念，由 RunTime 來處理。

如果在 APP 中建立了幾個設為 null 的變數，每個變數本身佔用一定的記憶體空間來存它的 metadata（如 類型, 名稱）在 stack 中，變數被設為 null 時，它的引用值會被設置為一個特殊的值，這個值被 Runtime 識別為 “無引用” 的標記，所以也不占用 heap

當 Runtime 成功識別為 null 時，它會特殊處理這種情況。
例如，如果嘗試調用 null 的方法，執行環境會拋出 NullReferenceException，而不是嘗試訪問無效的記憶體位置。

以上是整理一下網上搜尋資料後對 null 的了解，其實在網路世界也有許許多多的討論，畢竟寫 Code 誰不碰 Null 呢?

結語

想像一下，null 就像是一個空盒子。這個盒子本身是存在的（就像變數本身），但裡面什麼都沒有。當我們說一個引用型別的變數是 null 時，就好比我們有一個標籤（變數名稱），但這個標籤並沒有貼在任何實際的物件上。
在記憶體中，null 通常被表示為一個特殊的位元模式，例如全零。但這只是實現細節，對於程式設計師來說，重要的是理解 null 的概念意義：它代表 ‘ 沒有引用任何物件 ‘。
