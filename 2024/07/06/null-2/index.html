與 null 打交道一段時間，想說也差不多可以先整理一點筆記

is null vs == null

對於要選擇使用哪一種，其實也多有討論，大概整理幾個點

== null:

如果不知道哪個帥哥 overide 了 == 運算符，可能會出事。
符合大多數開發者的直覺和習慣。

is null:

對不熟悉 is 運算符的開發者可能造成困惑。(我覺得應該只是統一性的問題吧!?)
不受 == 運算符重載的影響，總是進行引用比較。

但我個人很少看到重載 ==，而且通常 Class 是自己建的，我自己是偏好 is null

?.(Null-conditional operator)

C# 6.0 開始引入

Null-conditional operator 的引入，讓我們可以少包一層檢查

使用一段時間，整理起來，大概會用在這些場景

chaining

?? (null coalescing operator)

C# 2.0 就有

有時候可能在處理複雜的結構時會被討論是否過度簡化導致閱讀困難

或也可以

使用上要在簡潔性和可讀性之間保持平衡。

Nullable

Nullable 是在 C# 2.0 版本中引入的

上一篇有提到關於 value type 在 C# 本身是不能為 NULL，但這會有一些限制

Database 的資料

我們常會需要從資料庫中的取值，這時候就有可能為 null，但 C# 的 value type （如 int, double）不能為 null。

缺失的資料處理需求

在許多業務邏輯中，還是會需要區分 “沒有值” 和 “值為零”。
例如，一個人的年齡可以是 0（剛出生），但也可能是未知的（null）。

甚至我有遇過可能會需要對 API 的 bool 傳入 null 作為其中一個判斷情境的處理，所以為了因應各種狀況，還是會給工程師彈性的處理空間!
