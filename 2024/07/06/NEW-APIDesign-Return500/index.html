API 怎麼回報錯誤?

Created2024-07-06|Updated2025-05-09|深夜守護者
|Post Views:

針對統一包成成功 200 ，就算伺服器
屬於特規行為，討厭，而且不適每個人都技術員都會認真寫文件維護
看某些支援服務是否都用正常的 RESTFUL API

WebAPI 出錯時，是否必須透過 HTTP Status 反映執行結果，例如：找不到時吐 404、系統出錯時回應 500？

RESTful 派

設計 RESTful API 時，使用 HTTP status code 來表示請求的成功與否，意味著 client 端可以完全的靠這個 status code 來判斷該次請求是否成功，有些不好的 API 設計會造成 client 的誤解。

RESTful 可享受超多程式庫及工具的強力支援則是鐵一般的事實

出錯時傳 HTTP 500 可直接反映監控數據上，對 Infra 及 SRE 比較友善

Status Code 真的很重要，4開頭就是client的的問題，5開頭都是server端的問題，這樣可以少吵很多架

我認為有錯誤還是要使用非 2XX 的
對於 infra 或者 SRE 人員監控系統才能得知目前系統有異常，較容易在第一時間通報相關人員來查問題

<cache 套件>
如果有使用一些 cache 套件的話，預設行為有可能就是只 cache 200
(不論是 client side api cache 或是 server side response cache)
像 https://web.dev/learn/pwa/workbox?hl=zh-tw#using_workbox
這邊就有一個利用 200 response 來做 client side api cache 的範例
<swagger/api 開發介接>
產生 api 文件時可以直接從 200 知道介接時，”正向” 回應時的資料格式
400 則是 request 格式有問題(比方說 form 表單其實有些值的驗證不正確)
因此如果都回 200，才又在根據裡面的 status 去設定 response 格式
很容易讓 api 的格式變得很亂導致介接出現非預期的狀況
但如果是正向回應是 2XX
那就可以每隻 api 各自處理自己的 2XX
但其他類型的 response 回應就用通用的方式處理
<UI/UX>
如果 api 是 404，可能 user 不小心 key 錯網址(或者什麼原因導致 url 不正確)
這時也可以做全域的規劃設計，像是固定導到首頁
如果是 200，然後又要再看裡面的狀態值
那這種常見的預設行為就要自己重複造輪子
<資安/開發 Debug>
有些比較先進的 web 框架，其實會讓程式開發有運作環境
當程式的運作環境是開發環境時，會預設讓錯誤訊息直接清楚顯示到前端
像是連 db 有問題之類的訊息
但切換成營運環境運作時，則是回傳一個不含機敏資訊的 trace id
方便可以回 log 去對出來這個 request 是發生什麼錯誤
基本上營運環境要避免把資訊傳到前端(可能會需要 trade off)
而如果怕程式的運作環境有設定問題的話
可以靠一些 CI/CD 的機制，去避免程式運作環境的變數沒有設定對
<營運 Debug>
承上，當拿到 trace id
甚至是可以串接 ELK 這種 service，加快營運問題的排障
以及直接靠 browser 的 F12 工具，直接去 filter 500 的 response
快速找出服務壞掉的範疇，加快讓營運環境復原的速度
甚至是靠統計每周每月，非 200 的 response 有多少，又是哪些類型
來檢視系統的穩定度


restful 精神雖然不是非要遵循，但 follow 的好處太多了
真的是 Z > B
當然不 follow 有時也會有些好處，甚至也可能很大
但通常都是因噎廢食，常常都是因為 XY問題才不 follow
(也就是不 follow 可能真的解決了什麼燃眉之急，但卻治標不治本
不過可能也是要看公司政治政策因素，以及薪水給得怎麼樣啦)
如果是新系統，沒有包袱，遵循 restful 可能會是最沒有風險的做法
如果是既有系統，那可能還是看前人怎麼做風險比較低 XD

我是認為，200、404、500這一類的error屬於http通訊協定的回應代碼，所以，算是”通訊”階段的狀態回應，而真正AP的處理結果，應該去看AP回應訊息裡面的error code才對…
不然，把通訊層跟AP層的error code混在一起，你不一定會知道系統到底死在哪一層…

我傾向錯誤訊息對外給 個 GUID 或流水序號，待使用者回報再跟內部 Log 對映

大部分的弱點掃描分析也會掃出如果你想把內部的exception丟出去到外界這件事情 這其實是很不好的偷懶習慣 前端只要知道系統內部有錯即可 不應該有細節 而且http的status code嚴格說來是L7以上的 不是通訊層 而本來就是某個超文本或者資源的狀態

REST 追求與 HTTP 協定融合而為一，用 URI 定位資源，HTTP 動詞指定動作，回應結果也直接用 HTTP Status 表示

應該說 restful api本來就是在操作某個http負責傳輸的資源 而且每次呼叫都是獨立的 不該有狀態存在 所以所有的status code就是http status code, 成功就是2xx , 導向就是3xx, 存取有問題就是4xx , 內部錯誤就是5xx

比方說:
我們也可以規定綠燈禁止通行, 紅燈才能通過, 只要大部分的人都遵循即可.
規則是人訂的. 至於符合對方的常識又是另外一回事了.

我是500派的
雖然client兩種方式都能接
但是你伺服器出錯就出錯
幹嘛騙人說200然後再遞紙條
說拍謝其實有出錯？

請教一下如果你提供給客戶的資訊的部分仰賴三方來的資料，這點合約上怎麼說明並告知可能會失效不可控的風險？
