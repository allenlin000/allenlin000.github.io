Asynchronous Programming (二. 從 Callback Hell 角度出發)

Created2024-07-25|Updated2024-07-25|Asynchronous Programming
|Post Views:

callback hell

假設我們今天沒有 async, await, Task 的概念，我們可以怎麼寫一個做早餐的異步程式?

定義做早餐的主流程，先燒開水，一旦燒完其他就可以一起開始進行不互等

模擬做早餐各步驟，每個方法都須帶上 callback function

執行

這就是土炮的寫出，每個任務都帶上 callback

但這樣的寫法會有幾個缺點 :

  1. Code 很複雜難讀
  2. 錯誤處理和異常情況的管理困難。
  3. 如果需要增加更多並行任務， Code 又更複雜了。

這種寫法很容易讓開發者陷入一個難以維護和擴展的困境。那麼，有沒有更好的方法來處理這種複雜的異步操作呢？這就引出了程式設計中 “任務” 的概念。

任務

任務的概念就像管理一個項目，你可以知道這個任務是 “正在進行中”、”已完成” 還是 “出錯了”，拿到一個狀態會自己告訴你，而不需要一直盯著它，並且你也可以決定哪些任務可以同時開始，可能因為人力或資源彼此不相依，又或者，哪些需要等其他任務完成後才能開始

不同的語言用會有 keywords 來描述這個概念：

C# 使用 “Task”
JavaScript 使用 “Promise”

Task、ContinueWith

Task 是 .NET 框架中用於表示異步操作的物件。可以將 Task 想像成一個代表「未來某個時間點會完成的工作」的容器。它是 .NET 中實現並行和異步的核心元素之一。

主邏輯改寫，我們讓燒開水方法執行後回傳一個 Task，包含目前進行狀態的訊息，接著烤麵包、煮咖啡、煎蛋在得到燒開水的成功完成的狀態後就可以同時開始執行
這邊使用 OnlyOnRanToCompletion 來表示，燒開水後續的任務必须在燒開水完成狀態才能执行

模擬各自實現

執行結果

雖然 Task 和 ContinueWith 確實能夠處理非同步操作，但 async/await 的引入帶來了許多重要的改進，我們首先用同一個例子試著改用 async, await 寫一遍

async await

主邏輯修改，非同步方法 就掛上 async 並在內容實作 await，而要一起執行並等待完成的使用 await Task.WhenAll()

模擬做早餐任務，美個任務都有一個 await 斷點，遇到這個斷點就會回到主程序並告訴主程序現在的狀態是甚麼，等到 await 的內容完成，就會再次從 await 下一步繼續走，可以這樣是因為狀態機已經記住剛才這個非同步方法執行到哪裡了

執行結果

從這一段改變我們可以看到邏輯流程更清晰，減少了 Nested 和 callback 造成的複雜性

而 async, await 還有一些特點，舉例

1.await 會幫你自動拆包例外

2.當回傳任務結果時，只要直接 return Result，Compiler 替你包裝成 Task of Result

或者是

在 async 方法中，可以直接返回預期的類型（在這個例子中是 string），而不需要顯式地把他包裝在 Task 中。
