Cookie

Created2024-07-28|Updated2024-08-04
|Post Views:

Stateless

HTTP 本身是一種無狀態的協議，也就是 stateless，表示每個 Request 都是獨立的，並不會在多個 Request 之間保留任何狀態資訊，假設購物流程是這樣:

1.登入
2.將商品加入購物車
3.結帳
4.顯示訂單紀錄

這背後都代表一個行為操作的 API，而若這個流程保持無狀態，表示彼此之間並沒有關連性，結帳時，他無法知道是誰結帳

然而,在實際的購物流程中,我們確實需要在不同的請求之間維護狀態,以便 Server 知道是哪個用戶在執行這些操作。這就是為什麼我們需要使用一些機制來保存狀態，而在瀏覽器設定 Cookie 就是保留狀態的一種方法

Cookie 的運作

  1. 當用戶登錄成功後，Server 建立一個 Session，並將Session ID 通過 “Set-Cookie” Header 發送給 Browser。Browser 將這個 Cookie 存起來，並在後續的 Request 中將其發送回 Server，Server 就可以識別這個用戶並維護他的登入狀態。

  2. 當用戶將商品加到購物車時，Server 可以將購物車的資訊存在 Server 端，並將一個關聯的 Cookie (如購物車 ID) 發送給 Browser。Browser 將這個 Cookie 存起來，並在後續的 Request 中將其發送回 Server，Server 就可以識別這個用戶的購物車。如果用戶在添加商品到購物車後登入，Server 可以將購物車與用戶的賬號關聯起來。

  3. 服務器可以將用戶的偏好或行為數據存在 Cookie 中，並在後續的 Request 中讀取這些 Cookie, 以提供個性化的推薦或廣告。例如，如果用戶經常瀏覽某個類別的商品，你可以將這個資訊存在 Cookie 中，並在用戶下次訪問時顯示這個類別的推薦商品。

以上是常見的應用方式

RFC 文件

參考 : https://github.com/aszx87410/blog/issues/45

Huli 幫大家導讀了 RFC 2109、RFC 2965、RFC 6265，另外所謂 RFC 是由網際網路工程任務組 (IETF) 發布的一系列備忘錄。檔案收集了有關網際網路相關資訊，只有 IETF 的發布管道才表示 IETF 對該協議的規範進行了審核並達成共識。

整理重點如下:

RFC 2109

  1. 介紹了兩個新的 header：Cookie 跟 Set-Cookie，藉由這兩個 header 在 server 與 user agent（通常就是指瀏覽器）之間傳遞資訊，簡單來說就是引入 Cookie 與 Set-Cookie 兩個 Header 來建立 Session，

  2. 如果 Server 需要的話，它可以初始化一個 session，而初始化的方法是回傳一個 Set-Cookie 的 Header，若是瀏覽器決定繼續這個 session 的話，可以回傳 Cookie 這個 Header

  3. 存在 cookie 裡的 session 資訊可以是明文或是編碼過後的文字，但如果存的東西太多會變得太笨重。所以，你可以選擇只在 session 資訊裡放一個對應到伺服器資源的 key。但這種方式會造成一些這個狀態管理機制本來就想避免掉的一些問題，主要是：

    1. 把狀態放在 server
    2. 如何以及何時把沒有用的狀態資訊清掉

Cookie-based session 以及 SessionID，前者的缺點就是存太多東西會變得笨重，後者則是需要把狀態放在 Server。
而比較常使用的還是 SessionID 那種方式

安全性問題

  1. Secure 這個屬性限制了 Cookie 只能在「安全」的管道上傳輸（安全的定義由 user agent 自己定義）。當一個 Cookie 有了 Secure 這個屬性，只有當 Request 在安全的管道（通常指的是 TLS）中傳輸時才會把 cookie 放進 HTTP Request 裡面。
  2. 一般我們所認知的「Secure 就是代表一定要 HTTPS 才會被傳送」其實指的是主流瀏覽器的實作，而不是 RFC 的規範。
  3. 設置 Secure 屬性代表這個 Cookie 只能透過 secure 的管道被傳輸，至於什麼是 secure，RFC 上寫說由瀏覽器自行定義。依據目前主流的實作，就是指只能透過 HTTPS 來傳送
  4. cookie 不是唯一能追蹤使用者的技術，但因為它可以在不同 host 被共享的特性，的確促進了這種追蹤的行為。
  5. 在渲染 HTML 頁面時，瀏覽器常會發送一些 Request 去其他的 Server（例如說廣告商的伺服器），所以儘管這些使用者從來沒有直接造訪這些網站，這些網站可以利用 Cookie 來追蹤使用者，就是所謂3方 Cookie
  6. 限制第三方 Cookie。舉例來說，有些不發送 Cookie header 給第三方，有些則是不處理第三方的 Set-Cookie header。
  7. Cookies 對 subdomain 並不具有完整性。舉例來說，foo.example.com 可以對 example.com 設置 cookie，而這個有可能把 bar.example.com 對 example.com 設置的 cookie 給蓋掉。最糟的情況下，當 bar.example.com 收到這個 cookie 時，區分不出是自己設置的還是別人設置的。foo.example.com 就可以利用這個特性來攻擊 bar.example.com。
  8. 攻擊還可以利用 http://example.com/ 來把 https://example.com/（前者是 http，後者 https）的 cookie 蓋掉，server 就無法分辨這個 cookie 是 http 還是 https 設置的。攻擊者一樣可以利用這個特性來進行攻擊。

三方 cookie 是甚麼意思?

它不是來自你正在瀏覽的網站，而是來自其他第三方網站

比方說，假設你正在瀏覽一個網購網站，上面有一個廣告欄位。這個廣告欄位是由另一家廣告公司提供的，當你進入該網站時，這家廣告公司也會在你的瀏覽器上放置一個 Cookie。這個 Cookie 就是所謂的三方 Cookie。

所以，在我們上一段所提到的 “ 限制第三方 Cookie 可以透過不發送 Cookie header 給第三方，有些則是不處理第三方的 Set-Cookie header” 這件事，就是這個意思!

Cookie Same Origin Policy

Domain 跟 Path 與 Cookie 一致，就會被視為同源．拜訪該網站時，Cookie 就會被送出
而 Scheme 的部分，則是要看 Cookie 是否有加入 secure 這個屬性，若是有加上 secure，便會判斷是否是 Https 才會送出。

比如說，假設你在 www.allen.com 這個網站上登入了，此時網站會在你的瀏覽器上設置一個 Cookie, 內容可能包括你的帳號資訊、session ID 等。
現在，當你瀏覽 www.allen.com/products 時，由於 Domain 和 Path 都與之前設置 Cookie 的網頁一致，所以這個 Cookie 就會被自動發送給 Server，讓網站知道你是已登入的使用者。
反之，如果你瀏覽 www.jacky.com, 雖然也可能是一個登入網站，但因為 Domain 不同，所以這個 Cookie 就不會被發送，網站也無法辨識你的身分。

但經過設定後，subDomain & Domain 間可以共用 Cookie，例如設置 Cookie 時，可以在 Cookie 的 Domain 屬性中指定一個較寬泛的域名，例如 .example.com，這樣不論是 www.example.com 還是 subdomain.example.com，都能訪問同樣的 Cookie。

SameSite

目的

Cookie 的 SameSite 屬性是在 2016 年由 Google 提出的，目的是為了解決跨站腳本攻擊 (Cross-Site Scripting, XSS) 和跨站請求偽造 (Cross-Site Request Forgery, CSRF) 等安全問題。

CSRF 通常是惡意網站利用瀏覽器自動帶 cookie 的機制，偽造了一個使用者自己的請求，從而在用戶不知情的情況下，對網站 X 進行惡意操作。

例如發一個圖片、連結給使用者點擊，而這個點擊的背後有埋下對原始網站進行某些操作，像是提交一個表單把使用者的帳款轉帳給駭客
參考 : https://blog.techbridge.cc/2017/02/25/csrf-introduction/

SameSite的定義

我們要防衛的是 “Cross Site” 這件事，而假如是從 “瀏覽器” 這方面去著手的話，就會談到 Cookie 的 SameSite 屬性了

所謂的 SameSite 定義是:
1.有相同的 scheme , 例如 http 或 https
2.有相同的 domain ，例如 allen.com, xd.com

不需要有相同的 port 或 subdomain

設定的做法會像是這樣

SameSite 類型

SameSite=Strict

最嚴格

當設定為 Strict，瀏覽器只會在完全符合 SameSite 時才會發送 Cookie。

所以這些情境無法發送 Cookie :

1.從其他 Domain 點擊一個連結進入你的網站
2.其他網站中的 form post 到你的網站
3.其他網站使用 JavaScript 的 AJAX 或 fetch Request
4.網站被嵌入到另一個不同 domain 的網站的 iframe 中
5.其他網站的 image 中 src 屬性 放入你的網站連結

所以這樣可以有效阻擋 CSRF 攻擊，但可能會影響某些需要跨站請求的功能，例如單點登入(SSO)。

SameSite=LAX

預設

LAX 相比於 Strict，釋放了一些限制:

1.使用者從其他 domain 點擊一個連結進入你的網站
2.其他網站中的表單，如果使用 GET 方法提交資料到你的網站

因為這些請求被認為相對較為安全。它們通常是由使用者直接觸發的，而不是由其他網站以程式化的方式觸發的。

所以某些情境下的使用者體驗可能會獲得改善，例如:

使用者從 email、facebook 等外部渠道點擊連結進入您的網站。
在 Strict 模式下，由於這是跨站請求，Cookie 不會被發送，導致用戶顯示為未登入狀態。
切換到 Lax 後，這種「Top Level 的 GET Request」也會發送 Cookie, 從而保持用戶的登入狀態，提升用戶體驗。

但是若這些機制還是需要靠跨站的 form post 實現，則還是會被阻擋

SameSite=None

就是基本上不設定 SameSite的限制了，所以是相對風險最高的

總結

整理這篇主要是整理了 Cookie 在 Web 應用程式中的重要作用，以及相關的安全性概念和機制。

1.HTTP 是一種無狀態的協議，為了在不同請求之間保持狀態訊息，Cookie 的機制應運而生。網站可以透過 Cookie 記錄用戶的登入狀態、購物車內容等，提供更好的用戶體驗。

2.然而，方便性也同時帶來了一些風險，Cookie 的濫用可能發生 Session Hijacking 等攻擊，網站需要對用戶輸入進行適當的驗證和過濾，防範 XSS 攻擊的發生。

3.Cookie 的 Same-Origin Policy 規定了 Cookie 的作用域，只有在請求的 Domain 和 Path 與設置 Cookie 時一致時，才會自動發送 Cookie。

4.Cookie 還有 Secure 屬性，限制只能在安全 (HTTPS) 的連線中發送 Cookie。

5.三方 Cookie 指的是來自非當前訪問網站設置的 Cookie, 這些第三方 Cookie 可能會被用於跟蹤用戶行為。

6.SameSite 屬性是為了解決 CSRF 攻擊而引入的，有三種模式：Strict、Lax 和 None。不同模式下 Cookie 的發送規則有所不同，需要權衡安全性和功能性。
