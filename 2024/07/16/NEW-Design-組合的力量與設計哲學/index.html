繼承的迷思與困境

Created2024-07-16|Updated2025-04-23|程思舞想
|Post Views:

在上一篇文章中，我們從克德族的故事看見了繼承帶來的限制與設計陷阱。這一篇，我們將把鏡頭轉向另一個更自由、更靈活的設計哲學 —— 組合（Composition）。

Composition 很像是在拆解、集中的感覺，有種明確分類職責的味道

用組合代替繼承，解決行為歧義（Ambiguity of Behavior）

讓戰鬥法師「擁有」戰士和法師的能力，而不是「繼承」它們。

用 Interface 來分離行為，避免多繼承，解決屬性重複（Duplicate State）

讓 Warrior 和 Mage 各自實現自己的行為，但把狀態留給 Character，並且避免通過多繼承直接繼承狀態。

這樣屬性只存在於 Character 中，行為分開，避免了屬性重複的問題。

用組合與單一職責原則（Single Responsibility Principle）解耦處理功能耦合過緊的問題

這樣每個 Class 專注於自己的功能，CompressedFile 可以靈活組合功能而不需要多繼承。

當有新功能需求時，我們不需要修改現有的 Class 。

我們可以建一個新的 Class 來實現這個功能，然後將它提供給需要使用的物件。這種方法允許我們動態地增加物件的能力。

清晰的界限與減少耦合，不用依賴 protected 或 virtual。

將可能變化的部分封裝起來

這樣，當我們需要修改或擴展這些可變部分時，不會影響到那些保持不變的部分。

Composition 的挑戰

重複實作行為

每個播放器類都需要實現相同的方法（Play, Pause, Stop, SkipTrack），導致了大量的重複 Code。

C# 的應對方式：Default Interface Methods（C# 8 起）

那這些問題在 C# 中會有嗎?

首先我們知道，C# 不支援多重繼承，但可以實作不同的 Interface，Interface 通常只定義方法簽章。

但當我們想要在 Interface 新增新的功能時，所有有實作的實體都會被影響到

所以從 C# 8 開始，提供 Default interface Methods ，可以在 Interface 寫 method Body，這個 Default Methods 不會影響到現有的實作 instance，若現有的實作想要調用，也可以自己 override 實作

使用

MP3Player 和 FLACPlayer 可以選擇性的覆蓋某些方法，否則，就無法直接用這個實體調用這個方法

並且，共同的功能（如 Play、Compress）在 Interface 定義一次，可以被多個 Class 重用。
每個 Interface 都專注於特定的功能集，做到 Interface 隔離

相比於傳統的繼承關係， Interface 提供了類似多重繼承的功能，但沒有傳統多重繼承的複雜性。
避免了菱形問題，因為每個方法都有明確的來源。

當我們考慮使用繼承時，我們應該轉換思維方式，將焦點放在物件的行為和功能上，而不僅僅是它們的結構。
我們需要深入理解 Parent Class 的設計意圖，以及使用者對這些 Class 的期望。這種思考方式自然會引導我們朝向更靈活的設計方向。

具體來說，我們可以考慮思考這些關鍵字 : Interface, Combination, SOLID, Contract… 來協助我們設計更好的程式!
