從克德族說起：繼承為何讓人生卡關？

Created2024-07-16|Updated2025-04-23|程思舞想
|Post Views:

好久好久以前，有兩個著名的家族——

克德族，是一個信奉血統主義的家族。他們深信純正血脈的力量，嚴格遵循世代傳承的家族傳統，認為唯有優良基因的延續，才能孕育出最優秀的後代。起初，這種方式的確造就了不少傑出人才。克德家的子女大多具備出色的音感，未來幾乎都走上了音樂的道路。

但隨著時間流逝，問題逐漸浮現。某一天，家族中一位孩子心懷成為機械技師的夢想，卻發現家族的傳統成了束縛。他無法脫離音樂的陰影，也難以獲得從事機械所需的技能，陷入深深的痛苦與迷惘。

海的另一端，是萊雅族。這是一個秉持開放思想的家族。他們相信卓越不僅來自血統，更來自個人的選擇與學習。他們鼓勵每位成員自由探索，組合各種不同的技能與知識。

這樣的理念帶來了極大的彈性。萊雅族的子女可以依興趣學習，不論是音樂、機械或其他領域皆不受限制。他們甚至可以同時成為音樂家與技師。

然而，自由也帶來困惑。因選擇過多，部分成員在成長過程中感到迷惘，沒有明確的方向。不僅如此，因為每個人都走著截然不同的路，外人甚至難以辨別他們是否屬於同一族群。

克德族的困境，就像物件導向中的繼承（Inheritance）。子類別（Subclass）往往難以擺脫父類別（Parent Class）賦予的行為與限制。明明想走自己的路，卻被迫實作那些與自身不相干、甚至不合時宜的方法，最後只能無奈地回傳 null、false，或乾脆拋出 NotImplementedException —— 彷彿也在拋下自己的夢想與初衷。

而萊雅族的自由，則象徵著組合（Composition）的靈活與挑戰。你可以自由注入不同的依賴、拼湊出多元的能力，但若缺乏方向與設計哲學，最終可能落得類別之間的耦合雜亂無章，像是一場技能拼貼的狂歡，卻少了意義與核心。

Inheritance 之美

Inheritance 為我們帶來的好處就是可以收攏相關的 Class，並且減少重複的 Code，實現多型 (polymorphism)
(clearly related reusable pieces of code that fit under a single common concept)

例子有 偵聽者（listner）、工廠（factory）模式

Inheritance 的問題與陷阱

Diamond problem

如果專案開始發展的多樣且複雜，如果謹守於使用繼承來解決問題，就有可能遇到多繼承困境，那將會是複雜與不可預期的，俗稱 Inheritance hell，例如 diamond problem

Base Class A
B、C 都繼承 A
D 繼承 B 和 C

感覺出問題惹嗎?

歧義在這，如果 A 定義了一個方法，而 B 和 C 都重寫了這個方法，那麼 D 應會去繼承誰的方法？
重複繼承：A 的屬性和方法可能會在 D 中被重複繼承兩次，一次通過 B，一次通過 C。
初始化順序：在創建 D 的實例時，A 的構造函數應該被調用一次還是兩次？

好像還是有一點抽象喔，來點實際的案例吧

當 Class 之間的關係被誤解為「純粹的 is-a」

今天開發一個遊戲戰鬥系統，描述「角色」的不同行為

Base Class A：角色（Character）定義所有角色的通用屬性（如 Name、Health）。

SubClass B：戰士（Warrior）加上近戰攻擊行為。
SubClass C：法師（Mage）加上魔法攻擊行為。
SubClass D：戰鬥法師（Battlemage）戰鬥法師既是戰士又是法師!

感覺很像楓之谷之類的設計欸

這時候設計者想說戰鬥法師應該同時繼承戰士和法師的行為，因此 D 繼承了 B 和 C，就會發生

  • 行為歧義：Character 定義了一個方法 Attack()，Warrior 和 Mage 都重寫了這個方法，那麼 Battlemage 繼承的是 B 的 Attack() 還是 C 的？這導致程式邏輯的不確定性。
  • 屬性重複：Character 有一個屬性 Health，那麼 Battlemage 會從 Warrior 和 Mage 中各繼承一份 Health，資料有可能會亂掉。
  • 初始化順序問題：Battlemage 的實例化會兩次初始化 Character 的構造函數，因為 Battlemage 通過 Warrior 和 Mage 分別間接繼承了 Character。

假設你問爸爸和媽媽一個問題：「晚餐吃什麼？」

爸爸說吃摩斯，媽媽說吃麥當勞，那要聽誰的？

多重責任導致類別設計混亂

在一個 Class 需要同時承擔多個角色或功能時，開發者可能嘗試透過多繼承來整合多個 Base Class。

建立一個文件系統模型：

Base Class A：可儲存的（Storable） 提供保存與載入的功能。
Base Class B：可壓縮的（Compressible） 提供壓縮與解壓縮的功能。
SubClass D：壓縮文件（CompressedFile）同時需要保存與壓縮功能。

所以開發者想說那就寫個 D : Storable, Compressible 吧，讚!

問題來了

  • 功能耦合過緊：如果某些 Class （例如 D）需要另一組功能（例如 Encryptable），開發者是要繼續找一個這種 Base Class 繼承然後吃下他所有的性質嗎?
  • 方法歧義：如果 Storable 和 Compressible 都定義了方法 Save()，那麼 D.Save() 的實現會來自哪個 Base Class？

結果來說還會導致

  • 可測試性降低：多重 Base Class 會使得 SubClass 變得難以測試，因為 SubClass 不僅耦合於 Base Class 的行為，還耦合於 Base Class 之間的交互邏輯。

多繼承試圖將不同的功能塞到一 Class 中，導致 SubClass 變成「一切的集合」，失去靈活性。如果每增加一個功能（如加密功能），就必須引入新 Class，會讓設計越來越複雜，且難以維護。

就像你有一個全能的背包，裡面可以裝書、筆電、裝餅乾、…。每次加東西就持續擴充背包。最後這個背包變得非常重，不僅難用，壞掉時還得重做一個。

無法將 parent class 轉為 subclass 使用

考慮一個公司的員工系統

然而有天 Allen 晉升了

如果 Allen 被晉升為經理，我們不能簡單地將 Employee 轉換為 Manager。我們需要創建一個新的 Manager 對象，並將 Allen 的資料拷貝過去。

可以這樣寫，把升職的邏輯封裝在 PromoteToManager 方法中

SubClass 被迫實作不適用的方法

假設我們有一個 HomeAppliance，然後我們想要建不同類型的家用電器作為 subClass。

HomeAppliance 定義了所有家用電器可能有的基本功能：開關機、設置定時器和連接 WiFi。

Refrigerator 繼承自 HomeAppliance，但它不需要定時器功能，也可能不支持 WiFi 連接（取決於型號）。因此，它必須重寫這些方法並拋出異常，表示不支持這些功能。

Microwave 實現了定時器功能，因為這對微波爐來說是常見的。
SmartTV 實現了定時器和 WiFi 功能，因為這些是智能電視的常見特性。

這個例子突出了繼承的一個問題：subClass 被迫繼承了它可能不需要或不適用的功能。在實際開發中，就違反里氏替換原則，因為實作了一些讓子類別的行為偏離父類別的預期，導致父類別的功能在子類別中表現異常，要馬爸爸的要求不對，要馬小孩不該屬於這個爸爸的 (嗯?)
