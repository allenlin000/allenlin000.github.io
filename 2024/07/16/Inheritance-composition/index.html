Inheritance & Composition

Created2024-07-16|Updated2024-07-18|OOP
|Post Views:

好久好久以前，有兩個著名的家族：血統主義的克德族，深信純正血統的力量。他們嚴格遵循家族傳統，認為只有通過世代相傳的優良基因，
才能培養出最優秀的後代。起初，這種方法確實產生了一些傑出的人才。克德家族的子女大多擁有絕佳的音感，未來都大致上都是朝音樂領域深耕

然而，隨著時間推移，問題開始浮現。有一天，克德族的一個孩子夢想成為一名機械技師。但是，家族的傳統限制了他的發展。他既無法完全擺脫音樂的影響，也無法輕易獲得成為技師所需的技能，這給他帶來了極大的統苦與迷惘…

另外一邊是萊雅族，他們採用了更為開放的方式。他們相信，卓越不僅來自血統，更來自於個人的學習和成長。這個家族鼓勵其成員自由選擇並組合不同的技能和知識。
這種方法帶來了極大的靈活性。組合家族的成員可以根據自己的興趣和社會需求，靈活地學習各種技能。一個人可以同時成為音樂家也可以學習機械
而這種自由也帶來了挑戰。家族成員因為選擇太多而感到迷茫，沒有一個可以遵循的法則，人生要怎麼走完全要靠自己思考出路，有時不知道該專注於哪些技能。

這比喻雖然不是絕佳，但對於理解繼承與組合的優劣我覺得想像起來滿有趣的

克德族的繼承困境：subClass 無法輕易改變或拋棄 Parent Class 的特性，導致靈活性降低，克德族的繼承困境：subClass 繼承了一些必須要做的事情，但因為不適合最後必須 return null, return false

萊雅族的過度自由，如同使用 combination 可能導致的問題： Class 變得過於複雜

再來我們分段落描述各自的好與壞

Inheritance 好處

Inheritance 為我們帶來的好處就是可以收攏相關的 Class，並且減少重複的 Code，實現多型(polymorphism)，這也是基礎的 OOP 會學習到的概念
(clearly related reusable pieces of code that fit under a single common concept)

設計模式裡的偵聽者（listner）、工廠（factory）模式等會使用這個概念

Inheritacne 壞處

1.如果專案開始發展的多樣且複雜，有可能遇到多繼承，那將會是複雜與不可預期的，俗稱 Inheritance hell，如 diamond problem

圖示

Base Class A
B、C 都繼承 A
D 繼承 B 和 C

就會有幾個問題

歧義：如果 A 定義了一個方法，而 B 和 C 都重寫了這個方法，那麼 D 應該繼承哪個版本的方法？
重複繼承：A 的屬性和方法可能會在 D 中被重複繼承兩次，一次通過 B，一次通過 C。
初始化順序：在創建 D 的實例時，A 的構造函數應該被調用一次還是兩次？

這種會讓人狀態不明確的情境應該要避免!

2.倘若未來有需求，我們無法將 parent class 轉為 subclass 使用

舉例

考慮一個公司的員工系統

然而有天 Allen 晉升了

如果 Allen 被晉升為經理，我們不能簡單地將 Employee 轉換為 Manager。我們需要創建一個新的 Manager 對象，並將 Allen 的資料拷貝過去。

3.後續想新增功能，不敢改 parent class 因為影響範圍會很大

4.subClass 被迫繼承了它可能不需要或不適用的功能

例子:
假設我們有一個 HomeAppliance，然後我們想要建不同類型的家用電器作為 subClass。

HomeAppliance 定義了所有家用電器可能有的基本功能：開關機、設置定時器和連接 WiFi。

Refrigerator 繼承自 HomeAppliance，但它不需要定時器功能，也可能不支持 WiFi 連接（取決於型號）。因此，它必須重寫這些方法並拋出異常，表示不支持這些功能。

Microwave 實現了定時器功能，因為這對微波爐來說是常見的。
SmartTV 實現了定時器和 WiFi 功能，因為這些是智能電視的常見特性。

這個例子突出了繼承的一個問題：subClass 被迫繼承了它可能不需要或不適用的功能。在實際開發中，這可能導致以下問題：

製造混沌：subClass 中充斥著不必要的方法重寫。
違反里氏替換原則：拋出 NotSupportedException 的方法違反了這個原則。
誤用風險：使用者可能會嘗試調用不支持的方法。

這就是典型的 Circle-Ellipse problem

Composition 好處

1.當有新功能需求時，我們不需要修改現有的 Class 。我們可以建一個新的 Class 來實現這個功能，然後將它提供給需要使用的物件。這種方法允許我們動態地增加物件的能力。

2.清晰的界限與減少耦合，不用依賴 protected 或 virtual。

3.將可能變化的部分封裝起來。這樣，當我們需要修改或擴展這些可變部分時，不會影響到那些保持不變的部分。

Composition 壞處

可能每個class 需再實做一次相同的行為 尤其是相似的功能

每個播放器類都需要實現相同的方法（Play, Pause, Stop, SkipTrack），導致了大量的重複 Code。

C#

那這些問題在 C# 中會有嗎?

首先我們知道，C# 不支援多重繼承，但可以實作不同的 Interface，Interface 通常只定義方法簽章。

但當我們想要在 Interface 新增新的功能時，所有有實作的實體都會被影響到

所以從 C# 8 開始，提供 Default interface Methods ，可以在 Interface 寫 method Body，這個 Default Methods 不會影響到現有的實作 instance，若現有的實作想要調用，也可以自己 override 實作

使用

MP3Player 和 FLACPlayer 可以選擇性的覆蓋某些方法，否則，就無法直接用這個實體調用這個方法

並且，共同的功能（如 Play、Compress）在 Interface 定義一次，可以被多個 Class 重用。
每個 Interface 都專注於特定的功能集，做到 Interface 隔離

相比於傳統的繼承關係， Interface 提供了類似多重繼承的功能，但沒有傳統多重繼承的複雜性。
避免了菱形問題，因為每個方法都有明確的來源。

當我們考慮使用繼承時，我們應該轉換思維方式，將焦點放在物件的行為和功能上，而不僅僅是它們的結構。
我們需要深入理解 Parent Class 的設計意圖，以及使用者對這些 Class 的期望。這種思考方式自然會引導我們朝向更靈活的設計方向。

具體來說，我們可以考慮思考這些關鍵字 : Interface, Combination, SOLID, Contract… 來協助我們設計更好的程式!
