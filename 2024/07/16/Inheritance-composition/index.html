Inheritance & Composition

Created2024-07-16|Updated2024-12-29|Design Pattern
|Post Views:

好久好久以前，有兩個著名的家族：

血統主義的克德族，深信純正血統的力量。他們嚴格遵循家族傳統，認為只有通過世代相傳的優良基因，才能培養出最優秀的後代。
起初，這種方法確實產生了一些傑出的人才。克德家族的子女大多擁有絕佳的音感，未來都大致上都是朝音樂領域深耕

然而，隨著時間推移，問題開始浮現。有一天，克德族的一個孩子夢想成為一名機械技師。
但是，家族的傳統限制了他的發展。他既無法完全擺脫音樂的影響，也無法輕易獲得成為技師所需的技能，這給他帶來了極大的痛苦與迷惘…

另外故事的這一頭是萊雅族，他們採用了更為開放的方式。他們相信，卓越不僅來自血統，更來自於個人的學習和成長。這個家族鼓勵其成員自由選擇並組合不同的技能和知識。
這種方法帶來了極大的靈活性。組合家族的成員可以根據自己的興趣和社會需求，靈活地學習各種技能。一個人可以同時成為音樂家也可以學習機械
而這種自由也帶來了挑戰。家族成員因為選擇太多而感到迷茫，沒有一個可以遵循的法則，人生要怎麼走完全要靠自己思考出路，有時不知道該專注於哪些技能，重點是你可能還看不出來他們是一個種族的!

回到程式設計

克德族的繼承困境就像是 subClass 無法輕易改變或拋棄 Parent Class 的特性，導致靈活性降低，因為 subClass 繼承了一些必須要做的事情，但因為做不出來也不適合最後必須 return null, return false，來敷衍了事，也敷衍了人生

萊雅族的過度自由，如同使用 combination 可能導致的問題，也就是 Class 變得過於複雜

Inheritance 好處

Inheritance 為我們帶來的好處就是可以收攏相關的 Class，並且減少重複的 Code，實現多型(polymorphism)，這也是 OOP 會學習到的概念
(clearly related reusable pieces of code that fit under a single common concept)

設計模式裡的偵聽者（listner）、工廠（factory）模式等會使用這個概念

Inheritacne 的問題

1.如果專案開始發展的多樣且複雜，如果謹守於使用繼承來解決問題，就有可能遇到多繼承困境，那將會是複雜與不可預期的，俗稱 Inheritance hell，例如 diamond problem

Base Class A
B、C 都繼承 A
D 繼承 B 和 C

感覺出問題惹嗎?

歧義在這，如果 A 定義了一個方法，而 B 和 C 都重寫了這個方法，那麼 D 應會去繼承誰的方法？
重複繼承：A 的屬性和方法可能會在 D 中被重複繼承兩次，一次通過 B，一次通過 C。
初始化順序：在創建 D 的實例時，A 的構造函數應該被調用一次還是兩次？

好像還是有一點抽象喔，來點實際的案例吧

當 Class 之間的關係被誤解為「純粹的 is-a」

今天開發一個遊戲戰鬥系統，描述「角色」的不同行為

Base Class A：角色（Character）定義所有角色的通用屬性（如 Name、Health）。

SubClass B：戰士（Warrior）加上近戰攻擊行為。
SubClass C：法師（Mage）加上魔法攻擊行為。
SubClass D：戰鬥法師（Battlemage）戰鬥法師既是戰士又是法師!

感覺很像楓之谷之類的設計欸

這時候設計者想說戰鬥法師應該同時繼承戰士和法師的行為，因此 D 繼承了 B 和 C，就會發生

  • 行為歧義：Character 定義了一個方法 Attack()，Warrior 和 Mage 都重寫了這個方法，那麼 Battlemage 繼承的是 B 的 Attack() 還是 C 的？這導致程式邏輯的不確定性。
  • 屬性重複：Character 有一個屬性 Health，那麼 Battlemage 會從 Warrior 和 Mage 中各繼承一份 Health，資料有可能會亂掉。
  • 初始化順序問題：Battlemage 的實例化會兩次初始化 Character 的構造函數，因為 Battlemage 通過 Warrior 和 Mage 分別間接繼承了 Character。

假設你問爸爸和媽媽一個問題：「晚餐吃什麼？」

爸爸說吃摩斯，媽媽說吃麥當勞，那你聽誰的？

要點 1 - 用組合代替繼承，解決行為歧義（Ambiguity of Behavior）

讓戰鬥法師「擁有」戰士和法師的能力，而不是「繼承」它們。

這就像你可以分別問爸爸「摩斯怎麼去」和媽媽「麥當勞怎麼去」，不會混亂。

要點 2：用 Interface 來分離行為，避免多繼承，解決屬性重複（Duplicate State）

讓 Warrior 和 Mage 各自實現自己的行為，但把狀態留給 Character，並且避免通過多繼承直接繼承狀態。

這樣屬性只存在於 Character 中，行為分開，避免了屬性重複的問題。

多重責任導致類別設計混亂

在一個 Class 需要同時承擔多個角色或功能時，開發者可能嘗試透過多繼承來整合多個 Base Class。

建立一個文件系統模型：

Base Class A：可儲存的（Storable） 提供保存與載入的功能。
Base Class B：可壓縮的（Compressible） 提供壓縮與解壓縮的功能。
SubClass D：壓縮文件（CompressedFile）同時需要保存與壓縮功能。

所以開發者想說那就寫個 D : Storable, Compressible 吧，讚!

問題來了

  • 功能耦合過緊：如果某些 Class （例如 D）需要另一組功能（例如 Encryptable），開發者是要繼續找一個這種 Base Class 繼承然後吃下他所有的性質嗎?
  • 方法歧義：如果 Storable 和 Compressible 都定義了方法 Save()，那麼 D.Save() 的實現會來自哪個 Base Class？

結果來說還會導致

  • 可測試性降低：多重 Base Class 會使得 SubClass 變得難以測試，因為 SubClass 不僅耦合於 Base Class 的行為，還耦合於 Base Class 之間的交互邏輯。

多繼承試圖將不同的功能塞到一 Class 中，導致 SubClass 變成「一切的集合」，失去靈活性。如果每增加一個功能（如加密功能），就必須引入新 Class，會讓設計越來越複雜，且難以維護。

就像你有一個全能的背包，裡面可以裝書、筆電、裝餅乾、…。每次加東西就持續擴充背包。最後這個背包變得非常重，不僅難用，壞掉時還得重做一個。

要點 - 用組合與單一職責原則（Single Responsibility Principle）解耦 處理 功能耦合過緊的問提

這樣每個 Class 專注於自己的功能，CompressedFile 可以靈活組合功能而不需要多繼承。

無法將 parent class 轉為 subclass 使用

考慮一個公司的員工系統

然而有天 Allen 晉升了

如果 Allen 被晉升為經理，我們不能簡單地將 Employee 轉換為 Manager。我們需要創建一個新的 Manager 對象，並將 Allen 的資料拷貝過去。

可以這樣寫，把升職的邏輯封裝在 PromoteToManager 方法中

其他還有…

3.後續 Base Class 想新增功能時，不敢改 parent class 因為影響範圍可能會很大
4.subClass 被迫繼承了它可能不需要或不適用的功能

例子:
假設我們有一個 HomeAppliance，然後我們想要建不同類型的家用電器作為 subClass。

HomeAppliance 定義了所有家用電器可能有的基本功能：開關機、設置定時器和連接 WiFi。

Refrigerator 繼承自 HomeAppliance，但它不需要定時器功能，也可能不支持 WiFi 連接（取決於型號）。因此，它必須重寫這些方法並拋出異常，表示不支持這些功能。

Microwave 實現了定時器功能，因為這對微波爐來說是常見的。
SmartTV 實現了定時器和 WiFi 功能，因為這些是智能電視的常見特性。

這個例子突出了繼承的一個問題：subClass 被迫繼承了它可能不需要或不適用的功能。在實際開發中，就違反里氏替換原則，因為實作了一些讓子類別的行為偏離父類別的預期，導致父類別的功能在子類別中表現異常，要馬爸爸的要求不對，要馬小孩不該屬於這個爸爸的 (嗯?)

Composition

我覺得 Composition 很像是在拆解、集中的感覺，有種明確分類職責的味道

當有新功能需求時，我們不需要修改現有的 Class 。

我們可以建一個新的 Class 來實現這個功能，然後將它提供給需要使用的物件。這種方法允許我們動態地增加物件的能力。

清晰的界限與減少耦合，不用依賴 protected 或 virtual。

將可能變化的部分封裝起來

這樣，當我們需要修改或擴展這些可變部分時，不會影響到那些保持不變的部分。

當然 Composition 還是會有一些問題

可能每個class 需再實做一次相同的行為 尤其是相似的功能

每個播放器類都需要實現相同的方法（Play, Pause, Stop, SkipTrack），導致了大量的重複 Code。

C#

那這些問題在 C# 中會有嗎?

首先我們知道，C# 不支援多重繼承，但可以實作不同的 Interface，Interface 通常只定義方法簽章。

但當我們想要在 Interface 新增新的功能時，所有有實作的實體都會被影響到

所以從 C# 8 開始，提供 Default interface Methods ，可以在 Interface 寫 method Body，這個 Default Methods 不會影響到現有的實作 instance，若現有的實作想要調用，也可以自己 override 實作

使用

MP3Player 和 FLACPlayer 可以選擇性的覆蓋某些方法，否則，就無法直接用這個實體調用這個方法

並且，共同的功能（如 Play、Compress）在 Interface 定義一次，可以被多個 Class 重用。
每個 Interface 都專注於特定的功能集，做到 Interface 隔離

相比於傳統的繼承關係， Interface 提供了類似多重繼承的功能，但沒有傳統多重繼承的複雜性。
避免了菱形問題，因為每個方法都有明確的來源。

當我們考慮使用繼承時，我們應該轉換思維方式，將焦點放在物件的行為和功能上，而不僅僅是它們的結構。
我們需要深入理解 Parent Class 的設計意圖，以及使用者對這些 Class 的期望。這種思考方式自然會引導我們朝向更靈活的設計方向。

具體來說，我們可以考慮思考這些關鍵字 : Interface, Combination, SOLID, Contract… 來協助我們設計更好的程式!
