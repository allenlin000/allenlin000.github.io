Recursion - 從遞歸學習如何拆解問題

Created2024-07-02|Updated2024-12-21|C# 舞想
|Post Views:

不知道大家有沒有被「無限螢幕」嚇到過？我記得自己年紀小一點的時候，第一次看到那種無限延伸的畫面，真的是心裡一緊，腳步一歪，還會本能地閃開，深怕自己一不小心就被吸進去(?)，成為下一個走失在虛擬世界的都市傳說主角。

長大才發現其實這邊的世界好像沒有比較好，被各種事情跟人類拉扯，如同橡皮筋一樣失去彈性 (厭世)

又離題了

無限螢幕的視覺現象就是一種遞迴，螢幕分享者分享自己的螢幕同時，他也看著分享給其他人的畫面的那個軟體呈現裡面分享的畫面的裡面也呈現分享的畫面…乾我已經不知道自己在講什麼惹

拆解一下

  1. 螢幕分享的起點：使用者啟動螢幕分享功能，開始分享自己螢幕上的畫面。
  2. 分享軟體的畫面顯示：被分享的畫面包含著該螢幕分享軟體的視窗。
  3. 視窗中的畫面再次重複：由於該視窗顯示的是「當前螢幕」，而當前螢幕中又包含了「分享軟體視窗」，因此再次將分享畫面顯示出來。
  4. 這個遞迴循環效果，畫面中的畫面會不斷地循環，產生類似「無限鏡像」的視覺現象

有種永世不得超生的感覺

Simplest Example

回到現實世界，我對 recursion 的理解是，想像成階層，然後我們一層一層去做某件事情，這些事情會累積成一個成果，並且處理的方向上是遞減的階層，也就是慢慢收束

最經典的例子， Factorial (階層)

資料夾刪除

當執行 rd /s MainFolder 時，刪除過程會:

先進入最深層的 SubSubFolder

  • 刪除 file3.txt
  • 刪除 SubSubFolder 資料夾
    返回上一層刪除 file2.txt
  • 刪除 SubFolder1
  • 刪除 SubFolder2 中的 file4.txt
  • 刪除 SubFolder2
  • 最後刪除 file1.txt 和 MainFolder

這也是遞迴操作 - 同樣的操作在不同的 “階層” 重複執行，直到完成整個結構的處理。

工作實戰的例子

我自己是很喜歡這種可以慢慢把大問題拆解的概念 畢竟我沒那麼聰明可以一次想通複雜的事嗚嗚

最近工作上需求是這樣，每一個商品有所謂的商品選項

例如 一雙靴子有分

選項 : 款式
項目清單 :高筒、低筒

選項 : 顏色
項目清單 : 紅、黃、藍

用戶想要傳入

實際資料庫可能儲存方式是長這樣
“-1:-1:款式:高筒;-1:-1:顏色:藍” => 一個 sku

在這裡要怎麼用遞迴來解呢?

  1. 每一個選項我們視為 “階層”
  2. 每個階層做的事情是一樣的 : 取完清單第一項就到下一層
  3. 最後的 Base Case 就是當階層走完時，單筆 sku 所需要的資料也就收集完了
  4. 因為我們要把每個階層的每個選項清單項目的都組合出來，所以搭配了 “loop” & 回朔

回到 depth = 0，移除 “高筒”，嘗試 “低筒”。

要注意的是:

  1. 遞迴到下一層時要記得把 depth + 1，否則會卡關
  2. 一定要設 base case，否則會無限遞迴
  3. 如果需要在最後得到最終回傳，每一層都要 return 回來
  4. 這個例子因為變數放在遞迴方法之外，因此不需要重複傳遞這個變數出來

河內塔

Tower of Hanoi，最後我們以這個有趣的問題，來結束第一篇的遞歸之旅吧

所謂的河內塔是醬

我們有一座層層堆疊的塔如圖所示由大到小，任務目標是把整座塔從起始柱全部搬移到目標柱上
搬移的規則麻…

  1. 大的不能堆在小的上面
  2. 一次只能搬一個

找到規律從直接操作開始…

start = 起始桿
aux = 輔助桿
target = 目標桿

層數 = 1

layer 1 搬到 target

層數 = 2

layer 1 搬到 aux
layer 2 搬到 target
layer 1 搬到 target

層數 = 3

layer 1 搬到 target
layer 2 搬到 aux
layer 1 搬到 aux
layer 3 搬到 target
layer 1 搬到 start
layer 2 搬到 target
layer 1 搬到 target

可以觀察出一些規律

  1. 固定有一步是第 n 盤放到 target
  2. 在那之前 ，n - 1 盤以上都要搬到 aux
  3. 在那之後 ，n - 1 盤以上都要搬到 target
  4. “每一步” 都是這個行為本身，例如 2 , 3 本身也可以把 aux 本身當作當下的 target
