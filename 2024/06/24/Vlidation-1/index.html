資料驗證在前端 & 後端都有其必要性

有效的前端驗證除了能有效減少廢 Request 的數量，也可以減少使用者的的抱怨，尤其按下送出等了很久之後才告訴使用者格式不符
但通常在前端會檔比較通用性的錯誤，客製化需求上可能還是會需要向 Server 的資料庫做資料比對已明確資料合法性

關於 ApiController

當我們隨手在 Api 掛上 ApiController Attribute時, 在這小小的舉動中，我們的 Api 被賦予了哪些能力?

答案就在 MS Doc 中

其中資料驗證方面 :

  1. model validation errors 會自動 trigger 400 BadReqeust，且會自動產生預設訊息

例如

  1. 複雜型別自動套用 [FromBody]
  2. 參數型別如果是 IFormFile 或 IFormFileCollection 的話會自動套用 [FromForm]
  3. 簡單模型或任何其他參數，全部都會自動套用 [FromQuery]

FluentValidation

為了解決驗證參數導致程式寫得亂七八糟，.NET 有一個很方便且使用起來很口語的參數驗證工具，他支援許多情境，例如 : 在某個條件成立時才驗證指定欄位、傳入自訂的規則、針對驗證規則指定失敗時的自訂訊息…等等
也可以控制 檢查清單中的一項不符合就 Fail 的情境
