Query Expression Syntax vs Method Chaining Syntax

Created2024-06-06|Updated2024-06-16|LINQ
|Post Views:

目前在專案上，LINQ 常會以 Method Chaining 或 Query Syntax 出現，有時，自己在撰寫時，免不了會想一下用哪一種方式好，除非特殊情況(複雜的 Query)，否則都沒有一個 “哪一個比較適合” 的感覺

舉個 MSDN 提供的例子:

當我們在 C# 中使用 LINQ 查詢語法（Query Syntax）時，這種查詢語法看起來簡潔且易於理解，但實際上，.NET 的 Common Language Runtime（CLR）並不直接支援這種語法。

CLR 主要還是處理方法呼叫 (method calls) 和其他基本的程式碼結構。

但我個人是還滿習慣，方法串接的方式，明確往一個方向讀過去的感覺，但也有許多人是比較習慣類 SQL 的語法

那我們甚麼時候建議使用選用哪一種呢?

為了解答這個問題，需要做一些研究!

LINQPad Lambda translations

LINQPad 提供了一個方便的環境，讓我們可以同時編寫查詢語法和方法呼叫，並即時查看它們之間的轉換。

差異

1.Method syntax 中可用的標準查詢運算子，有部分並未包含在 Query Syntax 中，例如：Distinct、Take 等等，若要使用這些運算子，就必須使用 Method syntax 來撰寫。

2.Method syntax 讓我們可以將查詢分解成多個步驟，每個步驟都返回一個中間結果（IEnumerable），這樣可以更容易地組合和重用查詢的不同部分。
查詢語法通常是一個完整的查詢表達式，不太容易將查詢拆分成可重用的部分。

3.無論是查詢語法還是方法語法，LINQ 查詢都是延遲執行的，即查詢只有在實際需要結果時才會執行。但是，方法語法可能可以更明確地控制查詢的執行時機，例如使用 ToList() 或 ToArray() 等方法立即執行查詢並返回結果。

再看一個 Left Join

通過比對，在處理複雜的連接操作時，Query Syntax 通常表達方式更加簡潔。而 Method Syntax 雖然功能強大，但在需要使用 JOIN 時我們很少看到有人選擇這條路
我自己是已經不太想看 Method Syntax 的語法了

let

假設我們有一個 Character 類別，表示遊戲中的角色，其中包含角色的名稱、等級和經驗值。我們想要查詢所有達到 5 級及以上的角色，並計算他們升到下一級所需的經驗值。

let 的作用是，讓我們可以暫時儲存運算後的資料作為後面的查詢使用

透過這個例子，我們看到 Query Syntax 中的 let 子句和 Method Syntax 中的 Select 方法結合匿名類型可以達到類似的效果。Query Syntax 使用 let 子句引入一個新的範圍變量，而 Method Syntax 則使用匿名類型的屬性來表示中間計算結果。如果理解 let 的用途的話， QUery Syntax 會比較好讀一些

從不同資料來源組資料時

這個與 JOIN 不同的是，我們可能不是要交集篩選他們的資料，而是單純從不同資料來源，列出不同的排列組合

Query Syntax 使用兩次 from 顯得簡潔許多
