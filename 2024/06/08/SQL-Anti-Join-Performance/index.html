Anti Join - Performance

Created2024-06-08|Updated2025-11-11|SQL 職場歷險記
|Post Views:

你是否曾經辦了健身房會員卡，結果只去了第一天，之後再也沒踏進去？
你是否曾經鼓起勇氣傳訊息給喜歡的人，卻總是已讀不回？
你是否曾經揪朋友出來玩，群組裡萬人響應，最後現場只來了一個？


如果以上情境你都心有戚戚焉，那麼…你可能是魯蛇 🤐


咳咳，我的意思是：這些看似悲傷的故事，其實背後都有個共同的規律：隱藏的多數。

  • 健身房的角度，他們想知道哪些會員繳了錢卻不來，就是所謂的「三分鐘熱度凱子」。
  • 感情的角度，你會想知道群發訊息後，哪些人始終沒有回應，好讓你直接封鎖省事。
  • 友情的角度，你會想知道哪些朋友總是「嘴巴答應，身體卻很誠實地缺席」。

這些情境背後，其實就是一個簡單的數學邏輯：

👉 全部的人 - 出現的人 = 那些看不見卻存在的名單


在資料庫的世界裡，這個概念就叫做 —— Anti Join!

現在有兩張表：

  • Friends（朋友名單）
  • Attendance（實際到場名單，代表「真的出現的人」）

你要知道哪些朋友總是「嘴巴說要來，結果缺席」


💩 NOT IN

  • 計算子查詢結果集：把 Attendance.FriendID 撈出來，形成一個清單（可能包含 NULL）。
  • 左表 Friends 掃描一次：逐筆取出 FriendID。
  • 對於每一筆 Friends.FriendID → 檢查是否存在於右表清單裡。
  • 如果右表清單有 NULL → 整個條件可能變成「UNKNOWN」，導致結果不正確。

子查詢通常會先計算一次，存成暫存集合。每筆 Friends 都要跟這個清單比對。如果清單大，效能可能下降。
NULL 會是邏輯陷阱。


🔹 NULL 問題

⚠️ 注意：如果 Attendance.FriendID 裡出現了 NULL，整個查詢結果可能會變「空集合」！因為 NOT IN 碰到 白名單 的 KEY 有 NULL，結果會變成「不確定」，最後導致 所有朋友都不會被選出來。

實務上，若用 NOT IN，一定要加上條件


🔹 效能分析

會被轉換成 Anti Semi Join，也就是一種「只取左表有、右表沒有」的最佳化操作。當右表（Attendance）有 索引 (Index) 時，效能很好。



💩 NOT EXISTS

簡單說：對於每個朋友，問一次「你有出現在 Attendance 嗎？」如果沒有，就被挑出來。

  • 掃描左表 Friends：逐筆讀取 FriendID。
  • 利用索引（如果有的話），快速查 Attendance 是否存在相同 FriendID。
  • 找到第一筆匹配就停止檢查（效率比 NOT IN 高）。
  • 決策：
    • 如果右表存在 → EXISTS = TRUE → 這筆不選。
    • 如果右表不存在 → NOT EXISTS = TRUE → 這筆選出來。

不需要完整掃描右表，只檢查「是否存在」。有索引時效能最佳。不會受 NULL 影響，結果永遠正確。


🔹 不受 NULL 影響

NOT IN 可能因為右表有 NULL 而導致整個查詢「不回任何資料」。但 NOT EXISTS 完全不在乎 NULL，因為它只是檢查「有沒有匹配行」。

在 SQL Server、Oracle、PostgreSQL 等主流 DB，NOT EXISTS 通常會被轉換成 Anti Semi Join。
當右表有索引（例如 Attendance.FriendID 上有 Index），效能幾乎跟 NOT IN 一樣，甚至更穩定。


🔹 效能

在有 Index 的情況下，與 NOT IN 幾乎相同；在無 Index 的情況下，兩者都可能慢



💩 LEFTJOIN… IS NULL

就是找「左表有，但右表沒有」的資料

  • 把 Friends 全部取出，跟 Attendance 嘗試做匹配。
    • 如果有匹配 → 把 Attendance 的欄位帶出來。
    • 如果沒有 → Attendance 欄位設成 NULL。

（這裡可能產生很大的「中間結果集」）。

  • 過濾條件，保留那些 a.FriendID IS NULL 的紀錄。也就是「左表有但右表沒有對應」。

語意直觀，但中間結果集可能龐大。如果右表很大且沒有索引 → 效能可能比 NOT EXISTS 差。在現代 DBMS，最佳化器有時會把它重寫成 Anti Semi Join，效能差異不大。


🔹 重複值的處理

如果右表 (Attendance) 有重複紀錄，LEFT JOIN 會把左表的資料「複製」多次
雖然最後 IS NULL 可以過濾掉，但中間結果可能會比 NOT EXISTS 更大，導致效能下降

LEFT JOIN … IS NULL 通常會被最佳化器轉換成 Anti Semi Join，效能跟 NOT EXISTS 幾乎一樣。
但當 右表很大、且無索引 時，LEFT JOIN 在中間產生的資料量會比 NOT EXISTS 多，效能可能會差一點


🔹 中間結果集問題

LEFT JOIN 的本質：先把 左表 (Friends) 和 右表 (Attendance) 做一次 Join。即使最後要的是「右表沒對上的資料」，但在 執行計畫 裡，資料庫還是會產生「合併後的結果集 (Intermediate Result Set)」

如果右表資料很大，這個中間結果集可能會非常龐大，再加上 IS NULL 過濾，效能就會下降。

LEFT JOIN 是「先全都比對，再篩掉不要的」


🔹 想要帶出「右表的一些欄位」

當你不只是要排除，而是想找出所有朋友，以及他們的出席紀錄（有來顯示，沒來就 NULL）



💩 EXCEPT

EXCEPT 在 SQL 裡表示「集合差集」：取左邊查詢的結果集合減去右邊查詢的結果集合

  • 左邊結果集算出來（Friends）。
  • 右邊結果集算出來（Attendance）。
  • 做一次「去重 (DISTINCT) + 排序 (Sort)」。
  • 算出「差集」。

👉 所以兩張表都會被完整掃描一次。


🔹 天生去重

EXCEPT 天生會幫你做去重 (DISTINCT)。如果 Friends 或 Attendance 有重複值，EXCEPT 會自動去掉
預設去重可能造成效能問題，EXCEPT 在內部會做 DISTINCT，等於要排序 + 去重。
如果你不需要去重，這個動作就是多餘的。在資料量大的時候，這會比 NOT EXISTS / LEFT JOIN … IS NULL 多一些效能開銷


🔹 處理 NULL

在 SQL 標準裡，EXCEPT 會正確處理 NULL，不像 NOT IN 那樣踩陷阱



💩 測試方法

在比較這些 Anti Join 的效能時，我們可以開啟 SQL Server 的統計資訊來觀察查詢效能。

接著在不同的查詢之間用 GO 區隔，這樣每個查詢的統計結果會分開顯示，方便觀察：


🔹 CPU Time

代表 SQL Server 在這次查詢中，真正花費在 CPU 運算 的時間。通常越小越好，表示 CPU 負擔較低。


🔹 Elapsed Time

代表查詢從開始到結束，實際花費的總時間。這個值會受到 CPU、IO、網路、記憶體等多種因素影響。


🔹 Logical Reads

代表 SQL Server 讀取了多少資料頁 (Page)，也就是 查詢需要存取多少次資料。越少越好，因為表示這個查詢需要讀取的資料較少，IO 負擔較輕。


🔹 執行計畫 (Execution Plan)

在 SSMS (SQL Server Management Studio) 中，可以點選「Include Actual Execution Plan」(快捷鍵：Ctrl+M)，或是執行前按下工具列的執行計畫按鈕。執行計畫會顯示 SQL Server 實際使用了哪種演算法（例如 Nested Loop Anti Semi Join、Hash Match Anti Semi Join、Merge Join…），這能幫助我們理解資料庫在不同語法下的最佳化差異。
