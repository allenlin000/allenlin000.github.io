當 API 設計遇上錯誤處理：該如何優雅地告訴用戶？

最近在設計 API 時遇到了一個小挑戰：當系統遇到錯誤，要以什麼樣的方式向外界溝通？
你可能覺得，錯誤就錯誤嘛，隨便噴個「Something went wrong, please contact admin」應該沒什麼問題吧？但這種偷懶的處理方式，只會讓使用 API 的人想寄刀片給你。

當我們設計一個對外開放的 API 時，錯誤訊息可以大致分為兩種需求：

  • 內部開發者調試用的錯誤資訊
    需要完整的 stackTrace，例如是哪一個 Service、第幾行出問題等，越具體越好，方便快速找到問題點。

  • 外部使用者需要的錯誤訊息
    比如你的 Request 格式錯了、某些資料不存在，或是你該做什麼調整，而不是一句冷冰冰的「Bad Request」。沒錯，就是在說那些只回一句「Something went wrong」的雞巴程式碼。

資料驗證

我們有這麼一個 ValidateContentCollection

這裡可以看到兩件事：

JSON 格式錯誤：直接噴 FormatException，包含具體的錯誤原因，方便開發者知道問題出在哪。
其他錯誤：記錄詳細的 stackTrace，但只對外傳遞簡潔的訊息，避免洩漏內部實現細節，主因是這整包 Service 就是個驗證服務

其中 ValidateCollectionProperty

這裡的驗證策略很明確，檢查集合格式是否正確、階層是否超出限制，以及是否符合預期的屬性規範。每一個錯誤都給出具體的錯誤訊息，而不是含糊地丟一個「格式錯誤」就敷衍過去。

ValidateAndDeserializeSKUContent

這段處理反序列化和進一步驗證，能夠保證資料內容符合預期格式，任何異常都會被精確記錄，對外提供清晰的錯誤原因。

可以看到，拋出錯誤會明確告訴你為甚麼錯誤，因為這是”已知的”並且 “做出對應警告的處理”
但為什麼是 ApplicationException 呢，這就要看 “全局的 ExceptionHandler” 是怎麼對應不同 Exception去做 Message 處理的

查看註冊區

GlobalExceptionHandler：統一管理錯誤處理

對於 ApplicationException，直接回傳具體訊息。
對於其他例外，則採用預設訊息，避免洩漏敏感資訊。

ex.Message vs ex.ToString()

在實務中，ex.Message 與 ex.ToString() 的選擇取決於使用場景：

ex.Message：只包含例外訊息文字，適合對外溝通，簡單明瞭。
ex.ToString()：包含完整的 stackTrace，方便內部除錯。

參考文章 :　https://blog.darkthread.net/blog/exception-innerexception-and-tostring/

如同文章內容講的，要抓出哪一行錯誤，應檢查 ex.InnerException 是否不為 null，則有內部錯誤資訊，再由 ex.InnerException.Message 取出底層錯誤訊息，但要留意 ex.InnerException 可能還會有 InnerException，真正的錯誤訊息藏在 ex.InnerException.InnerException.Message 之類的 換句話說，得寫段剝洋葱才程式碼挖出真正出錯原因。
