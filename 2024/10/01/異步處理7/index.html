Asynchronous Programming (七. Thread Pool & 什麼時候要自行建立執行緒)

Created2024-10-01|Updated2024-11-19|C# 舞想
|Post Views:

想了解 .NET 的執行緒概念，免不了要認識 .NET CLR 執行緒管理的機制 Thread Pool

初步認識

我們嘗試先用一個生活上的比喻相關的詞彙之間彼此的關係

ThreadPool : 大型快遞公司的配送中心
Threads (執行緒) : 配送員
Tasks : 包裹

首先，配送中心有一定數量的配送員，這些配送員隨時待命，準備送貨。每個需要執行的任務就像一個待配送的包裹。當有新包裹到達時，會被放入待配送區（任務隊列），空閒的配送員會從待配送區取走包裹進行配送。

在配送高峰期（出現大量任務時），公司會臨時僱用更多配送員，類似於 ThreadPool 在需要時建立額外的執行緒，但公司不會無限制地僱用配送員，有一個最大數量限制

配送完成後，配送員返回中心等待下一個包裹，而不是每次都僱用新的配送員，就像 ThreadPool 重用執行緒，而不是每次都創建新的執行緒

在配送淡季，公司會減少配送員數量以節省成本，就如同 ThreadPool 會在執行緒長時間空閒時逐漸減少其數量。

但對於特殊包裹（如大件物品），公司可能會安排專門的配送員。這類似於在某些情況下，開發者可能選擇建立自定義執行緒而不使用 ThreadPool。

預設值

.NET 有提供 Thread Class，其中我們可以使用 ThreadPool.GetMinThreads(Int32, Int32) 方法，查詢出該執行緒集區內預設會建立起多少個數量的執行緒。

GetMinThreads 方法中需要傳入兩個 out int 型別的參數，分別會得到
[執行緒集區視需要建立的背景工作執行緒最小數目]
[執行緒集區視需要建立的非同步 I/O 執行緒最小數目]
也就是分別會建立起 CPU Bound 與 I/O 會使用到的許多執行緒到執行緒集區中。

這兩個數字的意義是，即使沒有工作要做，這些執行緒也會保持活躍狀態，下限預設為 CPU Core 數 (含 Hyperthreading)
目的是減少在需要執行緒時的等待時間。

但當執行緒集區內的執行緒數量不夠用的時候，執行緒集區究竟可以產生出最多多少的執行緒到執行緒集區內呢？
可以使用 ThreadPool.GetMaxThreads(Int32, Int32) 方法來進行查詢
這裡的數量並不表示執行緒集區內現在擁有的執行緒數量，而是當執行緒集區內的執行緒數量不夠用的時候，執行緒集區額外產生的執行緒數量，最大的上限值為多少。

這些數值表示 ThreadPool 可以擴展到的最大執行緒數量，為的是防止系統資源被過度消耗。

如果我們的應用程序經常需要等待執行緒，可以考慮增加最小執行緒數，如果系統資源使用過高，就需要降低最大執行緒數。
也就是說，最小值太高可能導致資源浪費，最大值太低可能限制並發能力。

如何啟動 Multi-Thread

為了先聚焦在 ThreadPool 上，我們先用較早期的 .NET API ThreadPool.QueueUserWorkItem()來測試

QueueUserWorkItem () 方法可以啟動 Multi-Thread，QueueUserWorkItem () 方法有一個 WaitCallback 的參數

這個結果告訴我們:

  1. 開始和結束時的 Thread ID 是相同的，表示 Main 方法在同一個 Thread（Main Thread）中執行。
  2. Main 方法中的 “end” 輸出在 DoSomethingLong 方法執行之前出現，表示 ThreadPool.QueueUserWorkItem 是非阻塞的，他不會擋住 Main 往下執行，它會在任務加入排隊行列後立即返回。

觀察 ThreadPool 的重複使用

結果告訴了我們

  1. 我們看到一些 Thread ID 重複出現（如10, 9, 11），表示 ThreadPool 會將 Thread 重覆再利用。
  2. 多個不同的 Thread ID 同時出現，這些任務是並行執行的。
  3. 實測多次後 Thread ID 的使用順序不是固定的，表示 ThreadPool 動態分配 Thread 來執行任務
  4. Sleep 之後，我們仍然看到相同的Thread Thread ID 被重複使用，這表示 ThreadPool 會保持這些 Thread 活躍一段時間

ThreadPool 如何自己調節 Thread 數量

Starvation-Avoidance Mechanism (怕你餓機制)

Starvation-Avoidance Mechanism 基本上是發現 Queue 中等待項目沒減少，則增加工作 Thread 數
預期能夠 :

  1. Prevention of Resource Contention Deadlock
  2. Facilitation of Task Diversity in Concurrent Execution
  3. Enhancement of System Responsiveness and Throughput
  4. Potential Deadlock Breaking or Mitigation

用一個生活案例比喻，今天有一間餐廳，有以下幾個主要任務：

  1. 接受訂單
  2. 準備食物
  3. 送餐
  4. 結帳

每個任務都在 ThreadPool 中的一個 Thread 上執行。

假設我們的系統中出現了以下情況：

所有可用的 Thread 都在處理食物準備任務。
這些食物準備任務需要等待某些共享資源（例如，廚房設備）。
同時，系統隊列中堆積了大量的訂單處理和送餐任務。

可能會發生以下問題：

死鎖（Deadlock）：如果食物準備任務互相等待對方釋放資源，可能導致死鎖。
任務飢餓（Task Starvation）：新的訂單和送餐任務無法得到處理，因為所有Thread都被阻塞在食物準備任務上。

ThreadPool 的 Starvation-Avoidance Mechanism 的作法:

  1. ThreadPool 注意到隊列中等待的任務數量沒有減少。
  2. ThreadPool 決定增加 Worker Thread 的數量。

透過增加新的 Thread，系統有機會執行不參與當下死鎖的任務，例如，訂單接受和送餐任務或執行新任務像是補充設備，這些新任務可能會釋放一些被死鎖 Thread 等待的資源，從而打破循環等待，或者是新增的 Thread 會執行一些關鍵的任務，這些任務能夠間接地解除資源鎖定，例如，一個新 Thread 可能會處理一個超時操作，強制釋放某些被長時間持有的資源。

Hill-Climbing Heuristic 爬山捷思法

是 ThreadPool 中用於優化 Thread 數量的一種算法策略，”Hill-Climbing” 源自於優化理論，類比於在山上尋找最高點，”山” 代表系統的 Throughput，”爬山” 則是調整 Thread 數量以達到最佳 Throughput，因此，Hill-Climbing Heuristic 想要找到最佳的 Worker Thread 數量，以最大化系統 Throughput，平衡資源利用和系統開銷。

當發現隊列中有等待的工作項，且當下工作已持續一段時間（如超過 0.5 秒），會增加 Thread，如果 Throughput 提高，可能會繼續增加 Thread，如果 Throughput 下降或保持不變，可能會減少 Thread，可以透過縮短每個任務的執行時間，允許更頻繁的調整，有助於更快地適應變化的工作負載

至於增加 Thread，並不一定會增加 ThroughPut 的原因在於，當 Thread 數量增加時，CPU 需要更頻繁地在不同 Thread 間切換。每次切換都涉及保存和恢復 Thread 狀態，這會消耗 CPU 週期。過多的 context switch 可能會顯著降低整體性能，並且，Thread 會競爭有限的系統資源，如 CPU 核心、記憶體、緩存等，當 Thread 數量超過 CPU 核心數時，競爭加劇，可能導致效能下降，當然，每個 Thread 都需要自己的堆疊空間和其他相關資源。過多的 Thread 可能導致記憶體壓力，增加記憶體管理開銷。

.NET ThreadPool 執行緒數量管理實驗

實驗來源 : https://blog.darkthread.net/blog/threadpool-thread-management/

實驗一.塞入200 個工作，每個工作要設定 1 min 完成，觀察 Threads 數量增長情形

觀察結果

  1. 初始設置：ThreadPool 的最小工作線程數為 20，最大為 32767，最小 I/O 完成端口線程數為 1，最大為 1000
  2. 開始時，線程數從 21 逐漸增加到峰值 125。增長速度在前 60 秒較快，之後放緩，表示再增加以無益於改善效能與資源利用
  3. 任務完成後的行為是，97 秒後，所有任務完成，Pending 保持為 0，線程數和運行中的線程數開始緩慢下降。
  4. 由於每項作業會佔用工作 Thread 60 秒才結束，所以前 1 min 符合 Queue “有待辦工作且無工作完成” 的 Starvation 偵測條件，因此這段時間會從一開始 Thread 數以每秒鐘 1 ~ 2 條的速度新增 (符合文件說每秒兩條的增加速率)
  5. 滿 60 秒後，開始有工作完成，此時 Thread 增加速度趨緩，從每秒 2 條下降
  6. 97 秒左右，Thread 數已累積到 124，此時 Queue 已空，Thread 數達到 125 後便不再增加
  7. 大約再經過 20 秒後，Thread 數開始減少

關於面對瞬間流量的改善作法，就可以試著 調整 ThreadPool.SetMinThreads () ，因為其最大用處就在於縮短 ThreadPool 遞增 Thread 數到足夠數量的時間。若事先已預期可能一次湧入 200 件任務又希望儘快完成，可預先開好足夠的 Thread，省下一秒兩條慢慢提高產能緩不濟急的期間，降低作業在 Queue 中等待及總完成時間。而在網頁等待 Queue 長度有限的情境中，提高 ThreadPool 基本 Thread 數量，面對瞬間爆大量的情境，能改善 Thread 來不及增加導致 Queue 塞爆噴 503 的狀況。

實驗二.開頭即設定 200 條最低執行緒數量

結果時間縮短為 60 秒左右，也就是 200 個任務等於一個任務所需要完成的時間

每個任務時間縮短到 2 秒，每隔 30 秒塞 200 筆工作

200 件工作做完後約 20 秒，Thread 數由 202 降到 4，第 30 秒又來 200 件工作時，Thread 數瞬問回到 200 條，與預期相符，結案

可以看到 2 秒左右後， Running / Pending 已歸零，大概等到 20 秒左右，才把 Thread 收一收，但 28 秒左右因為又有 200項工作塞進來，執行緒又瞬間飆到 200

至於為什麼明明沒工作了，卻在 20 秒後才開始收攤，是因為 ThreadPool 有一個內置的機制，是會在線程閒置一段時間後才開始回收多餘的線程，這個閒置時間通常是幾十秒，在這個實驗中大約是 20 秒。
