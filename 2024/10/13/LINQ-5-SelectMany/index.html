這組方法跟上組方法的差異在於，多了一個 resultSelector，它可以傳入 TSource(原本集合的每一個元素) 及 TCollection(子集合中的每一個元素)，讓我們可以讓主檔的資料跟明細的資料合為同一筆資料。

參考 : https://peterhpchen.github.io/DigDeeperLINQ/12_HowToUseSelectMany.html#%E5%8A%9F%E8%83%BD%E8%AA%AA%E6%98%8E

Select: 將資料經由 Selector 轉換後放入 IEnumerable 中，其結果的數量跟原本相同
SelectMany: SelectMany 會將 Selector 的結果扁平化，輸出在同一集合中，其結果的數量會跟原本的不同

四個 overload:

來看最單純的一個

  1. 判斷 source 及 selector 是否為空，為空的話丟 ArgumentNull 例外
  2. 傳回 SelectManyIterator
  3. 到這裡跟 Select 幾乎一模一樣，唯一有差別的就只有最後的回傳值，SelectMany 是傳回 SelectManyIterator，相信特別之處就是在這裡，我們來看看 SelectMany 的定義:
  1. 此方法區塊為 yield 區塊，會轉為 Iterator Pattern，回傳的資料是 IEnumerable 的集合
  2. yield return 傳回每一個元素的資料
  3. 每個元素的 index 較前面的元素多加 1
  4. selector 執行後取得每個元素的子集合資料，再用 foreach 巡覽整個子集合
  5. 傳回子集合的每個元素

原來 SelectMany 已經幫我們把第二個迴圈要做的事情給做掉了。

案例閱讀

參考 : https://stackoverflow.com/questions/38311230/chain-selectmany-instead-of-using-a-join-statement?newreg=43ff320edfd84529aa6ae9a8b5d8e28f

這個 Stack Overflow 的討論串探討在 Entity Framework 中使用 LINQ 的 SelectMany 來執行查詢是否比使用 JOIN 語法更有效率。 討論中提出了一個案例，使用 SelectMany 來尋找屬於特定學校且屬於特定科系的課程。 作者想知道 Chain SelectMany 的方式是否是不好的做法，特別是在大型資料庫中。 討論結果顯示，使用 SelectMany 鏈式呼叫是 Entity Framework 的推薦做法，因為它利用了導航屬性來自動生成 JOIN 語法，簡化了查詢過程，同時也能確保查詢的效率。

這邊的結論是，在使用 Entity Framework 時，使用 SelectMany 來處理關聯式資料比手動使用 JOIN 語法更好。因為 EF 可以利用導覽屬性 (navigation properties) 的中繼資料自動產生 JOIN 語句，並且 Code 也更容易理解
