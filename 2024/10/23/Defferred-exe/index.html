一個簡單的 loop 背後發生了什麼?

這邊使用 yield return 為例(當然不是使用它也可以展現 loop 行為)

結果

當我們使用 yield return 時，Compiler 會自動為我們生成一個實作 IEnumerator 的類別，這個類別包含了 MoveNext 的實作，原始碼會像這樣

所以當我們執行

  1. city.GetEnumerator() 返回編譯器生成的 Iterator class 實例
  2. 這個生成的類別有完整的 MoveNext/Current 實作
  3. foreach 使用這個生成的迭代器來 loop 集合

因此這個 loop 過程是

  1. 進入 foreach
  2. 執行 GetEnumerator () , 得到一個 IEnumerator
  3. 執行 MoveNext () , 以判斷 loop 是否結束。若尚未結束則將 Current 屬性移動到下一個元素
  4. 回傳 Current 屬性給 item

不論是 IEnumerable 或是 IEnumerable 都提供一個 GetEnumerator () 方法。再透過所得到的 Enumerator 物件去執行 loop 這個動作，所以重點是 foreach 去找走訪器 “Enumerator” 這個行為，完成了我們常見的 loop

Deferred Execution 究竟是怎麼回事?

現在我們稍微往下一層知道了一個簡單的 loop 被後做了什麼，那跟延遲執行有甚麼關係呢，我們知道一般來說，直觀理解上今天程式跑到哪一行，那一行就會被執行。

但 ASP.NET 的 LINQ 有一個很重要的特性，叫做「延遲執行」（deferred execution）, 或稱為惰性求值（lazy evaluation）. 也就是當成是跑到這一行時，我們只是生成了一段指令但不實際對資料執行操作，叫他先等等，給我忍住，等到我們需要取用查詢結果的時候，( e.g. 跑到 foreach 時)，才去執行查詢表示式 !

Q : WHERE 不也是 實作 Yield Return 嗎，與 IQueryable 操作資料庫的延遲執行差異是什麼?

A : IEnumerable.Where()，內部確實有實作 yield return，但這是在記憶體內部操作的， 撈資料庫時，會把資料都撈出來才在記憶體做篩選行為(LINQ to Objects)，而 IQueryable.Where()則是轉換成 SQL WHERE 子句，也就是會在資料庫層面篩選，只傳回需要的資料，二者不同

列出幾項

  • Multiple Enumerations: If you enumerate over a deferred execution query multiple times, the query will be executed multiple times, which can lead to performance issues.

  • Side Effects: If the data source changes between enumerations, the results may differ each time you iterate over the query.

  • Captured Variables: Be cautious when using variables captured by a lambda expression in a deferred query. If the value of the variable changes before the query is executed, the new value will be used.
