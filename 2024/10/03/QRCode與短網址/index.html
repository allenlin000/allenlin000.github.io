QRCode vs 縮網址

QR Code 的複雜度與文字長度有關，因此為了簡化 QR Code 降低掃瞄失敗率，用縮網址服務將又臭又長的 QR Code 做縮網址

實作 CLI 指令能 「URL 轉短網址產生 QRCode 再 複製到剪貼簿」

  1. 縮網址用 recurl.cc，它只需要用 FB 或 Google SSO 登入，免填註冊資料就能拿到 API Key，額度為每小時 100 次

  2. QR Code 選擇 quickchart.io QR Code API

QRCode

起源

“ Quick Response Code “

1994 年的日本，由汽車零件業者 Denso Wave 公司發明，用來追蹤生產線上的貨物，並且讓條碼能容納日文漢字、假名訊息。

有 barcode 了為甚麼還需要 QR Code?

與一維條碼最大的差異在於，二維條碼能夠承載更多資料，而 QR Code 能儲存的資訊種類囊括了數字、字母、二進制 (位元組)、以及日文、漢字。可以這麼說，一維條碼代表的資訊以「數字編號」為多，號碼用來識別商品，但 QR Code 可以讓日文、漢字資料直接記錄在條碼裡，也就是讓條碼代表一句話、或是一串網址。

現在許多的應用都因為 “可以放更多資訊” 加上 “個人行動裝置的盛行” 提供了我們許多便利，例如 : 機票、活動門票或旅遊通行證信息編碼到 QR Code、支付使用 QR Code 進行交易，用戶掃一下就可完成支付、數位名片之類的。

除此之外還有辨識速度快、360 度全方位都能被識別、容錯能力高、佔位面積小等優勢

QR Code 怎麼避免錯誤?

問題：一家公司想在 QR Code 中嵌入公司標誌，同時確保 QR Code 在部分損壞或模糊的情況下仍能被掃描，他們應該選擇哪個容錯等級？

A. L (低) - 可修正 7% 的字碼
B. M (中) - 可修正 15% 的字碼
C. Q (中高) - 可修正 25% 的字碼
D. H (高) - 可修正 30% 的字碼

因為電腦只能接收 0 與 1，因此無論是數字、字母、還是漢字，所有訊息都要回到 0 與 1 的編排來轉換。QR Code 裡有無數個黑色與白色的方格，每一格黑白格子都是一個位元 (bits)，黑色方格代表 1，白色方格代表 0，掃碼時機器只要判斷反光與否，就能將黑白圖像轉換為 1 與 0 的組合。

當某部分資訊缺失的時候，其它部分會自動填補，讓整體資訊依然可以完整辨識，容忍錯誤發生。簡單來說，就是當 QR Code 條碼缺了一角，或是沒有辨識完全，資訊依然可以被掃瞄出來，這就是 QR Code 容錯能力的表現，是一維條碼辦不到的。

但甚麼事情都是有成本的，容錯能力越高，同等級的 QR Code 能夠容量的資訊量就越少，這是因為將原本能夠儲存資料的地方，拿去做容錯備份，於是一般儲存資料的空間被壓縮，能夠儲存的資料也就變少。

為什麼 QR Code 都會有幾個 “回” 在角落

在了解原因之前，自己先猜猜看吧，幫你準備了一個選擇題

問題：QR Code 中的三個”回”狀定位標記（位於上方兩側和左下角）的主要功能是什麼？

A. 增加 QR Code 的美觀性
B. 放秘密嘿嘿
C. 允許從任何角度掃描並正確讀取 QR Code
D. 防止 QR Code 被篡改

在 QR Code 的結構之中，最重要的就是位於上方兩側與左下的 3 個「回」狀定位標記。

從相對位置來說，三個定位點形成一個直角三角形的三個頂點，這種特定的幾何排列使得掃描器可以識別 QR Code 的方向。

從相對稱性來說，每個定位點本身是對稱的，這意味著無論從哪個角度看，它們都保持相同的形狀。

定位標記讓掃描端能夠 360 度辨識 QR Code 的方向，因此就算趴著掃、躺著掃、倒著掃，也能判斷出哪邊是開始、哪邊是結束，避免讀取錯誤。

QR Code 裡真正紀錄資料的地方僅有灰色的「資料與糾錯碼訊息」，而在該區塊裡，又會分為真正紀錄資訊的區域，以及容錯區域。QR Code 的定位標準與格式資訊位置都是固定的，至於定時資訊 (Timing Patterns，又稱時序圖案) 是用來讓機器辨識 QR Code 的單位格子大小、以及提供座標位置，像是基準點一樣，避免 QR Code 尺寸過大，或是掃描時變形，導至格子判讀錯誤。

但如果變形嚴重、毀損的話，還是會變得很難掃甚至掃不過，每次南港運動中心健身完都會卡在掃碼機前面受到健身人員的關切，一定是我太認真運動的關係都不小心讓 QR Code 變形了

隨著他的應用性慢慢被發現出來，不知不覺間，QR Code API 也是簡單易用，並佔領了各個角落，據說發明者是以「圍棋」作為發想喔

縮網址

使用縮網址的話背後發生了甚麼事?

短網址 = 網域名稱 + 網址代稱

背後的運作原理都是透過「重新導向」實現

網域名稱 (Domain)：預設是短網址服務商的網域，例如：reurl.cc、pse.is、lihi.cc 等等。
雖然方便，但採用自家網域，可以提升使用者的信任感。

網址代稱 (Slug)：一串英數夾雜的編碼，例如：vYD9C、5rkewq 等等。改成與網頁內容相關的名稱，可以使用者一目了然。

縮網址的好處有哪些

為了不要被 ptt 的網友撻伐

網址如果太長很容易斷行，你的分享反而造成別人的困擾

短網址除了能將網址縮短美化外，還具有以下好處：

  1. 易於分享：長網址不僅難以記憶，也不好輸入。將網址縮短更容易讓人分享，以利內容傳播。

  2. 節省空間：將網址縮短能節省空間，特別是在有字數限制的平台上 (X、Threads、簡訊等等) 更是如此。

  3. 提升品牌識別度：企業或個人可以使用「自有網域」的短網址來提升品牌識別度，使用者也比較安心點擊來自的品牌網域的短網址。

  4. 追蹤數據：透過短網址可以追蹤連結的成效表現，分析使用者從何而來、獲得多少點擊次數、透過什麼裝置查看內容等等。

  5. A/B 測試：如果你想比較哪一個網址能為你帶來更好的行銷效益，就需要透過 A/B 測試才能得知。

但老話一句，甚麼事情都是有成本的

短網址讓惡意網站得以隱藏自己的身分，所以我們通常我們也不太信任短網址，會建議更換成自己的 Domain

短網址雖然方便，但若伺服器停機或終止服務刪除，短網址資料，則短網址即失去效用，原先的長網址也會覆水難收

如果對來路不名的短網址抱持著不信任的態度，有的網站在跳轉都非自家網域的網頁會提供「預覽長網址內容，再決定要不要跳轉去長網址」的提醒設計，很貼心

可能我們會問，既然縮短網址好處多多，那為什麼大家沒事要把網址弄那麼長?

關於這個問題，已經有網友完成了一組問答
https://www.ptt.cc/bbs/ask-why/M.1182479336.A.B09.html

簡單來說，例如上面這個鏈結本身，我們可以看出開發者的設計就是明確的資料夾分層，bbs 底下分成不同的資料夾對應不同的板，例如 ask、boy-girl、gossip…，然後底下再對應每一篇文章推文等等，開發者都會有自己的資料管理層級不會全部放在一個資料夾裡面亂數儲存

並且縮網址一般也是屬於短期的效用，例如分享、App 導轉下載，通常還是不會把自家網站直接用第三方的網頁做跳轉

會影響 SEO 嗎?

使用短網址不會對 SEO 造成傷害

當我們輸入短網址後會發生什麼事？

短網址的運作原理：
當你使用短網址（如 http://goo.gl/test）時，就像是寄信到一個郵政信箱。郵局（DNS 和短網址服務器）知道這個信箱對應的實際地址，會將你的信（你的瀏覽請求）轉發到正確的地方。

具體步驟：

  1. DNS 解析：就像查找郵政信箱的位置。
  2. 服務器接收請求：郵局接收到你的信。
  3. 301/302 重定向：郵局將你的信轉寄到實際地址。

301 轉址是永久性轉址，一旦瀏覽器接受到 301 的轉址會將其新網址做暫存，下次造訪時則不會造訪短網址伺服器，意味著我們無法知道這個短網址被點擊的次數，因此可以設定 302 做暫時性轉址。

另外，Twitter、Facebook、Instagram 這類社群平台，都會自動將連結加入 nofollow 屬性，代表你再上面分享甚麼網站，不管原始網址、短網址都無法直接帶來 SEO 效益，使用者過去之後就自己憑本事把他留住了

轉導開 App ?

為了讓已經下載 APP 的用戶可以透過廣告，直接開啟購物 APP 下單結帳，於是 Deep Link 深度連結就因此誕生了，其原理簡單來說，就是透過一個定義好並註冊的 Customer URL Schema（例如 Shopee://XXX），來建立起 APP 與廣告投放 APP 內文之間的連接，避免斷點。

這樣一來，廣告投放不僅可以提高轉換率，還可以喚醒那些不常使用 APP 的殭屍用戶。但假如沒有下載 APP 的用戶又該怎麼處理呢？於是後來就演進了 Deferred Deep Link，（中文稱延遲深度連結），先判斷用戶是否已經安裝了 App 應用，如果沒有則先引導至 App 應用商店中下載 App， 在用戶安裝 App 後跳轉到指定 App 頁面 Deep Link 中。

Firebase Dynamic Links（FDL）服務，FDL 屬於一種智慧網址，簡單說就是可以透過開啟 APP 所對應的短網址，來達到開啟 APP 的服務，而這樣的方式也可以在行動版網頁上，透過行銷的方式來發送網址，來無縫接軌到 APP 的開啟上 (也就是所謂的 DeepLink)。而因為 APP 上架都需要慎密的把關，所以利用短網址開啟 APP 的方式，僅管用途被限縮，卻也相較開啟陌生網站來的安全。
