Asynchronous Programming (一. 初探)

Created2024-05-07|Updated2024-06-16|Asynchronous Programming
|Post Views:

最近公司專案上寫了一個Job，目的是能夠定期的去封存一些過期的資料，而Job執行過程中會有備份資料這一步，於是就想到往S3拋，開開心心寫完後執行，檢查了一下S3，ㄜ…資料沒上傳出去，這才發現自己對於非同步的寫法有許多不熟悉的地方

案件釐清

基本上傳S3的方法比照官網的作法這沒甚麼大問題

問題就在出在呼叫端沒有下等待，也沒有在最後取得Task結果

呼叫端沒有下 await (或更古老的 GetAwaiter)，結果就是程式碼，繼續往下執行，也不管S3上傳是否安好，不聞不問，彷彿一切與我何干

我們可以修正為以下方式來確保資料有上傳完成

1.以同步的方式等待方法執行完成

2.以非同步的方式等待方法執行完成才結束整個流程

回顧一下MS Doc說明，使用 async 修飾詞來將方法指定為非同步方法，就會啟用下列兩項功能:

1.標記的非同步方法可以使用 {% label await %} 來指定暫停點。 {% label await %} 運算子會告知編譯器，非同步方法只有在等候的非同步處理序完成後，才能繼續通過該點。 同時，控制權會返回非同步方法的呼叫端。

2.標記的非同步方法本身可以做為其呼叫方法的等候目標。(就是說我可以去 {% label await %} 這個有經過 {% label async %} 修飾的方法)

以MS Doc 的圖示進一步說明

1.步驟6指出，遇到 await ，暫且記住 await 敘述所在的程式碼位置，接著內部會起始一個非同步 I/O 工作，而且不等該工作完成便把控制流先返回呼叫端

2.等到 await 所等待的那個非同步工作執行完畢，控制流才又會切回來繼續執行剛才保留而未執行的程式碼。

也可以自行在在LINQPAD測試了一下同一個方法加上 await 的差別

正確的加上 await ，也就是正確的執行等待，讓這個方法去等待LogAsync去完成，此時的執行續可以釋放出來去幹別的事情，”等待”事情做完再繼續往下走，

非同步操作，回歸基本

其實根據上述的情境與測試結果，還是不算了解非同步的操作背後的意涵，經驗不夠深刻，以此為基礎，讓我們透過一個生活化的例子，來更深入地理解非同步操作的概念。

模擬一下洗內褲的情境：

值得注意的是，我們使用 task.Start() 來啟動洗內褲的任務，但不等待它完成，而是繼續執行後面的程式碼。在這個例子中，我們去看了進擊的巨人的結局，並花了一些時間感動(?)。

我們回來查看內褲是否洗完了，通過 task.IsCompleted 屬性來檢查任務是否已經完成。如果完成了，我們就輸出洗內褲的時間；如果還沒完成，我們就等待一下，直到任務完成，然後再輸出洗內褲的時間。

藉由這個例子，我們可以更清楚地理解非同步操作的概念。非同步操作允許我們在執行長時間運行的任務時，不會阻塞當下的執行流程，而是可以去做其他的事情。當任務完成後，我們再回來處理任務的結果。

Performance 與 Throughput

這裡我們以效能與產能的角度來看非同步

非同步的核心精神在於減少等待，讓執行緒同時處理更多作業藉以提升產能(Throughput)，而不是效能(Performance)，

這樣的說明感覺有點模糊，我根據自己的理解嘗試翻譯成白話文:

在這裡Throughput我們可以以整個工廠的維度理解，假設這個工廠需要製作產品A與產品B，今天執行續開始製作產品A，中間有一段需要交給外包來完成，這等待的時間裡，同一個執行續去製作產品B，而不是在那邊站著冥想，一段時間後，產品A的外包商再次交還產品A，執行續再次回來將它完成，這樣的時間利用，雖然相當沒有人性，但使得整個工廠可以單位時間內產出的產品變多了，但你說，製作單一產品的速度有變快(Performance)嗎? 並沒有，我們還是要等這個產品得所有細節一步步都要走完，才能產出

I/O Bound Work 與 CPU Bound Work

I/O Bound 表示作業的大部分時間都在等待 I/O 操作完成,例如:

從磁碟讀寫文件
向數據庫發送查詢並等待結果
通過網絡發送或接收數據
與外部設備或服務進行通信

在這些情況下,執行緒(Thread)大部分時間都處於空閒狀態,等待 I/O 操作完成。如果使用同步處理,執行緒會一直阻塞直到 I/O 操作完成,這期間不能處理其他任務。而使用非同步處理,當執行緒遇到 I/O 操作時,它可以將 I/O 操作交給系統處理,自己則空閒出來處理其他任務。當 I/O 操作完成時,執行緒再回來繼續處理。這樣可以大大提高執行緒的利用率和系統的整體產能(Throughput)。

但如果今天有大量的CPU Bound作業需要完成，就沒有空閒的Thread有在等待，也就沒有空閒去做別的事情，在這個情況下非同步的效果就會看不出來
