Dictionary (一. 多執行續操作測試與分析)

Created2024-05-18|Updated2024-11-18|C# 舞想
|Post Views:

這是一篇研究看到 Dictionary 爆炸很開心的奇怪文章

多執行續下操作Dictionary演示

情境 :

1.建立一個 Dictionary
2.建立 10 條執行續
3.每一條執行續從 0~999 確認是否擁有這些數字的 key，如果有則對 Value + 1，若無則設定為 1
4.如果很簡略的想像，第一條 Thread 剛進入迴圈會建立 1000 個 key，每個 value 皆為 1
5.依序 10 條 Thread，會把這 1000 個 Key 的 Value 加到 10

結果 :

噴得亂七八糟，好刺激啊

錯誤分析 ArgumentException :

讓我們再看一次這段錯誤

(頭上冒出雲)我不是只是在 Dictionary 添加 Key 嗎? 為甚麼會有 Array ? 為甚麼會有 destination 的概念 ? 超出了什麼 lower bounds ?

這就是所謂的，每個字都看得懂，但放在一起就什麼都看不懂得境界

首先我們先認知，一個 Dictionary 包含一個 bucket Array，其中這個 bucket 包含了指向第一個 Entry (第一組 KeyValue) 的指標

節錄一部分 Source Code

圖示

當我們今天添加了一組資料進來，會計算 HashCode 來得知要存放在哪個 bucket(index)，如果 bucket 是空的，會建立一個Entry，一個Entry包含了 Key, Value以及指向下一個Entry的位置

根據 Source Code，可以再拆解如下

1.計算 Key 的 HashCode
2.根據 HashCode 和 Array Size ,計算出在 Array 中的 Index 位置
3.如果該 Index 位置為空,直接將 KeyValue Pair 寫入該位置
4.如果該 Index 位置已經存在其他 KeyValue Pair (Hash Collision),則透過 next 串聯在後面
5.更新 Dictionary 的 Element Count
6.如果 Entry Array 要滿了,則觸發 Array Resize, 創建一個新的更大的 Array, 並將所有 Element 重新計算 HashCode 並寫入 New Array

因此，我們可以推測，由於 Dictionary 不是 Thread Safe 的關係，可能有以下情況

1.Thread A 正在計算一個 Key 的 HashCode ,並根據 HashCode 計算 Index 位置
2.Thread B 觸發了 Array Resize ,創建了一個新的更大的 Array , 將所有元素重新計算 HashCode 並寫入 Resize 後的 Array
3.Thread A 計算出的 Index 位置與新的Array 比對發現無法匹配而產生越界問題 ,導致 index 越界錯誤

==> 這就是典型的併發問題導致的資料更新過程的不一致性，解法有幾種

解法一. Dictionary + Locks

第一個作法就是使用 lock 來確保在任何時候只有一個 Thread 能夠做指定的操作(這裡是對 Dictionary 進行修改) ,避免了併發導致的不一致性。

因此它的優點就是作法算滿簡單清晰的，也不需要修改Dictionary的使用方式

至於缺點。。。 擔心會有衝突的地方，就都要上lock，這種很人工的事，其實很容易漏，曾有聽說 80 % 的 Bug 都是人為發生的

解法二. ConcurrentDictionary

System.Collections.Concurrent 的 namespace 下有 ConcurrentDictionary<TKey,TValue> 這東西

根據 MSDN，ConcurrentDictionary 就是設計來應對多執行續下操作的 Dictionary

Represents a thread-safe collection of key/value pairs that can be accessed by multiple threads concurrently.

參考 : https://learn.microsoft.com/en-us/dotnet/api/system.collections.concurrent.concurrentdictionary-2?view=net-8.0&redirectedfrom=MSDN

因此我們嘗試改使用ConcurrentDictionary

很神奇的我們避免了開篇看到的滿江紅，但實際上我們去看 Source Code，會發現內部的實現方式一樣是使用lock

因為很神奇，我們再來研究看看 ConcurrentDictionary 的 Source Code

在以上 ConcurrentDictionary Source Code 中，我們可以看到 :

1.ConcurrentDictionary 会在 Constuctor 中建立Table，整個 ConcurrentDictionary 資訊都封裝 Tables 中 ，Table 的內容包含 buckets、locks
2.buckets 是一個 Nodes Array，這裡的 Nodes 就是對應Dictionary的Entries，也就是存放 Key Value 的地方，這與 Dictionary不同，Bucket與Entry的概念融合在一起
3.lock 的數量根據 Constuctor 可以看到預設值使用 ProcessorCount 決定，我的筆電配置是 4 核心 8 執行緒，所以 DefaultConcurrencyLevel = 4(應該是這樣??)

我們可以發現，ConcurrentDictionary 也是使用lock來管理併發情境，既然Dictionary + lock 與 ConcurrentDictionary 都是用鎖，那他們的差異是甚麼呢?

首先，Dictionary 加鎖的概念就是整個 Dictionary 的所有 buckets 都上鎖，但對於沒有發生 Hash collision 的 bucket 也上鎖其實是一種過度保護及資源浪費

那 ConcurrentDictionary 的 locks 怎麼知道要保護哪些 buckets 呢?

我們再來看看 “TryAdd” 的 Source Code

可以看到使用 GetBucketAndLockNo 方法根據 HashCode 計算該元素所屬的桶 bucketNo 和 lockNo :

bucketNo 的計算方式是: bucketNo = (hashcode & 0x7fffffff) % bucketCount
lockNo 的計算方式是: lockNo = bucketNo % lockCount

例如, concurrencyLevel 為 4,bucketCount 為 16, 則 lock 的分配情況如下:

lock 0 保護 bucket 0, 4, 8, 12
lock 1 保護 bucket 1, 5, 9, 13
lock 2 保護 bucket 2, 6, 10, 14
lock 3 保護 bucket 3, 7, 11, 15

從上述過程可以看出,ConcurrentDictionary 使用了分段鎖的策略。每個 lock 負責保護多個 bucket,而不是每個 bucket 一個 lock。這樣可以減少 lock 的數量,降低記憶體開銷,同時還能提供足夠的併發度。

相對 Dictionary + locks 的作法，使用上更方便且降低了單純在 dictionary 加 lock 的效能損耗!
