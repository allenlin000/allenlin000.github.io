.NET CORE 組態

寫過程式一段時間後會發現，.NET CORE 使用了很多 Builder 模式，而組態 (Configuration) 也不例外， ConfigurationBuilder 內就像定義了構件組態的 SOP ，教你怎麼建立一個組態物件

什麼意思呢? 我們先來觀察 IConfigurationBuilder 的 Source Code

Add 作用是拿取組態的資料來源，來源可以是 一個 JSON 檔、環境變數、CLI 參數…so on

所謂的資料來源就是我今天寫了一份文件，內容可能是:

這個 App 哪個服務的設定值要打開
哪個 API 回傳有什麼限制
現在的環境是測試或正式環境
…

而這些文件稱為 ConfigurationSource ，我們就由 “Add” 把它讀進來，並且它會被放在 Sources 屬性裡面，所以這個 Builder 就有了搭建 “組態物件” 的 “材料”

有了材料之後，藉由 Build 個這動作把組態打造起來，得到 IConfigurationRoot 的實體 ConfigurationRoot，而 IConfigurationRoot 是繼承自 IConfiguration

在 Build 的 SourceCode 中我們還可以看到一個角色是 IConfigurationProvider，它的存在是用來封裝取得 ConfigurationSource 內資料給 Builder 構建 ConfigurationRoot 的中間管理層，每個 Source 都會有一個相對應的IConfigurationProvider

ConfigurationRoot 是整個組態的 “根”，是構建起來的整個組態的全貌，並提供了一個 Reload 方法來重新載入組態，今天若想要從某個字節點作為系統的 “根”，可以透過 GetSection 來取得子節點，形成另外一個組態系統 ConfigurationSection，而 ConfiguraitonSection 內部組成是 Key, Value, Path

讀取組態的作法

讀取組態分成 Weakly Typed Reading 與 Strongly Typed Reading，前者的優點就是比較簡單，並且可以動態訪問任何配置項目，不需要預先定義 Class，而缺點就是沒有 Type Safty，可能導致 Runtime Error 而難以發覺，然後讀進來都是 string，會需要手動做類型轉換，以下示範二者的作法

Strongly Typed Reading

基礎版主要 Get() Bind()的做法，其差異在於

1.建立 Class 時機：

Get 創建一個物件並填充它。
Bind() 填充一個已存在的物件。

2.靈活性：

Get() 稍微簡潔一些，因為它直接返回填充好的物件。
Bind() 允許你在綁定前後對物件進行自定義操作。

組態的覆蓋特性

當我們每增加一個 ConfigurationSource，它都會在 ConfigurationBuilder 上添加一個新的”層”或”集合”的組態值。
呼叫Build()方法時，所有這些”層”或”集合”的組態值都會被合併或”壓縮”成一個單一的 IConfiguration 物件。
如果後來添加的組態提供者有與先前提供者相同的鍵，後來的值將覆蓋先前的值。

例如，您可能在 appsettings.json 中有一個預設值，但您希望在正式環境中使用環境變數來覆蓋該值。由於環境變數提供者是在 JSON 檔案提供者之後添加的，所以環境變數中的值將覆蓋 JSON 檔案中的值。

通常的應用程式的設定方式

  1. 建立不同的設定檔
    appsettings.json：共用設定
    appsettings.Development.json：開發環境的特定設定
    appsettings.Production.json：正式環境的特定設定

  2. 設定環境變數：我們可以使用環境變數 ASPNETCORE_ENVIRONMENT 來設定當前的環境名稱。例如，在 Windows 上，可以使用命令設定環境為”Development”：set ASPNETCORE_ENVIRONMENT=Development

  3. 動態載入設定檔

資料結構及其轉換

組態資料經歷了一個從原始結構到邏輯結構的轉換過程。這個過程可以被描述為組態的「三態」轉換：

  1. 原始態（Raw State）：
    這是組態資料的初始形式，通常來自外部來源，如JSON檔案、環境變數、命令列參數等。
    在這個階段，資料還沒有被解析或轉換，它只是一個原始的、未加工的資料集。

  2. 中介態（Intermediate State）：
    當原始資料被讀取到組態系統中時，資料被解析成鍵/值對的形式，還沒有被轉換為應用程式中使用的邏輯結構。
    這是一個過渡階段，允許組態系統進行一些基本的資料操作，如合併不同的組態來源、覆蓋值等。

  3. 邏輯態（Logical State）：
    在這個階段，中介態的資料被轉換為應用程式中使用的邏輯結構。
    這通常涉及將鍵/值對繫結到POCO（Plain Old CLR Object）物件，使得應用程式可以更容易地使用和存取組態資料。
    這也是應用程式最終使用的組態形式。

總結

這篇文整理了使用組態的觀念包括 :

1.組態的核心：包括 ConfigurationBuilder、ConfigurationSource、ConfigurationProvider 和 ConfigurationRoot。

2.配置的建立方法：可以通過簡單的 Code 或在 App 啟動時進行配置。

3.配置讀取的兩種方式：

弱類型讀取（Weakly Typed Reading）：簡單但可能導致 Runtime Error。
強類型讀取（Strongly Typed Reading）：使用 Get() 或 Bind () 方法，提供更好的 Type Safety。

4.配置的覆蓋特性：後添加的配置源可以覆蓋先前的值，允許環境特定的配置。

5.配置的多環境管理：通過不同的 appsettings.json 文件和環境變量來管理不同環境的配置。

6.配置數據的三態轉換：從原始態（raw state）到中介態（intermediate state），最後到邏輯態（logical state）。
