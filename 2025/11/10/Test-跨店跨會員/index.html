核心原則：Tenant Boundary（租戶邊界）

不論是「店」或「會員」，都必須有一個一致的「租戶識別」，且整個系統要從上到下都尊重這個邊界：

使用者 → Token → Context → DbContext / Repository → SQL

每層都要帶著 ShopId，且驗證「你是否真的屬於這個 Shop」。

後端層面防範策略（最重要的三層）

1️⃣ Token / 身分驗證層：授權綁定 ShopId

會員登入時發 JWT 時，就要包含：

{
“MemberId”: “M123”,
“ShopId”: “A001”,
“Role”: “Member”
}

之後所有 API 都要從 Token 驗證：

Token 裡的 ShopId 與請求參數的 ShopId 必須相符。

不允許使用者「帶自己的 ShopId」覆蓋。

if (user.ShopId != request.ShopId)
return Forbid();

2️⃣ 資料查詢層：自動過濾（Global Filter）

在 EF Core 中，可利用 Global Query Filter 讓系統自動帶入租戶條件：

protected override void OnModelCreating(ModelBuilder modelBuilder)
{
modelBuilder.Entity()
.HasQueryFilter(o => o.ShopId == _currentShopId);
}

這樣你即使寫：

context.Orders.ToList();

底層也會自動變成：

SELECT * FROM Orders WHERE ShopId = ‘A001’;

➡️ 不需要每個查詢都手動加 where ShopId，可避免人為遺漏。

3️⃣ API 授權層：Claim + Policy 驗證

使用 ASP.NET Core 的 Authorization Policy：

options.AddPolicy(“ShopScope”, policy =>
{
policy.RequireClaim(“ShopId”);
});

然後在 Controller 裡：

[Authorize(Policy = “ShopScope”)]
public async Task GetOrders()
{
var shopId = User.FindFirst(“ShopId”).Value;
…
}

這樣即使有人手動改 ShopId 參數，也會被攔下。

前端與 API 層的安全策略

層級 防範手法
API Gateway 驗證 JWT 簽章、綁定 ShopId
Header 不允許客戶端手動傳 ShopId，由伺服器自動注入
URL 避免直接暴露 ShopId，例如 /shop/A001/order
前端 UI 僅顯示授權範圍資料（Ex: 管理後台只看自己的店）

會員隔離（Member Isolation）
