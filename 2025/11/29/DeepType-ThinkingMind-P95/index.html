P50, P95, P99

🕊️ P 的數學 —— 電梯裡的等待之譜

你每天早上都去買咖啡。熟悉的味道、熟悉的隊伍，但每次等待的時間都不一樣。有時三分鐘就拿到，有時十幾分鐘也等不到。你開始記錄這些等待時間 [90, 100, 110, 150, 180, 300, 1200]

這些數字，其實就像是「系統 latency 的樣本」：每一個等待時間，都是一個 request 等待被服務的過程。


P50 —— 大多數人的日常體驗

P50（第 50 百分位），代表「一半的人等得比它短，一半的人比它長」。

計算：0.5 × 7 = 3.5 → 向上取整 = 第 4 筆 → 150 秒

這就是 中位數。你大多數的等待經驗，大約就是這個水準。它代表「使用者的日常體感」，是你最熟悉的節奏。本質上

P50 沒有被極端值影響（不像平均值會被拖走）
P50 是一種「對群體最公平的代表值」


P95 —— 那些偶爾但令人印象深刻的等待

計算：0.95 × 7 = 6.65 → 向上取整 = 第 7 筆 → 1200 秒

這就是 P95。它代表的是：「95% 的人都等得比這更快，只有 5% 的人等得更久。」也就是說：有極少部分人，會遇到超長等待。P95 不代表「平均狀況」，而是觀察壓力邊界的指標，它讓我們發現：「有沒有小部分人，正在默默受苦？」


P99 —— 幾乎要走人的等待

它代表：「系統在最極端條件下會發生什麼事」。通常是某個人等了 20 分鐘都沒拿到咖啡、差點遲到——這是 P99。P99 是警報器，代表系統是不是快撐不住了



🕊️ 為什麼我們要在意 P95？

P95 代表的是「最慢的那 5% 使用者體驗」，也是業界普遍採用的一個重要觀測點。它不像 P50 那樣反映「日常感受」，而是專門用來嗅出壓力開始擴散的前兆。

當系統在低流量時，P50 和 P95 往往差不多，頂多偶爾有些 jitter，整體 latency 分佈還算穩定。但一旦進入高流量場景，例如搶票、特賣活動、促銷突襲，RPS（Request Per Second）攀升、系統資源變得吃緊、等待排隊的機率增加、Latency 開始「拉長」，整個分佈圖向右傾斜，而此時，P95 的位置，會比中位數更靠近「資源緊繃區」，可能正是進入排隊、鎖資源、等待資料庫連線等邊界處



🕊️ Request 上升，哪些情境會導致「少數 request 很慢」？


① Thread Pool Saturation

.NET、Java、Node.js… 都有自己的 thread pool，當 request 暴增時，大部分請求還能即時分派 thread，但總有那 5%~10% 的請求，只能乾等排班進線。這些等待時間，會直接落進 P95 區間。更糟的，當尾端那 1% 請求真的排不到 thread，甚至會 timeout，就像排隊排到打烊那刻


② DB Lock 先影響少數人，不會影響整體

A : checkout → 寫入 order table

同時

B : promotion → select + lock row

有人正在寫 → 其他 request 要等

這種 lock，只會影響少數人但不會影響全部，所以 P50 / P90 完全不動，P95 開始拉長


③ Cache Miss 機率本來就低，但一發生就很慢

95% 的請求打到 Redis 都秒回，很爽。但有 5% 的請求，剛好 miss → 只好去查 DB、或 call 外部 API，這過程一下子就慢了幾倍。而更複雜的是，在高併發下，因為 Redis 是單執行緒
當每個請求都得排隊處理，一有 GC / CPU context switch / background save，就可能「卡住幾十毫秒」

再加上：

  • network jitter
  • NIC / switch queue 擁塞
  • eviction 發生的不確定性（LRU / LFU / random）

④ Network jitter（網路抖動）通常影響 1%～5% 的封包

CDN → API Gateway → Service → DB，當其中任意一跳 jitter，部分 request 變慢，通常應是只會一小群 request 偶發的變慢，落在 P95 的範圍。


⑤ GC Pause（垃圾回收）

GC fGC（Garbage Collection）就像打掃阿姨突然封鎖走廊清理，原本流暢的 request 剛好遇上 GC 開始，就只能站在門口乾等。多數請求不會被影響（所以 P50 很快）少數 request 剛好卡在 GC 執行的時段，就會硬生生卡住 50～500ms，這種 latency spike 不會平均發生在所有請求，而是隨機命中少數 request——剛好落在 P95 的觀測區間。


⑥ Queue Delay（Message Queue / Buffer 滿）

如果你用了 queue-based system（像是 Kafka、RabbitMQ、buffer channel、或內部 task queue），當流量過高時，queue 滿了，進來的 request 不會失敗，而是「被排隊處理」，但這種排隊，不是所有人都會排，只會影響部分 request。於是就出現多數人請求秒完成，小部分人「等超久」，甚至 timeout



🕊️ 統計學的 P95

在統計學裡，當我們想「推估整個世界的真相」，需要在準確度 vs 成本之間做選擇

當我們想要更高信賴度（99%），固定的樣本數下區間變得太寬就會變得不實用，要把區間縮回來就需要樣本倍增，所以成本太高

若使用更低信賴度（90%），在相同區間下，所需要的樣本需求少（便宜），但錯判機率太高（不可靠
有 十分之一 的機率是真實重量根本不在這裡

相對來說給你稍微寬一點的區間，但錯的機率只有 二十分之一，就變得非常可信，做決策安全多了


信賴區間寬度公式

信賴區間的本質意義是「我根據手上的樣本，推估整個世界真正的範圍。」如果用同樣方式，一次又一次抽樣、計算 95% CI，最後會有 95% 的區間把真實值包含進去

CI (Confidence Interval) width = Z × σ / √n

  • Z ：信賴度（Confidence Level）對應的常數
  • n ：樣本數
  • σ：變異程度

信賴度越高（例如從 95% → 99%），Z 從 1.96 → 2.576（↑31%）、區間變更寬（= 測量變不精準），如果不想區間變寬 → 必須大幅增加樣本量（成本 ↑）

只換 Z 值，你會得到更寬的區間，但現實上，我們通常 不想 區間變寬（不想結果變不精準）。因此只能靠增加樣本量 n 來抵消 Z 變大的影響

要提升到 99% 信賴度，樣本量要變成 1.7 倍以上，這就是為什麼「提高信賴度 ⇒ 一定需要更多樣本」

  • P90 太樂觀，錯判機率太高
  • P99 太昂貴，樣本量要變成 1.7 倍以上
  • P95 「夠準」且「測量成本不高」
