SQL - 從 GroupBy 的陷阱到 Window Function 的救贖

Created2025-10-06|Updated2025-10-08|SQL 職場歷險記
|Post Views:


💼 避開 GroupBy 轉型造成索引失效

🔹 GroupBy 轉型的寫法

這樣每一筆資料在做 GROUP BY 和 ORDER BY 的時候，都要重複計算多次 CAST(h.OrderDate AS DATE)
執行計劃可能是這樣，每一筆都會經過一次 Compute Scalar（做 CAST 運算），再進入分組邏輯

優化後

這裡 CAST() 只會做一次，存在中間結果集中（記憶體暫存或 tempdb），後續的 GROUP BY 直接用「已經轉好的值」來分組
這樣「CAST」的開銷不會被重複放大，而且 SQL Optimizer 能更有效率地調度資源（例如記憶體 Grant）



💼 MAX + GROUP BY + JOIN

對整張表做一次 GROUP BY + 聚合（MAX）再做一次 JOIN 原始表去對照聚合結果，問題是多了一層 JOIN，尤其當資料量大、分組多、索引不足時，JOIN 成本會爆炸


🔹 RANK + PARTITION（推薦用於 Top N）

只掃描一次資料表，不需要再 JOIN 回去，可直接利用排序索引加速!
RANK 是「一次走完、一路排名」，掃一遍資料就結束了。更能吃到索引優化 RANK() OVER (PARTITION BY CustomerId ORDER BY OrderDate DESC)
可以對 (CustomerId, OrderDate) 上的複合索引 完全命中，速度飛快。而 MAX + JOIN 常常無法完全命中索引，JOIN 還得比對每個 CustomerId 和 MaxDate，可能造成大量 I/O。



💼 取 Max / Min 改用 Window Function


🔹 子查詢 + JOIN（傳統寫法）

  • 內層：找出每組的最大 Key4
  • 外層：再回 JOIN 取完整資料
  • 需要兩次掃描 MyTable（一次找 MAX，一次 JOIN 回），若沒有好索引（如 (Key0,Key1,Key2,Key3,Key5,Key4)），會使用 Hash Aggregate + Hash Join，非常吃 CPU/記憶體
  • 若 ties（同組最大 Key4 有多筆），會 JOIN 回多筆資料。

🔹 CTE + JOIN（語意分層）

  • 只是把子查詢拉到上面命名成 CTE，語意一樣，只是可讀性更高
  • Optimizer 會「內聯」這段，產生與上面同樣的執行計畫。除非 CTE 被多次引用，否則效能幾乎完全相同。

🔹 Window Function（最佳實務）

本質上，Window Function 就是要取那個代表那組的一筆資料!

  • 通常只要掃描一次表，即可完成排序與分組
  • 不需要再做回連 JOIN
  • 對於大資料表，這是最穩定、最容易被優化的寫法，若索引是 (Key0, Key1, Key2, Key3, Key5, Key4 DESC)，可以直接走 Index Ordered Scan → 幾乎不用 Sort！
面向 傳統 GROUP BY + JOIN Window Function
資料掃描次數 至少 2 次 (聚合、回連) 1 次
需排序或 Hash Hash Aggregate + Hash Join 單一排序（可用索引）
中間結果大小 聚合結果 + JOIN 結果 直接輸出
ties 處理 需額外 DISTINCT 或 TOP 可直接控制（ROW_NUMBER / RANK）
Optimizer 可調度性 較複雜 更可預測、記憶體使用穩定
