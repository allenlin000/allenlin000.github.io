🎵 問題的起點 — async lambda 與 Fire-and-Forget 陷阱

我們可以在 Lambda 前加上 async，這個 Lambda 的回傳型別會變成 Task。也就是說它不是「直接回傳結果」，而是「回傳一個代表結果的 Task」，async 方法不回傳結果，而是回傳「未來的結果」。

實際上這段程式有兩個潛藏的問題：

  • Fire and Forget：ForEach 本身不會等待內部的 async 動作完成，等於是「丟出去就不管了」
  • Thread Safety：results 是共用的字典，在多個非同步任務同時 Add() 時，可能會出現執行緒安全（Thread Safety）問題


🎵 解法 1：簡單但不平行

這樣的寫法雖然「安全又可預期」，但它的問題是：它是同步的等待方式。每個請求都要等上一個完成後才能繼續下一個，等於是「排隊」的概念



🎵 解法 2：真正的非同步 + 平行執行

這段才是「真正的非同步思維」。透過 Select() 建立一組非同步任務（每個 URL 對應一個 Task）。用 Task.WhenAll() 等待所有任務完成。回傳的結果可以整齊地整理成字典（Dictionary）



🎵 LINQ 的 Select 不會「馬上執行」

因為 LINQ 的 Select() 是 延遲執行（Deferred Execution） 的。它只是一個「規劃」，直到你真的去「取用結果」時（例如 .ToList()、.Count()、await Task.WhenAll()），它才會開始跑
