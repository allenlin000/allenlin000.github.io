ANTI-JOIN Scenerio

Created2025-10-07|Updated2025-10-07|SQL 職場歷險記
|Post Views:


JOIN

拿到左邊有右邊沒有的

「從 MyTable1 找出那些在 MyTable2 裡沒有出現過的 ID。」

用 LEFT JOIN 連接兩張表，如果 MyTable2 沒有對應的資料，[MyTable2].[ID] 會是 NULL
所以 WHERE [MyTable2].[ID] IS NULL → 就代表「MyTable2 沒有匹配」

這種「反向 JOIN」是很有用的查詢模式 ( 或可改用 NOT EXISTS（效能通常更好）)

找出「尚未出貨」的訂單

✅ 案例 1：找出「尚未出貨」的訂單

假設：

Orders：訂單主表

Shipments：出貨紀錄表

你要找出那些「有下訂但還沒出貨」的訂單：

SELECT o.OrderID, o.CustomerID, o.OrderDate
INTO #PendingShipments
FROM Orders o
LEFT JOIN Shipments s ON o.OrderID = s.OrderID
WHERE s.OrderID IS NULL;

📦 結果：

#PendingShipments 就是「待出貨清單」。

常用於倉儲系統的「出貨任務批次」。

✅ 案例 2：找出「未付款」的訂單

假設：

Orders：訂單主表

Payments：付款紀錄表

SELECT o.OrderID, o.TotalAmount
INTO #UnpaidOrders
FROM Orders o
LEFT JOIN Payments p ON o.OrderID = p.OrderID
WHERE p.OrderID IS NULL;

💰 結果：

找出那些還沒有付款紀錄的訂單。

常用於「付款提醒」「對帳批次」。

✅ 案例 3：找出「會員資料已存在，但還沒建立購物車」的使用者

假設：

Users：會員主檔

Carts：購物車表

SELECT u.UserID, u.Email
INTO #UsersWithoutCart
FROM Users u
LEFT JOIN Carts c ON u.UserID = c.UserID
WHERE c.UserID IS NULL;

🛒 結果：

可用來推播「建立你的第一個購物車」的通知。

✅ 案例 4：找出「商品已下架但庫存表還有殘留紀錄」

假設：

Products：商品主檔（含上下架狀態）

Inventory：庫存表

SELECT i.ProductID, i.StockQty
INTO #InvalidStock
FROM Inventory i
LEFT JOIN Products p ON i.ProductID = p.ProductID
WHERE p.ProductID IS NULL OR p.Status = ‘Discontinued’;

📦 結果：

找出庫存中「孤兒」商品（主檔不存在或已下架）。

通常用於資料清理或庫存稽核。

✅ 案例 5：找出「行銷名單中未成為正式會員」的人

假設：

MarketingList：潛在客戶清單

Users：正式會員表

SELECT m.Email, m.Name
INTO #UnconvertedLeads
FROM MarketingList m
LEFT JOIN Users u ON m.Email = u.Email
WHERE u.Email IS NULL;

🎯 結果：

找出那些從行銷名單中還沒註冊帳號的用戶。

可用於再行銷、email remarketing。

為什麼要用 INTO #temptable

✅ 暫存結果重複使用

一次查出結果放進臨時表後，可以在後續幾個查詢中多次使用。
例如：

SELECT COUNT(*) FROM #PendingShipments;
SELECT * FROM #PendingShipments WHERE OrderDate < ‘2025-10-01’;

✅ 效能提升

若這筆資料要被重複 JOIN、篩選、或用於報表，就不需重複計算。

✅ 分批任務處理

例如：把未出貨訂單暫存在 #temp，後續批次任務再依這個清單去生成出貨單。

✅ 資料快照

有時候你要做對帳或報表，要在特定時間點取資料快照（snapshot），
放在臨時表就是最簡單方式。
