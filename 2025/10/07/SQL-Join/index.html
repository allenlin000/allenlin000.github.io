💼 JOIN 背後的選擇


🌀 Nested Loop Join

有兩份名單

Outer Table：10 個學生名字
Inner Table：100 萬筆成績紀錄

要做的事是

「拿出第一個學生 → 去成績表查他的資料」
「再拿第二個學生 → 去查他的資料」
…
…
…

這適合小表 JOIN 大表、JOIN key 有索引、條件很精準（可以 Seek），如果成績表有索引（例如以 student_id 為索引），那每次查都很快 O(logN)；但如果沒有索引，用這個方法每次都要掃整張表 O(N)，那就爆炸!


🔗 Merge Join

就像你有兩份都照學號排序的名單

學生表（100 筆）
成績表（100 萬筆）

你拿著兩份表，從頭開始同步往下比

適合兩邊都已排序（或有可用的 Clustered Index）、JOIN 條件是等值比較（=）、資料量大、但排序成本可接受


⚙️ Hash Join

想像你有一張表（A）比較小，一張表（B）很大，你先把小表 A 的關聯欄位做成一張「Hash Map」，然後掃大表 B，對每一筆都用 Hash Key 檢查有沒有對應

適合沒有索引、JOIN key 不一定排序、大型資料倉儲型查詢，因為它完全忽略索引，所以如果原本表有很好的索引（例如 user_id 上有 Clustered Index），會反而浪費了那個高效的結構
在小表 JOIN 大表的查詢中，Nested Loop + Index Seek 通常比 Hash Join 快很多


引擎怎麼做選擇

條件 Optimizer 傾向使用
JOIN 條件為等號（=）
而且兩邊都大、沒索引		 ⚙️ Hash Join
JOIN 條件為等號（=）
而且兩邊都已排序或有索引（例如 Clustered Index）		 🔗 Merge Join
JOIN 條件為等號或範圍（=, <, >, BETWEEN）
而且外表小、內表有索引		 🌀 Nested Loop Join
JOIN 條件是非等值（例如 <, >） 🌀 Nested Loop Join（唯一可行）
表的統計資料估錯（Row Estimate 太低） 可能誤選 Nested Loop 而慢爆
Join 類型 索引需求 排序需求 記憶體需求 大資料表效能 小表 vs 大表效能
Nested Loop ✅ 強烈建議 ❌ 不需 🔹 低 ❌ 差 ✅ 極佳
Merge Join ⚙️ 可用 ✅ 需要排序 🔹 中 ✅ 優 ✅ 優
Hash Join ❌ 不需 ❌ 不需 🔸 高 ✅ 優 ❌ 中等


💼 隱式轉換

資料庫在做 JOIN … ON a.k = b.k 時，兩邊型別必須能比較。如果一邊是 string、另一邊是整數（bigint），引擎可能會依「隱式轉換」規則，把兩邊都先轉成 double（浮點數）再比較


🔹 精確度

有個問題是，double（IEEE-754）只能精確表示到 2^53（≈ 9.0e15）以內的整數。超過這個範圍，double 會以「格子」的方式表示數字，每個格子相距不是 1，而是 2、4、8、16、32…（距離隨數字變大而變大），於是兩個原本不同的整數（例如 111111111111111111 與 111111111111111110），一轉成 double 就被「四捨五入」到同一個格子，變成相等，JOIN 就被誤判連上了，導致「重複或錯配」

過程是這樣

P.S. 其實不只 JOIN，連 GROUP BY、DISTINCT、WHERE 比大小，只要跨型別被轉到 double，都可能產生精度陷阱


🔹 顯式轉型

如果我們手動轉型再 JOIN 呢?

他解決了精確度問題!

至於效能麻，因為這個例子兩邊都無法使用 Index，因此效能會很差

因為「轉型發生在 JOIN 條件的右邊（b.k）」，在資料庫優化器（Query Optimizer）看來，CAST(b.k AS string) 不是一個靜態欄位，而是一個「運算結果」，因此它沒辦法直接利用 b.k 的索引，而 a.k 雖然是 string，但沒有直接對應到 b.k 的索引資料型態，因此也無法使用 Index，而造成

  • 掃描範圍變大（可能全表掃描 Full Table Scan）
  • CPU 計算變多（每筆都要做型別轉換），如果表中有百萬筆資料，那 CAST(b.k AS string) 就會被呼叫百萬次
  • 記憶體使用變高（暫時儲存轉換後結果）

🔹 Temporary Table

這是最常見、最乾淨、不改 schema 的方法

#tmp_str 是「本地臨時表（Temporary Table）」，只在當前連線（Session）中有效，SQL Server 在執行 SELECT INTO 時會自動建立表結構
b.k 是 BIGINT，a.k 也已經轉成 BIGINT，所以 JOIN 條件型別一致，SQL Server 可以使用 tbl_int.k 上的索引（Index Seek），而轉型只發生一次（在建立臨時表時），JOIN 不會再重複轉換

若臨時表很大、JOIN 的資料量不少，可以加索引來進一步加速



💼 前置過濾 vs. 事後過濾


📜 SQL 執行順序

  • ON 是 JOIN 的階段 用來「決定怎麼連」
  • WHERE 是 JOIN 完之後才「過濾結果」

假設資料整這樣子

tbl1 (a)

tbl2 (b)


🔹 事後過濾（WHERE）

  • 先做 LEFT JOIN
  • 沒匹配到的列，b 全部是 NULL
  • 接著 WHERE b.ds = ‘20210101’ → 因為 NULL 不符合條件，被濾掉

結果就會變成 只剩匹配到的列（像 INNER JOIN）

a.k b.k
1 1

🔹 前置過濾（放在 ON 或右表子查詢）

  • 先把右表篩成「只包含 ds=’20210101’」的資料
  • 再做 LEFT JOIN
  • 沒對上的 a 仍然保留，b 是 NULL
a.k b.k
1 1
2 NULL
3 NULL

把條件放在 ON 或右表子查詢，等於「先把要 JOIN 的右表縮小」，再做 LEFT JOIN → 能保留左表 a 的所有列，只是 b 只有符合條件的才連上，否則為 NULL。


🔹 情境

場景 錯誤寫法（WHERE） 正確寫法（ON / 子查詢） 為什麼
全員工 + 是否打卡 只剩有打卡的員工 所有員工都列出，沒打卡為 NULL 想保留所有人
全商品 + 當日是否有訂單 只剩當天有訂單的商品 所有商品列出，沒訂單為 NULL 才能算「訂單覆蓋率」
全會員 + 是否兌換優惠券 只剩有兌換的會員 全會員列出，沒兌換為 NULL 才能算「滲透率」
全部出貨單 + 是否異常 只剩異常單 全出貨單列出，異常為 NULL 才能算「異常率」


💼 拆查詢


原本的查詢是這樣

從 customers 去 JOIN orders，然後算每個客戶有多少訂單
但問題是，當 orders 表非常大（例如幾百萬筆），而 customers 只有幾千筆時，這個 JOIN 實際上會建立一個巨大的中間結果（Intermediate Result Set）
假設：

customers = 1,000 筆
orders = 10,000,000 筆

那麼在 JOIN 的階段，SQL 引擎可能會產生 數百萬筆中間結果，即使最後 GROUP BY 只剩下 1,000 筆結果，這個過程依然耗費了大量的 I/O、記憶體暫存空間 與 CPU 計算。
這就是 JOIN 對效能的最大挑戰，JOIN 的過程中，會產生「資料乘積」的中間結果


Temptable / CTE / Table Variable

這裡把「訂單數的統計」先算好、先壓縮成一個小表，這張表只有每個客戶一筆資料，例如：

再拿這張「小小的統計表」去 JOIN customers，這樣 JOIN 的對象就不再是「幾百萬筆的 orders」，而是「幾千筆的 temp_order_counts」，自然效能提升非常明顯

✅ 先減少資料量，再做 JOIN

JOIN 的花費 ≈ 左表筆數 × 右表筆數（在最壞情況下）。如果能事先把右表（orders）壓縮成只有每個客戶一筆的彙總結果，那 JOIN 的複雜度就從「數百萬 × 數千」變成「數千 × 數千」

結語

夜色深了，杯裡的冰塊只剩半融。我們看著執行計畫裡的箭頭，一條條地流向那個「Hash Join」，SQL 的美，不在於寫出最炫的語法，而在於懂得「什麼時候該停手」。
正如京都老匠人說的那句話：

「刃を研ぐより、心を研げ。」（與其磨刀，不如磨心。）

學會選擇 JOIN 的方式，就像學會在職場裡調整自己的節奏。不是所有的事都要一口氣 JOIN 完，有時候，先慢下來、先整理、再連結，反而會讓查詢更優雅，也讓人生的效能更好
