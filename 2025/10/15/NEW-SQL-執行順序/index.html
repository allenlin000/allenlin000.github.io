💼 SQL 執行順序的兩個階段

階段 主要工作 時機
解析階段（Parsing / Compilation） 檢查語法、解析欄位、最佳化查詢計畫 在執行前發生
執行階段（Execution） 根據計畫實際去資料表取資料 在執行時發生


💼 SELECT * 會影響「解析階段」

資料庫要先知道「* 代表什麼欄位」，它才能生成查詢計畫。但 * 不是實際的欄位名稱，所以資料庫會

  • 1️⃣ 去 資料字典（Data Dictionary / System Catalog） 查詢這張表的欄位結構。
  • 2️⃣ 把 * 展開成：
  • 3️⃣ 再根據這份完整的欄位清單建立執行計畫（Execution Plan）

SQL 的邏輯執行順序是這樣

「查資料字典展開 *」其實發生在邏輯執行前的 解析階段，也就是在「查詢計畫建立前」完成的，
也就是多了一步!



💼 SQL 效能優化實例：先 GROUP 再 JOIN

假設你有以下幾個資料表

表格 筆數 說明
a 1,300,000 筆 主資料表，存放每筆紀錄（包含 name、bid、cid、did、count、position）
b 5 筆 小表，包含欄位 col1
c 5 筆 小表，包含欄位 col2
d 5 筆 小表，包含欄位 col3

需求是針對每一組 (a.name, a.bid, a.cid, a.did)，計算 SUM(count) 與 AVG(position)，並且 JOIN b、c、d 取出對應的欄位 col1, col2, col3

❌ 效能差的版本

SQL 執行順序中，會先把 a 與 b/c/d 做 JOIN，表格 a 有 130 萬筆，表格 b/c/d 各只有 5 筆，但 JOIN 完後仍會生成一個 超大的中間結果集（每列資料內容膨脹了），接著在這個「已經變很大」的中間結果上進行 GROUP BY。SQL 引擎必須建立 temporary table（暫存表） 來做分組、計算。這會使用大量的 記憶體（Memory）與 CPU 時間。最後再計算 SUM()、AVG()，得到你要的結果。

缺點是太早 JOIN，導致資料集變大，GROUP BY 處理的資料量暴增，記憶體壓力大、查詢速度慢。特別是當表 a 很大時（像這裡有 130 萬筆），影響更明顯。


✅ 高效能版本（先彙總再 JOIN）

先在表 a 內做 GROUP BY，這一步會先把 130 萬筆資料，彙整成「每組條件唯一」的結果，例如，假設 name+bid+cid+did 的組合只有 5000 種，那結果就只剩 5000 筆，資料量立刻縮小幾百倍。這個彙總後的小資料集，再去 JOIN b、c、d，JOIN 的對象小得多（只有幾千筆），執行起來非常快，也更節省記憶體，最後再取出需要的欄位，即可得到結果。

優點是，GROUP BY 在原始大表中先執行 → 最少的資料量就完成彙總，JOIN 處理的資料更少 → 更快、更省資源，避免產生巨大的中間暫存表，結果正確、邏輯清晰、效能大幅提升。



💼 子查詢 vs JOIN

需求是找出「月薪高於部門平均薪資」的員工

❌ 原始寫法（子查詢版本）

某些 Provider 中，這個查詢通常會被執行成 Nested Loop Join，也就是外層（employee e）逐筆讀取員工資料，每次都要執行一次子查詢，計算該部門的平均薪資，如果有 25 個部門，子查詢就會執行 25 次。
這會造成重複計算相同的部門平均值，浪費時間，若員工資料多（例如上萬筆），效能會非常差


✅ 優化後（JOIN + GROUP BY）

先在子查詢中用 GROUP BY 一次性計算出每個部門的平均薪資 → 這個結果只會產生部門數量筆（例如 25 筆），再將這個小結果集 JOIN 回 employee 表 → JOIN 是在記憶體中進行的，效率高、避免重複計算，這種方式在執行時會使用 Materialization（物化） 技術，將子查詢結果暫存在記憶體中，然後再與主查詢進行連線，每個部門平均薪資只算一次，JOIN 時使用 Hash Join 或 Merge Join，比 Nested Loop 快很多，適合大筆資料查詢的場景。



💼 OFFSET實現分頁的效能問題

分頁查詢的原理就是先跳過指定的行數，再返回 Top-N 記錄。分頁查詢的示意圖如下：

資料庫一般支援 FETCH/LIMIT 以及 OFFSET 實現 Top-N 排行榜和分頁查詢。當表中的資料量很大時，這種方式的分頁查詢可能會導致效能問題。

SELECT *
FROM large_table
ORDER BY id
LIMIT 10 OFFSET N;

隨著 OFFSET 的增加，速度會越來越慢；因為即使我們只需要返回 10 條記錄，資料庫仍然需要訪問並且過濾掉 N（比如 1000000）行記錄，即使透過索引也會涉及不必要的掃描操作。

對於以上分頁查詢，更好的方法是記住上一次獲取到的最大 id，然後在下一次查詢中作為條件傳入

SELECT *
FROM large_table
WHERE id > last_id
ORDER BY id
LIMIT 10;

如果 id 欄位上存在索引，這種分頁查詢的方式可以基本不受資料量的影響。

邏輯執行順序

(1) FROM t1 JOIN t2
(2) ON (join_conditions)
(3) WHERE where_conditions
(4) GROUP BY col1, col2
(5) HAVING having_condition
(6) SELECT [DISTINCT | ALL] col1, col2, agg_func(col3) AS alias
(7) UNION [ALL]
…
(8) ORDER BY col1 ASC,col2 DESC
(9) OFFSET m ROWS FETCH NEXT num_rows ROWS ONLY;

首先，FROM 和 JOIN 是 SQL 語句執行的第一步。它們的邏輯結果是一個笛卡爾積，決定了接下來要操作的資料集。注意邏輯執行順序並不代表物理執行順序，實際上資料庫在獲取表中的資料之前會使用 ON 和 WHERE 過濾條件進行最佳化訪問；
其次，應用 ON 條件對上一步的結果進行過濾並生成新的資料集；
然後，執行 WHERE 子句對上一步的資料集再次進行過濾。WHERE 和 ON 大多數情況下的效果相同，但是外連線查詢有所區別，我們將會在下文給出示例；
接著，基於 GROUP BY 子句指定的表示式進行分組；同時，對於每個分組計算聚合函式 agg_func 的結果。經過 GROUP BY 處理之後，資料集的結構就發生了變化，只保留了分組欄位和聚合函式的結果；
如果存在 GROUP BY 子句，可以利用 HAVING 針對分組後的結果進一步進行過濾，通常是針對聚合函式的結果進行過濾；
接下來，SELECT 可以指定要返回的列；如果指定了 DISTINCT 關鍵字，需要對結果集進行去重操作。另外還會為指定了 AS 的欄位生成別名；
如果還有集合運算子（UNION、INTERSECT、EXCEPT）和其他的 SELECT 語句，執行該查詢並且合併兩個結果集。對於集合操作中的多個 SELECT 語句，資料庫通常可以支援併發執行；
然後，應用 ORDER BY 子句對結果進行排序。如果存在 GROUP BY 子句或者 DISTINCT 關鍵字，只能使用分組欄位和聚合函式進行排序；否則，可以使用 FROM 和 JOIN 表中的任何欄位排序；
最後，OFFSET 和 FETCH（LIMIT、TOP）限定了最終返回的行數。

例如 WHERE 子句在 HAVING 子句之前執行，因此我們應該儘量使用 WHERE 進行資料過濾，避免無謂的操作；除非業務需要針對聚合函式的結果進行過濾。

– 錯誤示例
SELECT emp_name AS empname
FROM employee
WHERE empname =’張飛’;

該語句的錯誤在於 WHERE 條件中引用了列別名；從上面的邏輯順序可以看出，執行 WHERE 條件時還沒有執行 SELECT 子句，也就沒有生成欄位的別名。

GROUP BY

SELECT dept_id, emp_name, AVG(salary)
FROM employee
GROUP BY dept_id;

由於經過 GROUP BY 處理之後結果集只保留了分組欄位和聚合函式的結果，示例中的 emp_name 欄位已經不存在；從業務邏輯上來說，按照部門分組統計之後再顯示某個員工的姓名沒有意義。如果需要同時顯示員工資訊和所在部門的彙總，可以使用視窗函式。

如果使用了 GROUP BY 分組，之後的 SELECT、ORDER BY 等只能引用分組欄位或者聚合函式；否則，可以引用 FROM 和 JOIN 表中的任何欄位。

第一個查詢在 ON 子句中指定了連線的條件，同時透過 WHERE 子句找出了“張飛”的資訊。

第二個查詢將所有的過濾條件都放在 ON 子句中，結果返回了所有的員工資訊。這是因為左外連線會返回左表中的全部資料，即使 ON 子句中指定了員工姓名也不會生效；而 WHERE 條件在邏輯上是對連線操作之後的結果進行過濾。

篩選條件放在 on 和 where 後的區別

說到 ON 就是 JOIN，JOIN 有分 CROSS JOIN, INNER JOIN, OUTER JOIN, 其中 OUTER JOIN 有區分為 LEFT JOIN / RIGHT JOIN

而所謂的 CROSS JOIN 就是把所有可能的組合列出來，所以篩選條件放 ON / WHERE 沒差
再說到 INNER JOIN ，它的結果是把有 Mapping 到的資料才會呈現出來，所以篩選條件放在 ON / WHERE 最後的結果也都會篩選掉

所以重點就是 OUTER JOIN 了

所謂的 OUTER JOIN，就是以某一邊的資料表為基礎，去找出另外一個資料表中匹配的資料列，假設在 “ON” 中沒有匹配到資料，就是在另外一邊呈現 NULL，例如:

也就是說，ON 的篩選條件在 JOIN 操作期間發生，不會減少左表(LEFT JOIN)最後的記錄數。

從執行順序的角度可以看到，ON 的邏輯會先執行

a. FROM 子句執行，確定要連接的資料表。
b. ON 對左表（Employees）的每一行，檢查右表（Departments）中的匹配行。
ON 條件（e.DepartmentID = d.DepartmentID AND d.DepartmentID > 1）被評估。
如果條件為真，則形成匹配；如果為假，右表列填充為 NULL。
c. 形成 JOIN 結果，包括左表的所有行
d. WHERE 子句應用於 JOIN 完的結果。
e. 選擇指定的列（SELECT）。
