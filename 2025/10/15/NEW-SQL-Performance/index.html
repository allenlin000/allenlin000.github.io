💼 SQL 執行順序的兩個階段

階段 主要工作 時機
解析階段（Parsing / Compilation） 檢查語法、解析欄位、最佳化查詢計畫 在執行前發生
執行階段（Execution） 根據計畫實際去資料表取資料 在執行時發生


💼 SELECT * 會影響「解析階段」

資料庫要先知道「* 代表什麼欄位」，它才能生成查詢計畫。但 * 不是實際的欄位名稱，所以資料庫會

  • 1️⃣ 去 資料字典（Data Dictionary / System Catalog） 查詢這張表的欄位結構。
  • 2️⃣ 把 * 展開成：
  • 3️⃣ 再根據這份完整的欄位清單建立執行計畫（Execution Plan）

SQL 的邏輯執行順序是這樣

「查資料字典展開 *」其實發生在邏輯執行前的 解析階段，也就是在「查詢計畫建立前」完成的，
也就是多了一步!



💼 SQL 效能優化實例：先 GROUP 再 JOIN

假設你有以下幾個資料表

表格 筆數 說明
a 1,300,000 筆 主資料表，存放每筆紀錄（包含 name、bid、cid、did、count、position）
b 5 筆 小表，包含欄位 col1
c 5 筆 小表，包含欄位 col2
d 5 筆 小表，包含欄位 col3

需求是針對每一組 (a.name, a.bid, a.cid, a.did)，計算 SUM(count) 與 AVG(position)，並且 JOIN b、c、d 取出對應的欄位 col1, col2, col3

❌ 效能差的版本

SQL 執行順序中，會先把 a 與 b/c/d 做 JOIN，表格 a 有 130 萬筆，表格 b/c/d 各只有 5 筆，但 JOIN 完後仍會生成一個 超大的中間結果集（每列資料內容膨脹了），接著在這個「已經變很大」的中間結果上進行 GROUP BY。SQL 引擎必須建立 temporary table（暫存表） 來做分組、計算。這會使用大量的 記憶體（Memory）與 CPU 時間。最後再計算 SUM()、AVG()，得到你要的結果。

缺點是太早 JOIN，導致資料集變大，GROUP BY 處理的資料量暴增，記憶體壓力大、查詢速度慢。特別是當表 a 很大時（像這裡有 130 萬筆），影響更明顯。


✅ 高效能版本（先彙總再 JOIN）

先在表 a 內做 GROUP BY，這一步會先把 130 萬筆資料，彙整成「每組條件唯一」的結果，例如，假設 name+bid+cid+did 的組合只有 5000 種，那結果就只剩 5000 筆，資料量立刻縮小幾百倍。這個彙總後的小資料集，再去 JOIN b、c、d，JOIN 的對象小得多（只有幾千筆），執行起來非常快，也更節省記憶體，最後再取出需要的欄位，即可得到結果。

優點是，GROUP BY 在原始大表中先執行 → 最少的資料量就完成彙總，JOIN 處理的資料更少 → 更快、更省資源，避免產生巨大的中間暫存表，結果正確、邏輯清晰、效能大幅提升。



💼 子查詢 vs JOIN

需求是找出「月薪高於部門平均薪資」的員工

❌ 原始寫法（子查詢版本）

某些 Provider 中，這個查詢通常會被執行成 Nested Loop Join，也就是外層（employee e）逐筆讀取員工資料，每次都要執行一次子查詢，計算該部門的平均薪資，如果有 25 個部門，子查詢就會執行 25 次。
這會造成重複計算相同的部門平均值，浪費時間，若員工資料多（例如上萬筆），效能會非常差


✅ 優化後（JOIN + GROUP BY）

先在子查詢中用 GROUP BY 一次性計算出每個部門的平均薪資 → 這個結果只會產生部門數量筆（例如 25 筆），再將這個小結果集 JOIN 回 employee 表 → JOIN 是在記憶體中進行的，效率高、避免重複計算，這種方式在執行時會使用 Materialization（物化） 技術，將子查詢結果暫存在記憶體中，然後再與主查詢進行連線，每個部門平均薪資只算一次，JOIN 時使用 Hash Join 或 Merge Join，比 Nested Loop 快很多，適合大筆資料查詢的場景。



💼 OFFSET實現分頁的效能問題

分頁查詢的原理就是先跳過指定的行數，再返回 Top-N 記錄。分頁查詢的示意圖如下：

資料庫一般支援 FETCH/LIMIT 以及 OFFSET 實現 Top-N 排行榜和分頁查詢。當表中的資料量很大時，這種方式的分頁查詢可能會導致效能問題。

隨著 OFFSET 的增加，速度會越來越慢；因為即使我們只需要返回 10 條記錄，資料庫仍然需要訪問並且過濾掉 N（比如 1000000）行記錄，即使 id 有建立索引，OFFSET 的效能問題仍然存在。
索引雖然能讓排序更快，但仍要遍歷前 N 筆索引項目。也就是說，它只是「有順序地走很快」，但還是要走過去

🔹 「鍵值分頁（Keyset Pagination）」優化

與其讓資料庫「跳過」一堆資料，不如我們自己記住「上一頁最後一筆的 id」，這種方式被稱為 Keyset Pagination 或 Seek Method

如果 id 欄位上存在索引，這種分頁查詢的方式可以基本不受資料量的影響!
