Index - 發揮作用...了嗎

Created2025-10-12|Updated2025-10-12|SQL 職場歷險記
|Post Views:


很多時候聽到「查詢慢」第一個反應就是：「加索引！」

但卻遇到明明已經對條件欄位建了索引，查起來還是慢得要命!


💼 情境

假設我們有一個「事件」的系統，事件分成兩類：

  • 新聞事件（News）
  • 社群事件（Social）

我們需要查出在一段時間內，某個類別（例如體育、政治…）裡「最熱門的事件」
查詢條件會包含：

  • 語言（Language）
  • 類別（Category）
  • 時間範圍（EventTime）

並根據「熱門程度（Amount）」排序，取前 N 筆

想當然地認為：「既然有 WHERE 條件，就幫這些欄位建索引吧！」

但結果查詢依然慢得不行，用 SQL Server 執行計畫 (Execution Plan) 一看，發現

  • 🔸 有使用索引，但用錯索引，或只用了部分索引
  • 🔸 有時乾脆 Table Scan（整表掃描）


💼 原因一：SQL Server 自動選擇「他覺得最划算」的路徑

SQL Server 在執行查詢前會經過 Query Optimizer（查詢最佳化器）它會評估：

  • 各種索引能過濾多少資料
  • 是否需要排序
  • 是否要 JOIN
  • I/O 成本預估

如果它覺得「直接掃整張表比用索引還快」，它就不會用你的索引，這就是為什麼有索引卻沒用上



💼 原因二：索引沒有覆蓋（Covering Index）

這個索引裡 沒有包含 Amount 欄位。所以 SQL Server 雖然可以用這個索引快速找到符合條件的記錄，但它還得「回主表 (Key Lookup)」拿 Amount ➡️ 這樣會造成大量的隨機 I/O


✅ 解法：改成「Covering Index」

這樣查詢需要的所有欄位都在索引裡，SQL Server 就不用回主表查資料，速度通常可提升數十倍



💼 原因三：複合索引的順序不對（Leftmost Prefix 原則）

在 SQL Server 裡，複合索引 也是有順序的。這順序會直接影響查詢能不能用上索引

能有效支援的條件是：

查詢條件 能用到索引嗎？
WHERE Language = … ✅ 可以
WHERE Language = … AND Category = … ✅ 可以
WHERE Language = … AND Category = … AND EventTime = … ✅ 可以
WHERE Category = … AND EventTime = … ❌ 不行（漏掉最左邊 Language）
WHERE Language = … AND EventTime = … ⚠️ 部分使用（可能效能不佳）

這個原則叫 Leftmost Prefix Rule（最左匹配原則）。也就是說，索引中的欄位必須從最左邊開始連續使用，
中間不能跳過。



💼 原因四：排序欄位不在索引中（ORDER BY）

如果查詢裡有 ORDER BY，而排序欄位不在索引中，SQL Server 通常會先掃出所有資料，再在記憶體（或 tempdb）排序；最後才取前 N 筆。這樣查詢就會超慢

💼 原因五：索引欄位的「資料分佈」不好（Cardinality）

Cardinality（基數）：就是「某個欄位有多少種不同的值」的程度。

  • IsActive 只有 0/1 兩種 → 低 Cardinality。
  • UserId 幾乎每筆都不同 → 高 Cardinality。

Query Optimizer 要決定走哪個 Index（索引）時，需要先估計「這個條件會篩出多少列」。這個估計，主要靠 Statistics（統計資料）。如果 Statistics 過期或不準：估計錯 → 可能選錯索引或選錯執行計畫，導致變慢。Optimizer 透過 Statistics 來估測 Cardinality；估錯，就可能選錯索引


💠 情境 1：低 Cardinality 欄位（例如 IsActive）

IsActive 只有 0/1，選擇性（Selectivity）低。如果大多數資料都是 IsActive = 1（例如 95% 都是 1），即使有 IsActive 的索引，也不一定會使用 Index Seek，因為掃幾乎整張表跟用索引再回表（Key Lookup）成本差不多甚至更高。這種情況，Optimzer 常會選 Index Scan 或 Table Scan，這不是「壞」，只是比較符合成本模型

若你只會查活躍資料，考慮 Filtered Index

這樣針對 IsActive = 1 的查詢，索引更小、更精準

若還有其他選擇性高的條件，如 CategoryId 或 UpdatedAt，把選擇性高的欄位放在複合索引（Composite Index）的前面


💠 情境 2：高 Cardinality 欄位（例如 UserId）

UserId 幾乎唯一，選擇性高 → 很適合走 Index Seek。但如果 Statistics 過舊（例如剛大量匯入/刪除資料，或資料分佈變化很大），Optimizer 就可能低估或高估回傳列數，導致選錯索引（例如選擇掃描一個大索引）

改善就是先把統計更新到新鮮的狀態



💼 原因六：查詢條件的順序與統計資料不符

💠 重點 1：WHERE 子句的寫法順序不重要，索引（Index）欄位的排列順序才重要

Optimizer 會重排條件順序，但索引的「鍵順序」決定它能不能高效率地用到「左側開始的一段（leftmost prefix rule）」。


💠 重點 2：複合索引的第一個欄位，會擁有最完整的統計（Statistics Histogram）與查找能力。

多欄位索引只有從最左邊開始的一段能做 Seek，也只有索引鍵的排列會產生對應的多欄位統計。例：索引鍵 (Language, Category) → Optimizer 對 Language 的分佈最清楚，對 (Language, Category) 也有統計；但若查詢是以 Category 為主，估算可能不準、Seek 力也變差。


💠 重點 3：索引欄位順序該依「選擇性（Selectivity）」與「使用頻率」決定。

一般經驗：把選擇性高、最常用來過濾的欄位放在最左邊；再放次要條件。若同時有排序/群組或回表成本等需求，再一併衡量。


💠 Equality 與 Range 的順序

若條件一個是等號（=），一個是範圍（BETWEEN、>、<、LIKE ‘x%’），把等號欄位放前面，再放範圍欄位，Seek 能力最好。
