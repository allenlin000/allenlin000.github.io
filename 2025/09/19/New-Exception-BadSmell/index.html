Return Code

當我們使用特定的回傳碼來代表某種錯誤時，代表著正常邏輯與錯誤邏輯混雜!

例如常見的呼叫 api

呼叫端變成這樣

這會有哪些問題?

流程混雜

在此，呼叫端就必須同時考慮正常回傳值和錯誤碼，程式的「主流程」和「錯誤流程」攪在一起，閱讀時腦袋必須同時記得 -1 是錯誤碼，這讓程式碼難以維護。

錯誤容易被忽略

如果開發者忘了檢查 -1 呢？這樣會誤以為呼叫成功，實際上失敗了，卻沒人發現。這就是「Return Code 容易被忽略」的核心問題。錯誤沒有被強迫處理，系統的可靠性下降。

資訊不足

回傳 -1 只代表「錯了」，卻沒告訴我們為什麼錯。可能是：

url 有誤？
系統異常？
參數有誤？

這種「資訊不足」會讓錯誤處理程式碼充滿 hard code 邏輯。

在 C#，推薦的寫法是 透過例外來處理錯誤情況。

呼叫端

✅ 這樣有幾個好處：

  • 正常邏輯與錯誤邏輯分離
  • 正常流程在 try 區塊，異常流程在 catch 區塊 → 理解起來更清楚!
  • 錯誤不被默默忽略，如果沒有 catch，例外會往外拋，程式甚至可能中斷，強迫開發者處理。
  • 錯誤資訊更完整，例外物件 (Exception) 可包含訊息、Stack Trace，甚至自訂錯誤類型。

UnProtected Main Program

如果最上層（Main 或 Thread）沒用 try/catch 包住，程式就可能因為一個沒處理的小錯誤直接崩潰，使用者會覺得這程式很不穩定。

例外傳遞就像泡泡：在水裡（呼叫堆疊）冒上來，沒人接住就一直往上飄，最後到水面（Main / Thread）。

如果水面上也沒人接，就「砰」一聲爆掉 → 整個程式崩潰。

所以 主程式要加一張「保護網」（try/catch），避免泡泡跑到最外層直接炸掉，這樣系統才不會看起來「隨便就掛掉」。
