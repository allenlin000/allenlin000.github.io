SOLID：五則皆變

原來這五個原則，都在談「改變」這件事

https://teddy-chen-tw.blogspot.com/2011/12/2.htmlS

鄉民們知道開發軟軟體最怕的是什麼嗎？請花 30 秒想一下…
開發軟體最怕的不是『寫程式』或是『不會寫程式（寫不出程式）』，最怕的是『改程式（改那些原本已經寫好測試過可以動的程式）』。先談一下既然程式已經寫好了，為什麼還要去改它呢？兩個主要的原因：
要增加功能。
要修正問題（bugs）。

藉由增加新的程式碼來擴充系統的功能，而不是藉由修改原本已經存在的程式碼來擴充系統的功能
( If the OCP is applied well, then further changes of that kind are achieved by adding new code, not by changing old code that already works.)

Plug-in architecture

Dependency Injection

https://www.jyt0532.com/2020/03/19/ocp/

這下你終於懂了 close到底是close什麼 就是要有一個不會改變的抽象 open到底是open什麼 就是要能夠提供彈性的實作

SOLID 只是一個心法 而工廠模式跟策略模式這兩個是我們用來遵循OCP的手段

類別有超過一個reason to change

https://igouist.github.io/post/2020/10/oo-11-open-closed-principle/

「對擴展開放」和「對修改封閉」

軟體並不是製造完畢就完工的東西，而是隨需求而生、隨需求而變的動態作品

基因適應環境並生存下去的能力

程式必須具有彈性

就是模組化

組裝電腦

Visual Studio 或是 Chrome 這類軟體的時候，我們也都會使用擴充套件

主機板上面事先會留好許多讓你接顯示卡或記憶體等等的插槽

留下一個供對接的地方，也就是擴充點。

答案就是抽象。（《無瑕的程式碼：敏捷完整篇》）

在主要邏輯和附加邏輯之間，加入抽象層來解耦合

實現 OCP 的案例參考

介面

  • 建立一個 IService
  • 各種行為用實作 class 來表達
  • 主流程只依賴 IService

常見於：策略模式、服務導向架構、DI 容器

策略模式（Strategy Pattern）

將變動行為抽出來注入

好處：行為可動態切換
OCP：新增一種叫聲不改 Animal

委派 / 函式注入（Func、Delegate）

不用定義一堆 interface，只要傳入 lambda
OCP：新增邏輯時不用改原本 class

事件機制 / 觀察者模式（Event / Observer）

加新行為用 += 不改 OrderService
實務常見於：事件匯流排、訂閱/發佈模式、UI

組態驅動（Config-Driven）

改設定不用改程式 ⇒ OCP

資料驅動（Data-Driven）

Step ActionType
1 SendEmail
2 SaveDB

你程式只要讀表，就知道要做什麼。加新流程改資料，不改邏輯

插件架構（Plugin / 模組化 / MEF）

把每個擴充功能封裝成獨立 DLL 或模組

用反射、MEF、Assembly.Load 動態載入

實務中常用於開放給第三方加功能

新增功能丟一包 DLL 就好，不動主程式 ⇒ OCP

管線（Pipeline） / 中介軟體（Middleware）

可插入/移除 middleware

ASP.NET Core 完整實作 OCP 的範例

加功能：加 middleware
不需動現有處理流程 ⇒ OCP

===> 設計可能會需要修改或擴充的行為, 提供插槽
