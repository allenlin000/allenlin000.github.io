WebAPI - ApplicationBuilder

Created2025-09-26|Updated2025-09-27|WebAPI
|Post Views:

在 ASP.NET Core 中，整個 HTTP 請求(Request) 和 回應(Response) 的處理，就像一條 管線 (Pipeline)。
這條管線是由一個一個的小元件（我們叫它 中介軟體 Middleware）組合起來的。決定要不要把請求交給下一個中介軟體。在下一個中介軟體之前做事（例如：紀錄 log、驗證身分）。在下一個中介軟體之後做事（例如：修改回應、壓縮資料）。如果一個中介軟體選擇不把請求往下傳，它就會 「短路 (Short-circuit)」 管線。這時它就是 終端中介軟體 (Terminal Middleware)。

簡易的實作

Use → 可以在 前後 做事（像一個攔截器 Interceptor）。
Run → 是一個 終點，到這裡管線就結束了。
Map → 可以做條件式分支，像是「如果路徑是 /api，就走另一條管線」。

中介軟體是一種 組合模式 (Composable Pattern)，把一堆小元件串起來，最終形成整個請求處理流程。

為什麼 Middleware 裡要寫 async？直接呼叫 next() 不行嗎？

在 ASP.NET Core 的 Middleware 中，next() 代表 下一個中介軟體的委派 (RequestDelegate)。這個 next() 方法 幾乎一定是非同步 (async) 的，因為 HTTP 請求/回應的處理（例如讀寫 Body、資料庫存取、API 呼叫）通常都是 非同步 I/O。如果不用 await，你呼叫 next() 只是「丟出一個工作 (Task)」但沒有等它完成，就會往下繼續執行，導致回應流程錯亂。

UseCors

CORS 的本質是瀏覽器端的安全機制（Same-Origin Policy 的例外白名單）。伺服器只透過回應標頭「宣告」哪些 Origin、方法、標頭被允許；最後放不放行是瀏覽器決定，他的角色是偵測跨域請求，依 Policy 加上允許的回應標頭。並且透過預檢且允許的方式，直接回 204/200 給瀏覽器，省去進一步處理。當不允許時，伺服器通常不加標頭、也不主動丟 403；接著的行為取決於你的管線/其他中介。

情境 A：Simple Request（簡單請求）

瀏覽器送出真正的請求

例如 GET https://api.contoso.com/orders，同時自動帶上 Origin: https://app.example.com 標頭。

CORS Middleware（UseCors）判斷

如果政策允許這個 Origin，就在回應上加 Access-Control-Allow-Origin: https://app.example.com（必要時還會加 Vary: Origin 等）。如果不允許，它通常不會擋下請求，也不會主動回 403，只是不加任何 CORS 回應標頭。

瀏覽器決定要不要給前端程式讀資料

若回應有正確的 Access-Control-Allow-Origin，瀏覽器把資料交給你的 JS。若沒有，瀏覽器就把資料「擋在外面」，Console 會報：No ‘Access-Control-Allow-Origin’ header…（就算伺服器回的是 200）。
重點是，CORS 是瀏覽器在執行與裁決；伺服器只是「宣告政策」的角色。

情境 B：Preflight（預檢）

當請求「不算簡單」時（例如 PUT/DELETE、自訂 Header、Content-Type 不是 application/x-www-form-urlencoded|multipart/form-data|text/plain、或要帶憑證），瀏覽器會先發一個 OPTIONS 預檢請求

瀏覽器先送 OPTIONS

UseCors 依政策評估

允許：回 204 No Content（或 200），並加上 Access-Control-Allow-Origin,Access-Control-Allow-Methods, Access-Control-Allow-Headers（必要時 Access-Control-Allow-Credentials）。
這表示「好，你可以發真正的 PUT/帶這些標頭」。

不允許：通常就不加這些允許標頭（可能讓下一個 Middleware 處理），結果 預檢失敗。

瀏覽器據此決定是否送出「真正的請求」

預檢通過 → 才會送出實際的 PUT /orders。
預檢未通過 → 真正的請求根本不會送，Console 直接報 CORS 錯。
（你可能在網路面板看到 OPTIONS 200/204 或其他碼，但只要少了那些 Access-Control-Allow-* 標頭，瀏覽器就判為失敗。）

UseAuthentication()

這個 Middleware 會去呼叫 已經在 Program.cs / Startup.cs 註冊的驗證方案 (Authentication Schemes)。驗證方案會去「讀請求裡的東西」，像是

Header (Authorization: Bearer xxx) → JWT Token
Cookie (.AspNetCore.Cookies) → Cookie 驗證
第三方登入（Google、Facebook OAuth）

驗證成功後，就會建立一個 ClaimsPrincipal（代表使用者），存到 HttpContext.User。
UseAuthentication 本身不需要你手動寫邏輯，它只是一個「管線上的鉤子」，實際邏輯來自你註冊的 Handler（例如 AddJwtBearer、AddCookie）。

👉 這一步只負責「知道你是誰」，不會擋請求。

UseAuthorization()

這個 Middleware 會在進入 Controller / Endpoint 前，去檢查「你能不能存取這個資源」。

它會檢查：

[Authorize] 屬性

如果驗證失敗

  • 沒登入 → 回 401 Unauthorized
  • 有登入但沒權限 → 回 403 Forbidden

UseAuthorization 本身也不需要你寫程式，它會呼叫 Authorization Middleware + Policy Provider，去比對 [Authorize] 的需求與 HttpContext.User 的 Claims。
👉 這一步才決定「你能不能做這件事」。

假設請求帶了 Authorization: Bearer ，Middleware 會呼叫 JWT 驗證器，去解析 Token。

驗證成功 → 產生一個 ClaimsPrincipal：

驗證失敗 → HttpContext.User 就是一個「空的匿名使用者」。
如果進到 /admin → 系統看到 [Authorize(Policy = “AdminOnly”)]。
去檢查 HttpContext.User 是否有 Role = Admin。

有 → 通過。
沒有 → 回 403 Forbidden。

UseStaticFiles

在 ASP.NET Core 中，app.UseStaticFiles() 是一個 Middleware，專門用來處理 靜態檔案 (Static Files) 的請求。

靜態檔案 指的是：HTML、CSS、JavaScript、圖片、字型、PDF… 這些不需要後端程式邏輯處理的檔案。
當瀏覽器發送請求（例如 GET /css/site.css），這個 Middleware 會去檔案系統找對應的檔案（預設是 wwwroot 資料夾）。如果找到，就直接把檔案傳回瀏覽器，不會再進到後面的 Controller 或 API。
它就是讓 ASP.NET Core 也能像一個「檔案伺服器」一樣，回應靜態內容。

如果專案裡有 wwwroot/css/site.css
瀏覽器請求 http://localhost:5000/css/site.css
Middleware 會直接找到檔案，回傳給瀏覽器。

當然也可以自訂資料夾

檔案實體位置：MyFiles/report.pdf
瀏覽器存取路徑：http://localhost:5000/files/report.pdf

Middleware 會根據副檔名設定正確的 Content-Type（例如 .css → text/css、.jpg → image/jpeg）。

如果副檔名不在已知清單裡，預設會拒絕回應（避免洩漏機敏檔案）。
預設只開放 wwwroot，避免洩漏整個伺服器檔案系統。

如果需要其他資料夾，必須明確設定 StaticFileOptions。

UseRateLimiter

他會限制「單位時間內允許多少請求」，藉以保護伺服器，避免被濫用（DDoS 攻擊或 API 遭過度呼叫）。
它會在管線前面就檢查

這個用戶的 IP / Token / 使用者身分 → 目前的請求數量是多少？若超過上限 → 直接回 429 Too Many Requests。

如果把 UseRateLimiter 放在 UseRouting 前面，它不知道請求要去哪個 Endpoint，就沒辦法判斷要不要套用哪個 Policy。

需要知道「這個請求對應的 Endpoint」才能判斷是否要套用限制。因此它通常要放在 UseRouting 之後。它會根據 Endpoint 的設定（像 .RequireRateLimiting(“fixed”)）來決定要不要限制。

UseResponseCaching

UseResponseCaching 是 ASP.NET Core 提供的一個 Middleware，用來提升效能。它的工作方式有點像「快取伺服器」，但它不是獨立的 Proxy（像 Nginx、Cloudflare），而是直接內建在 ASP.NET Core 的 Middleware 管線裡。減少伺服器重複處理相同的請求，做法是當某個請求第一次進來時，後端照常產生 Response，Middleware 會把 Response 存起來。下一次如果有一模一樣的請求，Middleware 可以直接回傳快取內容，而不用再跑 Controller/Service/資料庫。它會根據「HTTP Header + 設定」判斷能不能重用之前的回應。

電商網站通常不用 ResponseCaching，因為它太容易因為 Header 設錯而造成安全漏洞。
快取洩漏使用者資料

假設 /profile 回傳會員個資，但開發者不小心加上了 Cache-Control: public。

那麼 Middleware 就可能把「某個會員的 Response」快取下來，再提供給另一個會員。

結果就是 A 用戶看到 B 的資料 → 資料外洩。

敏感資料被代理快取住

有些 Proxy/CDN 也會根據 Response Header 快取。

如果 Header 設錯（例如把 Set-Cookie 的 Response 當成可快取），可能會讓敏感 Session 資料被共享。
它的功能很基礎

只能快取在 單一 Web 伺服器的記憶體 裡。

沒有分散式，伺服器重開快取就消失。

無法跨多台 Web Server 共用快取（在雲端或 K8S 環境就很雞肋）。

快取條件限制很多

只支援 GET / HEAD。

Response 不能超過大小上限（預設 100MB）。

不能快取帶 Cookie 或需要使用者身分的內容。

難以精細控制

它主要依靠 HTTP Header（Cache-Control, Vary, Expires）。

對於需要複雜快取策略（例如依角色、依 QueryString 條件、依商店/語言切分）的場景就不太夠用。

UseHttpsRedirection

UseHttpsRedirection 是 ASP.NET Core 內建的一個 安全性 Middleware，專門處理「把 HTTP 請求自動轉成 HTTPS」。

這個 Middleware 會檢查：

請求是不是 HTTP (http://)。如果是，就 回應一個 307 或 308 Redirect，把用戶導到 HTTPS (https://) 的版本。這樣使用者就不會不小心透過「不安全的 HTTP」來傳送資料。它就是自動幫你把 http:// 轉成 https://。

UseHsts

透過 HTTP Strict Transport Security (HSTS) 標頭，強制瀏覽器之後連線這個網站時都只能用 HTTPS

當你呼叫 app.UseHsts() 時，ASP.NET Core 會在 所有 HTTPS 回應裡加上這個標頭

意思是：

  • max-age=31536000 → 這個網站在未來 1 年（31536000 秒），都必須用 HTTPS。
  • includeSubDomains → 這個規則也適用於所有子網域（例如 shop.example.com、api.example.com）。

結果就是瀏覽器會記住規則，以後即使用戶輸入 http://example.com，瀏覽器也會自動改成 https://example.com，根本不會送 HTTP 請求到伺服器。

map

一般 UseXxx 中介軟體會形成一條 直線式管線：
👉 Request 進來 → Middleware1 → Middleware2 → Middleware3 → Response

但實務上我們常常需要 依條件走不同的管線：

依照路徑（例如 /api 一條管線，/admin 一條管線）。

依照 QueryString 或 Header（例如有特殊參數才走另一個邏輯）。

依照狀態（例如測試用戶走不同流程）。

這時候就會用到 Map、MapWhen、UseWhen。

  1. Map (依路徑分支)

根據 URL Path 來決定要不要分支。

一旦路徑符合 → 會把匹配的部分從 HttpContext.Request.Path 移除，存到 PathBase。

果

/map1 → Map Test 1

/map2 → Map Test 2

/map3 → 落到主管線 (non-Map delegate)

👉 特性：

只看「開頭是否匹配」。

一旦進入分支 → 不會再回到主管線。

MapWhen (依條件分支)

不是依 Path，而是依 謂詞 (predicate) 判斷要不要進分支。

謂詞是 Func<HttpContext, bool>。

app.MapWhen(
context => context.Request.Query.ContainsKey(“branch”),
branchApp =>
{
branchApp.Run(async context =>
{
var branchVer = context.Request.Query[“branch”];
await context.Response.WriteAsync($”Branch used = {branchVer}”);
});
});

結果

/ → Hello from non-Map delegate.

/?branch=main → Branch used = main

👉 特性：

一旦進入分支 → 也不會回到主管線。

適合「完全不同處理邏輯」的情境。

標準順序

(1) 錯誤處理要最前面

UseExceptionHandler、UseDeveloperExceptionPage 要先設定，這樣才可以攔截後面所有 Middleware 的例外。

(2) HTTPS / HSTS

要盡早強制安全性，確保所有請求進來就被轉到 HTTPS。

(3) 靜態檔案

UseStaticFiles 要放在 Routing 前面，因為它不需要 Routing，如果找到對應檔案就直接回應，不會繼續往下傳。

(4) Routing → Authentication → Authorization

UseRouting 會把請求和路由表對應，知道要進哪個 Controller/Action。

UseAuthentication → 確認「你是誰」。

UseAuthorization → 確認「你有沒有權限」。

👉 所以順序一定是：先找路徑 (Routing)，再確認身分 (Authentication)，再檢查權限 (Authorization)。

(5) MapXXX (Endpoints)

最後才是真正執行對應的控制器或 Razor Page。

如果放太前面，會跳過驗證或授權檢查。

RequestDelegate

RequestDelegate 本質上就是一個 委派 (delegate)，確實跟 Func 或 Action 的概念很像，只是它專門用來代表 ASP.NET Core 的一個 HTTP 請求處理函式

自訂

CorrelationIdMiddleware

在每一次請求產生一個 Correlation Id（若前端已帶 X-Correlation-Id 就沿用），把它放到 HttpContext 與 Response Header（方便前後端對齊 Log），並透過 ILogger.BeginScope 串進所有 Log，之後查問題時可以一鍵串整條請求。

自訂

注入管線

一次性攔截所有請求，回傳 503 Service Unavailable。

一般請求

Health Check（例如 /health）

白名單 IP（例如 127.0.0.1）

MaintenanceModeMiddleware.cs

ApiKeyMiddleware

對「給合作夥伴或內部系統」的 API，用最輕量方式要求 API Key。
若 Header/Query 沒帶或帶錯 → 直接回 401，避免落到後面昂貴的處理。
