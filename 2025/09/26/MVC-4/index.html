MVC - CRUD

Created2025-09-26|Updated2025-09-26|MVC
|Post Views:

Detail

想顯示他的那些資料?

例如學生，學生的基本資料（例如姓名、入學日期），不會顯示他選修的課程，因為 Enrollments 是一個集合 (collection)。

這個過程依靠 Routing (路由) 和 Model Binding (模型綁定) 把 URL 裡的 id 參數，傳給 Controller 的 Details 方法，最後再把資料交給 View 顯示出來。

URL 通常長這樣
http://localhost:1230/Student/Details/1

Routing
url: “{controller}/{action}/{id}”

  • 設計為 int? 為了讓牠不要直接拋無法 Binding 的錯誤
  • 找不到資料可以選擇
    • 回404 找不到資源
    • 回 200, 資料有撈, 但查無資料, 並且提供明確的 api code

Create

  • 防止 Overposting (過度繫結), 用 ViewModel 或 DTO，只包含需要的欄位，再把它轉換成實體 (Entity)。

CSRF (Cross-Site Request Forgery) 中文叫 跨站請求偽造

防 CSRF 攻擊, [ValidateAntiForgeryToken] 仍然存在，搭配 View 的 @Html.AntiForgeryToken()

簡單來說，就是駭客引誘使用者點擊某個連結或表單，讓使用者在 不知情 的情況下，替駭客「發出一個合法的請求」，而這個請求會帶著使用者的 合法身分 (例如 Cookies)，導致系統以為「這是使用者自己送出的」，結果做了使用者其實不想要的動作（例如轉帳、刪除資料、修改帳號設定）。

在 View 的 form 中放一個隱藏的「一次性驗證碼 (Token)」。

用 @Html.AntiForgeryToken() 來產生。

在 Controller Action 上加 [ValidateAntiForgeryToken]，驗證這個 Token 是否正確。

View (Razor 頁面)
@using (Html.BeginForm(“TransferMoney”, “Account”, FormMethod.Post))
{
@Html.AntiForgeryToken()



}

👉 @Html.AntiForgeryToken() 會幫你產生一個隱藏欄位，大概長這樣：

Controller
[HttpPost]
[ValidateAntiForgeryToken]
public IActionResult TransferMoney(string toAccount, decimal amount)
{
// 驗證成功才會進到這裡
// 這裡才會真的執行轉帳邏輯
return Content(“轉帳成功！”);
}

為什麼這樣能防禦 CSRF？

因為這個 Token 是：

隨機生成的，每次頁面載入都不一樣。

伺服器簽章的，不能被偽造。

綁定使用者的 Cookie，所以就算駭客偷到 HTML 表單，也拿不到正確的 Token。

換句話說，即使駭客在另一個網站上偷偷放了一個表單，騙使用者去提交，但因為 那個表單沒有正確的 AntiForgeryToken，所以後端驗證時會失敗。

前提：使用者已經登入你的系統，瀏覽器裡面有合法的 Cookie (例如 Session ID 或 JWT 存在 Cookie 裡)。

攻擊方式：駭客在自己的網站偷偷放了一個
