EF Core - DbContext 生命週期

Created2025-05-20|Updated2025-05-25|資料疆界的航圖
|Post Views:



Entity Framework contexts

By default, Entity Framework contexts are added to the service container using the scoped lifetime because web app database operations are normally scoped to the client request.EF DbContext 預設都會註冊為 Scoped 生命週期，理由是一次客戶端請求的資料庫作業常會自成一體。

標題要怎麼下

我們分別建立兩個 Repository，皆解析並注入 AdventureWorks2022Context

ARepository

BRepository

並且對應用程式皆註冊為 Scoped (Http Request 生命週期)

在 Program.cs 仿造部落格文章撰寫 Minimal API 測試(方便)，只是改成非同步版本

註冊為 Scoped，ARepository 與 BRepository 取得的 DbContext 是同一個物件，二者的操作也被包成同一個交易

改註冊為 Transient，ARepository、BRepository 的 DbContext 不同，甚至 ARepository 啟用交易後，可能還會引發資料庫鎖定導致 BRepository 無法寫入，兩邊也無法包成交易，不過寫入會不會互卡還關乎 Provider 是誰

EFCore 會幫我們管理連線的開啟與關閉 那麼為何還有些地方會看到這樣寫 using var context = new ProductContext(); 🤔

to Ho.Chun，那是自己管理 DbContext 的應用情境，例如在 Console Application 裡，應該沒有在 MVC 裡這樣用的範例吧？
若是寫在 ASP.NET Core MVC Action，DbContext Instance 由建構式參數獲得，生命週期由底層管理，基本上我們不需插手。

如果是 Entity Framework 內部透過連線字串自行建立的連線，就不該由你手動釋放（Dispose）這個連線。

相反地，如果是你自行建立 DbConnection 物件，然後交給 EF 使用，那麼 你就要自己負責釋放這個連線。

EF Core 會自己根據這個連線字串建立 SqlConnection。

EF 自己會管理它的生命週期。

❌你不應該自己去 Dispose 這個連線（否則可能造成連線異常或早期關閉問題）。

✅ 正確使用範例（二）你提供 DbConnection，你要負責 Dispose

你建立了 SqlConnection，然後交給 EF。

EF 不會幫你管理它的生命週期。

✅你要自己在適當時機 Dispose conn，例如在 using 區塊內或在應用程式結束時。
