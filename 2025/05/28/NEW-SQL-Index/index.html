EF Core - TimeoutException

Created2025-05-28|Updated2025-06-07|資料疆界的航圖
|Post Views:

✅ 題目二：在經常變動的欄位上建立索引，有什麼風險？
答案：更新與插入成本會提高，可能導致效能反而下降。
說明：
索引會隨著資料的變動而更新。如果某欄位更新非常頻繁，會導致：

索引更新成本高（尤其是在高併發下）

索引碎片增加，查詢效能下降

❌ 不建議在以下欄位加索引：

使用率低且經常變動的欄位（如「登入次數」）

隨時更新的狀態欄位（如「IsOnline」）

建議：
儘量只在查詢頻繁且變動不大的欄位上加索引。

可使用定期維護索引（REBUILD / REORGANIZE）來避免碎片嚴重。

✅ 題目三：為什麼索引太多反而可能拖慢系統效能？
答案：每次資料異動時，資料庫都要同步更新所有相關索引。
說明：
新增、更新、刪除一筆資料時，資料庫會：

寫入主資料表

同步更新該筆資料所屬的所有索引（這些動作不是免費的）

商業情境：
假設一張「訂單」表設了 10 個索引，每筆新增都要更新這些索引 → 若為高頻訂單系統，會造成效能瓶頸。

建議：
實務上建議一張表的索引數量控制在「4~6 個有效索引」左右。

每加一個索引，先問自己：「這是經常查詢條件嗎？」

✅ 題目四：什麼是 Filtered Index？在什麼情況下會很有幫助？
答案：Filtered Index 是針對特定條件下的資料建立的索引，能提升效能並減少空間。
說明：
sql
Copy
Edit
CREATE INDEX IX_Orders_PaidStatus ON Orders (Status)
WHERE Status = ‘Paid’;
這個索引只針對 Status = ‘Paid’ 的資料建立。

✅ 查詢這種資料的效能會大幅提升
❌ 但查詢其他狀態就用不到這個索引

商業情境：
在訂單系統中，90% 訂單都是 Status = ‘Pending’，只有 10% 是 Paid。若常查詢 Paid 訂單，Filtered Index 是極佳選擇。

建議：
適合用在稀疏資料（sparse data）上，如：「已完成訂單」、「啟用帳號」等。

避免為少量查詢情境建立龐大全域索引。

✅ 題目五：聚集索引（Clustered Index）與非聚集索引（Non-Clustered Index）有什麼不同？應該怎麼選？
答案：
項目 聚集索引（Clustered） 非聚集索引（Non-Clustered）
資料儲存方式 資料本身按照索引排序儲存 索引儲存 RowId，資料在別處
數量限制 每張表僅能一個 可有多個
適合欄位 常用來排序、範圍查詢 常用做搜尋條件（WHERE）

商業應用建議：
如果你的查詢常用「CreatedDate DESC」來排出最新資料，CreatedDate 是聚集索引的理想人選。

若資料表為「寫多讀少」，則聚集索引應選擇變動性低的欄位。
