Case 1 - DB Timeout

DB Timeout 屬於預期部會發生但難以完全避免的異常，因為資料庫讀取異常多是由於底層系統問題引起的，例如：

  • 網路連接中斷
  • 資料庫伺服器過載或當機
  • 資料庫鎖定衝突,導致 Timeout 等等…
  • 資料完整性問題

這些問題通常不是 Server 能直接解決的，當然我們也可以去避開容易 Lock 的時間點執行操作，並且設置一些有效的策略例如 Rollback & Retry 機制去避免我們耗費過多的時間釐清問題是必要的

而某種程度上我們知道怎麼處理的異常，例如 Rollback & Retry，並且在失敗時，也期望程式能告訴我們

  • 哪個 DB
  • 哪個 Table
  • 為了甚麼原因- 做了甚麼操作

這邊自定義 OrderProcessingException，主要是想要功能性區分，因為我們可以對訂單做很多操作，例如讀取訂單、更新訂單狀態等也就是達到重用性，如此一來，值班一看到就知道是 “訂單” 相關的錯誤，也因此了解了問題處理的 “優先級別”，而在我們的訊息中，也告訴了值班，在這個訂單更新 flow 中，是做什麼操作時掛掉的

其他類似情境還有

  • 第三方服務掛掉
  • 與 Windows API 交互
  • 網路錯誤
  • 讀檔問題(咬檔, 權限變動…)

適當的捕捉並識別這些問題可以讓我們節省時間與精力在釐清異常上

UI

還有一種做法是，在 Controller 層做 try…catch ，根據 Service 層 throw Exception，回給使用這介面舒服得訊息
如果程式可以從使用者的使用情境來思考，當錯誤發生時儘量顯示對使用者有意義的訊息，讓他們有機會自行排除大部分的問題

並行或是平行的錯誤處理

Task
Concurrency
並行或是平行處理算是程式語言裡面比較進階的項目，如果程式語言的例外處理有考慮到平行
處理，則同時間可以有數個例外一起被產生，而例外處理機制也要考慮如何來處理這些同時產生的例外狀況。

在單執行緒程式中，例外處理很單純：程式丟出一個例外（throw），然後透過 try/catch 把它攔截下來。

但在 多執行緒 或 非同步平行程式 中，多個工作可能同時失敗、同時丟出例外，你就會遇到兩個問題：

❓例外在哪個執行緒中發生？

❓主程式怎麼知道那些執行緒出錯了？

C# 對於並行處理的例外支援是「有限支援」的
✅ 有基本支援：每個執行緒可以丟出例外，也能捕捉

範例 1：Thread 不會自動回傳例外

new Thread(() =>
{
throw new Exception(“這裡的例外會 crash 執行緒，但主執行緒不會知道”);
}).Start();
✅ 執行緒內部會 crash

❌ 主程式不會 catch 到這個例外

你可以想成：每個執行緒是一個獨立的 try 區塊，錯了就自己死，其他人沒感覺。

🧠 C# 比較現代的並行處理方式：使用 Task
範例 2：Task 是支援例外傳遞的（這是 C# 的精華）

var task = Task.Run(() =>
{
throw new Exception(“Task 裡的錯”);
});

try
{
task.Wait(); // 或 task.Result
}
catch (AggregateException ex)
{
Console.WriteLine(“Task 發生例外：” + ex.InnerException.Message);
}

📌 重點說明：
Task 的例外不會立刻丟出，而是會被「包裝成 AggregateException」保留起來

等你 .Wait() 或 .Result 的時候才會丟出來（同步化等待）

多個 Task 同時錯，就會有多個 InnerExceptions！

這算是 比傳統 Thread 更完整的例外設計機制，但仍屬於「有限支援」，因為你還是得自己設計處理方式（不像資料庫的交易機制，會自動 rollback 等等）。

✅ C# 的其他補強機制
Task.WhenAll – 同時等待多個 Task，集中例外處理

var tasks = new[]
{
Task.Run(() => throw new Exception(“任務 1”)),
Task.Run(() => throw new Exception(“任務 2”))
};

try
{
await Task.WhenAll(tasks);
}
catch (Exception ex)
{
Console.WriteLine(“有任務失敗”);
foreach (var inner in ((AggregateException)ex).InnerExceptions)
Console.WriteLine(inner.Message);
}

TryParse

你要把文字檔的內容轉成 int，如果可能很多值都轉換失敗，那就應該用 int.TryParse()，而不是用 int.Parse() 然後再靠 try/catch 去抓。

  1. 什麼時候該用 try/catch

如果錯誤情況「很少發生」（真的屬於 例外），那就應該用 try/catch，因為這正是例外機制的用途。

如果 API 提供了 TryXxx（像 TryParse、TryGet），通常就該優先使用它，因為設計者已經告訴你「這個操作可能失敗，而且失敗是常態」。

  1. 為什麼有些 API 沒有 TryXxx

以 開檔 為例：你可能會想先 File.Exists，再 File.Open。但這樣並不安全：

檔案在你檢查時存在，但在你打開前就被刪了。

檔案在你檢查時不存在，但你要開時剛好又被建立了。

所以「檔案開啟」這類操作本質上就是「要嘛成功，要嘛例外」，無法完全靠 Try 來解決。

微軟因此沒有提供 TryOpen，而是要你直接呼叫 File.Open，如果失敗就讓它丟例外，因為這樣才正確。

每行資料都要處理。
部分資料行格式不符 → 不能整個流程中斷，要記錄下來。
最後統計有多少錯誤，並提示使用者或開發者。

  1. 不要靠 general exception (catch (Exception)) 控制流程

因為 parsing 失敗是「預期常態」，不是「程式出 bug」。
👉 用 TryParse 或類似的「安全 API」處理。

範例：

var errors = new List();
int lineNumber = 0;

foreach (var line in File.ReadLines(filePath))
{
lineNumber++;

if (int.TryParse(line, out var number))
{
    // ✅ 正確格式 → 處理
    ProcessNumber(number);
}
else
{
    // ❌ 格式錯誤 → 記錄下來
    errors.Add($"Line {lineNumber}: '{line}' 格式錯誤");
}

}

// 統計錯誤
if (errors.Any())
{
Console.WriteLine($”總共有 {errors.Count} 行資料格式錯誤，請查看 log。”);
File.WriteAllLines(“parse-errors.log”, errors);
}

  1. 只針對「真正不該發生」的錯誤用 try/catch
    例如：

檔案不存在 → FileNotFoundException。

硬碟 IO 出錯 → IOException。

權限不足 → UnauthorizedAccessException。

這些才是例外，應該交給 try/catch。

try
{
foreach (var line in File.ReadLines(filePath))
{
// 用 TryParse 處理格式問題
}
}
catch (IOException ex)
{
Console.WriteLine($”檔案讀取失敗: {ex.Message}”);
throw; // rethrow，交給上層處理
}

  1. 額外加值做法

統計報告：除了錯誤數量，可以把錯誤行數記錄到 log file，方便 debug。
容錯模式 (Tolerant Parsing)：如果格式允許，有時候可以寫 parser 去「盡可能修復或跳過錯誤資料」。
