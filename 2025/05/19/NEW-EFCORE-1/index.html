🧩 一、檢視產生效能取決於什麼？

簡單來說，產生檢視的速度（也就是 EF 建立查詢/更新規則的速度），
主要取決於兩件事：

模型有多大（多少個實體）

模型之間的關聯有多複雜（關聯、繼承、外鍵等）

在概念模型中：

Customer 和 Order 有一對多的關聯 → 它們是連線的。

Employee 繼承自 Person → 它們也是連線的。

在資料庫模型中：

Orders 表有一個外鍵指向 Customers 表 → 它們是連線的。

👉 當這些連線越多，系統在產生檢視時，就得考慮更多「彼此之間的關係」，
這會讓整體計算變得更複雜。

檢視產生會建立並驗證一個「查詢樹（Query Tree）」來描述對應邏輯。
在最糟的情況下，它的計算量是「指數級（Exponential）」。

也就是說：

如果模型有 5 個實體，效能可能還好；

但當實體變成 20 個、彼此又有很多關聯時，
系統必須同時檢查許多可能的對應路徑，
導致時間會「爆炸性地增加」。

💡 不過別擔心：Entity Framework 內部有一些優化機制（例如快取與依存分析），

當 EF 啟動一個模型時，會產生對應檢視（Mapping Views）。
這個過程會花時間，尤其在大型模型中。

👉 所謂「降低檢視產生成本」，就是讓這個過程不要在「程式執行時」才做，
而是在「建置或設計階段」先做好。

模式 發生時間 優點 缺點
動態產生檢視 執行時第一次載入模型時 不需手動設定 第一次執行會變慢
預先產生檢視（Pre-generated Views） 在建置階段完成 加快啟動速度 開發時仍需重建檢視

所以預先產生只是把工作時間往前挪，並不是真正消除成本。
因此，還需要從「根本上」讓檢視更容易產生 ——

使用「外鍵關聯（Foreign Key Association, FK）」而非「獨立關聯（Independent Association, IA）」

什麼是外鍵關聯（FK Association）？

簡單來說，外鍵關聯是指在實體類別裡，
直接存在一個外鍵欄位（CustomerId 這樣的屬性）。

🧩 什麼是獨立關聯（Independent Association）？

這是早期 EF 的做法，關聯資訊不透過外鍵欄位表達，
而是靠「隱藏的關聯描述（Association Mapping）」記錄。

EF 內部必須額外追蹤「哪個 Customer 對應哪個 Order」，
這就讓 檢視產生要處理更多隱含的關聯邏輯。

EF 中的三種內建快取

快取類型 範圍 作用 別名
🧩 物件快取（Object Cache） 在同一個 DbContext 內 快速重用同一筆資料，不重複建立實體 第一層快取（First-Level Cache）
⚙️ 查詢計畫快取（Query Plan Cache） 應用程式層 重用相同查詢所產生的 SQL 語句 Query Compilation Cache
📚 中繼資料快取（Metadata Cache） 跨連線共用 快速載入模型結構資訊（實體、關聯等） Metadata Workspace Cache

什麼是「物件快取」（Object Cache）
✨ 本質理解：

每當你用 DbContext 查出一筆資料時，EF 會在背景的 ObjectStateManager 中「記住」這個實體。
如果你下一次又查出同一筆主鍵資料，EF 不會再建立新的物件，而是直接拿快取中的那個。

可以避免重複具體化（materialization）的開銷

結果是 True，代表 EF 直接從快取拿出同一個物件，
而不是重新去資料庫再建立一個新的 Product。

即使使用快取，EF 還是會發出查詢到資料庫，
只是它會跳過「重新建立物件」的步驟。

如果你只想直接從快取拿資料，可以用 Find() 方法。

Find() 方法 — 直接從快取撈資料（若沒有再查資料庫）

Find() 是 EF 4.1 新增的 API，用來「依主鍵查找實體」。
它的行為可以想成：

先看記憶體中（Object Cache）有沒有那個主鍵；

如果有 → 直接回傳；

如果沒有 → 去資料庫查一次；

查到的資料會放進快取中。

行為 說明
會使用主鍵查找 只能用主鍵找，不支援其他條件
會回傳暫存（未 Save）實體 如果你用 Add() 新增了實體但還沒 SaveChanges()，Find 也能找到它
不會自動載入關聯實體 只回傳主實體，要載入關聯資料需用 .Include() 或 Lazy Loading
效能受 AutoDetectChanges 影響 若自動偵測開啟，快取很大時會變慢

AutoDetectChanges 對效能的影響

EF 會在某些操作（如 Find()、Add()、SaveChanges()）前，執行一個叫做 DetectChanges() 的動作，來檢查物件有沒有被修改。當快取裡的實體很多時，這步驟就會非常耗時。

暫時關閉自動偵測
context.Configuration.AutoDetectChangesEnabled = false;

var product = context.Products.Find(productId);

context.Configuration.AutoDetectChangesEnabled = true;

關掉可以顯著提升大量操作效能，但記得用完要打開。因為關閉後 EF 不會自動同步物件狀態，若忘了開啟可能導致資料更新錯誤。

因為 DetectChanges() 要檢查整個物件圖（Object Graph）裡的每一個實體，
確認它們的狀態（新增、修改、刪除），這是 O(n) 的成本

✅ 改善方式：使用 AddRange() / RemoveRange()
這是 EF 6 新增的 API，用來一次新增或刪除一整批資料。
優點是：DetectChanges() 只執行一次，而不是每筆都執行。

查詢計劃快取

每次你在 EF 執行查詢（無論是 LINQ 或 Entity SQL），
EF 都要經過一個「查詢編譯（Query Compilation）」的步驟，把高階查詢轉成資料庫能懂的命令（例如 T-SQL）。

這個過程包括：

分析查詢（Parsing）

驗證模型（Mapping）

轉換成查詢樹（Query Tree）

生成 SQL 命令（Store Command）

👉 這整個過程會花不少時間。
如果每次都重做，效能就會浪費在「重複編譯」上。

查詢計劃快取的作用就是：

把第一次產生好的「查詢計畫」記下來，以後相同的查詢就能直接拿來用，不用重新編譯。

⚙️ 二、查詢計劃快取的運作流程

以下是 EF 處理查詢的流程：

第一次執行：
LINQ/Entity SQL → Query Compiler → SQL Command → 資料庫執行
↓
存入查詢計劃快取

第二次執行：
LINQ/Entity SQL → 直接從快取取出 SQL Command → 資料庫執行

🧩 三、查詢計劃快取的特點與適用範圍
✅ 共用範圍

查詢計劃快取在同一個 AppDomain 內共用。
→ 不需要保留相同的 DbContext 實例。
→ 換句話說，不同的 DbContext 也可以共用這個快取！

使用 CompiledQuery 來提升 LINQ 查詢效能

當你在 EF 裡執行 LINQ 查詢時，系統會經過一個「查詢編譯（Query Compilation）」的過程：

分析你的 LINQ 運算式（Expression Tree）

驗證模型對應

產生對應的 SQL 語句

傳給資料庫執行

這個「編譯過程」會花時間。
如果相同的 LINQ 查詢會被重複執行上千次（例如 Web API 查詢清單），
那麼每次都重新編譯就會浪費效能。

👉 CompiledQuery 就是讓你「先編譯一次 LINQ 查詢、重複使用」的功能。

執行階段、Proxy（代理類別）、POCO

執行階段（Runtime）就是「程式實際運行的時候」，
跟「設計階段」或「編譯階段」相對。

var p = new Product();
p.Name = “Coffee”;
Console.WriteLine(p.Name);

階段 發生什麼事
設計階段（Design time） 你在 VS 裡寫程式碼
編譯階段（Compile time） C# 編譯器把程式轉成 .NET IL 指令（電腦能懂）
執行階段（Runtime） 程式真的跑起來，在記憶體裡建立 Product 物件

EF 會在「執行階段」去動態產生一些輔助類別（例如 Proxy），
來幫助它完成延遲載入與追蹤變更。

Proxy 就是「替你原本的類別做代理工作的類別」。

在 EF 裡，Proxy 類別會繼承你的 POCO 類別，
並「加上 EF 需要的功能」，例如：

偵測屬性何時被修改（變更追蹤）

當你第一次存取導航屬性時，自動幫你載入（延遲載入）

假設你有一個類別：

public class Product
{
public int Id { get; set; }
public string Name { get; set; }

public virtual Category Category { get; set; } // 導覽屬性

}

當 EF 執行時（Runtime），它不會真的建立 Product，
而是建立一個「繼承 Product 的動態類別」：

public class ProductProxy : Product
{
// EF 會覆寫屬性，加入額外邏輯
public override Category Category
{
get
{
// 延遲載入邏輯：第一次取用才去資料庫查
if (base.Category == null)
LoadCategoryFromDatabase();

        return base.Category;
    }
    set
    {
        base.Category = value;
        // 通知 EF：這個屬性被改了
        NotifyChange("Category");
    }
}

}

這樣 EF 就能在「你不知不覺中」：

自動載入資料；

自動追蹤變更；

自動知道哪些屬性被改動。

POCO 全名：

Plain Old CLR Object

意思是「純粹的 C# 類別」，不繼承任何 EF 的特殊基底類別，也不需要標記特別屬性

ublic class Product
{
public int Id { get; set; }
public string Name { get; set; }
}
這就是一個 POCO。
它就是你最普通的 C# 類別，沒有繼承 EF 的 EntityObject、沒有用 [Table] 或 [Column]。

EF 可以用「設定（Fluent API）」或「Attribute」的方式把它對應到資料庫。

┌──────────────────────────────┐
│ 你寫的程式 │
│ public class Product {…} │ ←── POCO
└──────────────────────────────┘
↓ （執行階段，Runtime）
┌──────────────────────────────┐
│ EF 動態建立的 Proxy 類別 │ ←── Proxy
│ class ProductProxy : Product │
│ + Lazy Loading │
│ + Change Tracking │
└──────────────────────────────┘
