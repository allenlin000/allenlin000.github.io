EF Core - Transaction

Created2025-05-19|Updated2025-05-25|資料疆界的航圖
|Post Views:



🐧關於 Transaction

「交易（Transaction）」是用來確保一連串的動作只接受全部成功或全部失敗的一種保護機制。是一個「全部做完才算數」的保證。如果中間某一步出錯，你在畫畫，如果畫錯了，你要把整張圖擦掉重畫，而不是只改其中一筆錯的，讓整張圖不一致。交易就是這塊橡皮擦，它保證資料不會「一半對一半錯」。

Using Transactions


SaveChanges() 自動包成 Transaction

在 EF Core 中，只要你呼叫 SaveChanges() 或 SaveChangesAsync()，它就會自動包一個隱含的交易（Implicit Transaction）。這代表：



  • 適合簡單 CRUD
  • 乾淨簡單、效能佳
  • 若發生例外，自動 Rollback


使用 BeginTransaction() 控制交易範圍，可以包多次 SaveChanges()

有些情境你會需要呼叫多次 SaveChanges()，但仍然希望它們「要嘛全成，要嘛全不成」。這時就可以使用 BeginTransaction() 取得 IDbContextTransaction：



  • 可多次 SaveChanges()
  • 手動控制 Commit / Rollback
  • 單一 DbContext


多個 DbContext 共用同一個交易

現代大型系統中，往往會有多個 DbContext（例如：User 資料庫、Order 資料庫），你會希望這兩邊能「一起成功或一起失敗」。這可以透過共用同一個資料庫連線與交易物件：



  • 手動控制交易跨多個 DbContext
  • 適用微服務式資料設計（拆表或分庫）


使用 TransactionScope 自動處理多資源交易

若要涵蓋多個 DbContext，甚至是 ADO.NET、Dapper、外部系統，使用 TransactionScope 是最簡單的方式。
TransactionScope 是一個更高層的做法，它會自動幫你處理交易的範圍。你只要把程式碼包在 using (var scope = new TransactionScope()) 裡，它會自動幫你記得要 Commit() 或 Rollback()（當然你得呼叫 scope.Complete() 才會成功）。



  • 自動偵測參與的資源
  • 自動決定 Commit / Rollback
  • 必須使用 TransactionScopeAsyncFlowOption.Enabled 才能搭配 async


🐧 Transaction 比較實測

你寫了一段看似完美的程式碼，部署上線後也沒紅字、沒爆炸，一切風平浪靜。但沒過多久，主管拿著報表衝過來問：

「欸？那筆訂單寫了一半就停了，現在資料變超奇怪，是不是你程式有問題？」

此時你一身冷汗，手開始抖，立刻打開資料庫查詢…

然後你鬆了一口氣。

因為你早就在程式裡用交易（Transaction）包好了整個流程，當寫入過程中某一筆資料因為欄位長度太長、資料格式錯誤而失敗時，整個操作其實早就 自動 rollback，資料根本沒寫進去！世界恢復和平，你也得以繼續活下來，平安的一天過去了，感謝 Transaction 的努力!

在 EF Core 中，交易就像是你後台的保險絲，不論是用預設的 SaveChanges()，還是明確的 BeginTransaction()、甚至是跨越多個資源的 TransactionScope，目的只有一個：

不是全部成功，就全部放棄。

接下來，我們就來實測三種交易處理方式，用真實錯誤來驗證 rollback 的可靠性



SaveChanges() 測試



執行這段程式後，你會收到熟悉的錯誤訊息：

String or binary data would be truncated.
SQLUpdateException

兩筆資料都沒進資料庫！這代表 EF Core 在呼叫 SaveChanges() 的時候，自動幫你包了一層交易（Transaction）。發生錯誤時，自動 Rollback。好貼心。

甚至可以用 SQL Profiler 看到，BEGIN TRANSACTION -> 然後一連串 SQL 執行 -> 接著是 ROLLBACK TRANSACTION
SQL Profiler



BeginTransaction() 測試



由於 DoJob1() 發生錯誤，整個交易還沒 Commit 就失敗，自然也會 Rollback。
結果兩筆資料都沒進資料庫。
SQL Profiler 證實有 BEGIN TRANSACTION -> 沒有 COMMIT -> 最後收場的是 ROLLBACK
SQLUpdateException

檢查資料庫，第一筆資料並未寫入，證明有 Transaction 且 Rollback 了。由 SQL Profiler 也成功蒐集到證據，兩次 sp_executesql 前有 BEGIN TRANSACTION，之後有 ROLLBACK TRANSACTION：
SQL Profiler



TransactionScope 測試



只要你沒呼叫 scope.Complete()，或者中途有一個爆了，就會自動回到初始狀態。資料沒寫入，SQL Profiler 出現 ROLLBACK。
SQL Profiler



☘️ 結語

在資料寫入的世界裡，有一種義氣叫做 Transaction。
不論你是單純地用 SaveChanges()，還是自己手動開局 (BeginTransaction())，甚至升級為能跨越一切的 TransactionScope，他們的共同信念只有一句：

「我們是資料兄弟，要衝一起衝，有人滑倒就全體退場！」
